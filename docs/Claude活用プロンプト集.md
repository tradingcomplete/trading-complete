## 1. 良いプロンプトの5要素(コンパナの仕事に当てはめる)

良いプロンプトには5要素が必ず入っている。元記事の汎用論を、コンパナの実例で書き換える。

### 要素1: 役割

Claude にどの専門家として振る舞わせるかを明示する。

| 悪い例 | 良い例 |
|--------|--------|
| 「TCの新機能の要件定義書を書いて」 | 「あなたは6万行のSaaS開発を支援するシニアプロダクトマネージャーです」 |
| 「TCAXの投稿を書いて」 | 「あなたはコンパナ口調を完璧に再現できるFXトレーダーで、ファンダメンタルズとダウ理論で語れる発信者です」 |
| 「note記事を書いて」 | 「あなたはAirPods Pro 3 レビュー記事の口調(語尾「ですね」「ですしね」、親しみワード「マジで」「神」)を完全に再現できるブロガーです」 |

役割を与えるだけで、語彙・粒度・文体が変わる。

### 要素2: 目的

何を作るのか、何を判断するのかを明確にする。

| あいまいな目的 | 具体的な目的 |
|----------------|--------------|
| 「TCAXの品質を上げたい」 | 「TCAX の Stage 1 検証で誤判定が出ている。原因を特定し、誤判定を月1回未満に抑えるプロンプト修正案を出す」 |
| 「LINEのリード獲得を強化したい」 | 「Edge Finder(無料ツール)のLP文章を作る。ターゲットは『簡単な記録ツールで負け続けた30代副業会社員』、目標は登録率30%」 |
| 「note記事を書いて集客したい」 | 「7年のFX経験者が記録ツール開発に至った話を、note 1500字で書く。読了後にLINE登録に進む流れを作る」 |

目的が具体的なほど、出力が「使える何か」になる。

### 要素3: 前提条件

読み手、背景、制約、業界、利用シーンを渡す。

TC関連でよく抜ける前提:
- ターゲットは「副業FXで負けてきた30代会社員」
- 価格は中核オファー要件定義書の指定値(プレースホルダ参照)
- 競合は myfxbook、トレダビ、自作Excel
- 差別化は強制チェックリスト・規律形成
- 文体ルールは note記事ライティングスタイルガイド v2.1 準拠
- アスタリスクは出力に含めない(AI感が出る)

T-CAX関連でよく抜ける前提:
- 投稿は朝5時、昼、ロンドン、NY、夜の5タイプ
- プロンプトは現在約23,665字(Phase R-4後)
- v13.0 検証統廃合済み(Stage 1 + Stage 2)
- 文字数上限300字
- 鉤括弧3段防御を破ってはいけない
- v14.0 Phase 1 のホットトピック構造は維持

### 要素4: 出力形式

箇条書き、表、見出し構成、ステップ順などを指定する。

コンパナの実務でよく使う形式:
- 要件定義書 → 「Phase 構成 + 検証手順 + 削減見込み + ロールバック条件」
- 投稿原稿 → 「冒頭フック + 因果 + ダウ理論 + 次の見方 + 文字数明示」
- note記事 → 「冒頭50字結論 + 見出し3〜5個 + 具体数値 + LINE誘導CTA」
- LINE配信 → 「冒頭フック + 1メッセージ1論点 + CTA」

### 要素5: 品質基準

「具体例を入れる」「曖昧表現を避ける」など評価基準を明示する。

コンパナの定番品質基準:
- 抽象論禁止、必ず具体的な数字またはコードを含める
- アナリスト調を避け、コンパナ口調(「〜ですね」「〜なんですよ」)で書く
- 専門用語は「要するに」で噛み砕く
- ネガティブ情報は代替案とセットで出す
- 表形式はnote記事には使わない(箇条書きに変換)

## 2. コンパナがハマりやすい失敗パターンと対策

### 失敗1: 役割だけ与えて目的を書かない

NG: 「あなたはTCAX設計者です。プロンプトを改善してください」
OK: 「あなたはTCAX設計者です。promptBuilder.gs の【投稿フォーマット】が4回重複しているのを1回に統合し、ペルソナの一貫性を保ったまま -800字 削減してください」

### 失敗2: 制約を書かない

NG: 「note記事を書いて」
OK: 「1500字、冒頭50字で結論、見出し4個、各見出しに数字を1つ、表形式禁止、LINE登録CTAで締め」

### 失敗3: 一度に複数の作業を頼む

NG: 「要件定義書を書いて、コードも書いて、テスト手順も書いて、Xにも投稿できるようにまとめて」
OK: 段階分割して進める。
- ターン1: 要件定義書(Phase 構成だけ)
- ターン2: その Phase 1 の詳細実装
- ターン3: 検証手順
- ターン4: X投稿用の要約

事件簿の事件8(投稿タイプ設計の根本誤り)も「一度にやろうとして根本がズレた」が原因。

### 失敗4: フィードバックを言語化しない

NG: 「もう少し良くして」
OK: 「冒頭が弱い。ヘッドニュース型ではなく、値動きインパクト型に変えて。3段落目の『〜と思われます』はコンパナ口調から外れているので『〜なんですよ』に書き直す」

### 失敗5: 一発で完成を求める

TC開発でも、T-CAXのプロンプト改修でも、一発完成は危険。事件簿の事件1〜15のうち多くが「一気にやって戻れなくなった」事件。

対策:
- まずシミュレーション
- バックアップ取得
- 1段階ずつテスト投稿で確認
- 品質低下が出たら即ロールバック

### 失敗6: 前提情報を渡しすぎる

要件定義書を5本まとめて貼り付けて「これを踏まえて」と聞くと、Claudeはどこに集中すべきか分からなくなる。

対策: 最重要情報を3〜5点に絞り、詳細情報は「補足」として後付けする構造にする。

## 3. 応用テクニック7選(コンパナの実例つき)

### テクニック1: ペルソナの二重設定

Claudeへの役割と、読者ペルソナを両方書く。

実例:
```
あなたは6万行SaaSを開発するシニアエンジニア(役割)。
読者はバイブコーディング歴1年の非エンジニア(コンパナ自身)。
コードレビュー結果を、専門用語を最小限に絞り、必ず修正案つきで出してください。
```

### テクニック2: 否定形の制約

「こう書くな」を明示した方が効く場面がある。

T-CAX投稿の禁止リスト例:
- アナリスト口調禁止(「〜と思われます」「〜と考えられます」)
- 比喩表現禁止(「氷山の一角のように」など)
- 抽象的な精神論禁止(「メンタルが重要」など)
- 投稿冒頭の「皆さんこんにちは」禁止

note記事の禁止リスト例:
- アスタリスク禁止
- 表形式禁止(箇条書きに変換)
- 「いかがでしたでしょうか」禁止
- 「〜について解説していきます」禁止(冒頭で結論を出す)

### テクニック3: 段階的な深掘り指示

ターン1で大枠、ターン2で深掘り。

実例(Trading Complete の機能改修):
```
ターン1:「機能Xの要件定義書を、Phase構成の見出しだけ出して」
ターン2:「Phase 1 を詳細化。実装ステップ・テスト手順・ロールバック条件を入れて」
ターン3:「Phase 2 と 3 も同じ粒度で」
ターン4:「全体を1つのドキュメントに統合して、ファイルとして保存」
```

一発で完成させようとしないことで、各段階で軌道修正できる。

### テクニック4: 品質評価軸の明示

採点基準を渡すと、Claudeは基準に合わせて自己調整する。

実例(T-CAX投稿):
```
以下の基準を満たしていれば合格とします。
1. 文字数 280字以内
2. 冒頭が値動きインパクト型(数字スタート)
3. 因果→ダウ理論→次の見方の順
4. ハッシュタグ2個以下
5. 鉤括弧の開閉一致
6. 「〜と思われます」が含まれない
この基準を全て満たす投稿を生成してください。
```

### テクニック5: 競合比較形式

評価軸を自分で決める。

実例(マーケティング):
```
TCを myfxbook、トレダビ、Excel自作と
評価軸 [強制力, 規律形成, 日本語対応, 価格透明性, 自己選別]で
比較した表を作り、最後にTCのポジショニングを1文で示してください。
```

軸をClaude任せにすると毎回違う軸で比較されて再利用できなくなる。

### テクニック6: 例示から始める

文体を指定する時は、サンプルを3本貼って「このトーンで書け」と指示する方が圧倒的に精度が高い。

実例(note記事):
```
以下3本のサンプルを分析し、文体特徴を抽出してください。
[AirPods Pro 3 レビュー記事を1本貼る]
[コンパナの過去note記事を2本貼る]
分析項目:文の長さ、語彙の硬さ、断定の強さ、段落の長さ、使わない表現の傾向
その後、新テーマ「FOMC予測の振り返り」で同じトーンの記事を1500字で書いてください。
```

### テクニック7: 出力後の自己採点を求める

「書き終わった後、上記の品質基準に照らして自己採点し、改善点があれば修正案も出してください」と付け加える。

これだけで成果物の密度が上がる。事件簿の「Stage 1 検証」も思想は同じで、自分の出力を評価し直すことで品質が上がる。

### A. Trading Complete 開発系

#### A-01. 要件定義書ドラフター

```
あなたは6万行SaaSのシニアプロダクトマネージャーです。
新機能[機能名]の要件定義書を書いてください。

前提:
- 対象ファイル:[ファイル名・行数]
- 既存設計:MODULES.md準拠、Public API、プライベートフィールド(#)、EventBus統合
- ユーザー:副業FX会社員、30代、バイブコーディング歴1年の開発者(コンパナ)

出力構成:
1. 背景と目的
2. Phase 構成(Phase 0 調査 → Phase 1 実装 → Phase 2 検証)
3. 各Phase の作業内容、検証手順、削減見込み(または効果見込み)
4. ロールバック条件
5. MODULES.md 準拠確認チェックリスト

品質基準:
- 抽象論禁止、必ず具体的なメソッド名・行番号・ファイル名を含める
- 「やってみる」「考えてみる」のような曖昧な表現を使わない
- 各Phaseは独立してロールバック可能であること
- アスタリスク禁止
```

なぜこの設計か: コンパナの実際の要件定義書(TCAX_v13_検証統廃合_シミュレーション など)の構造を再現している。Phase 構成を強制することで、一発実装の事故を防ぐ。

#### A-02. デバッグ診断コード生成

```
あなたはJavaScript/Supabase/GASに精通したデバッガーです。

問題:
[エラーメッセージまたは症状]

前提:
- 環境:[ブラウザ/Node/GAS]
- 関連ファイル:[ファイル名]
- 直前の変更:[変更内容]

タスク:
1. 原因候補を3つ出してください
2. 各候補を確認するためのデバッグコードを書いてください
3. デバッグコードは console.log で値を出すだけの最小実装にしてください
4. 修正は提案しないでください(まず原因確定が先)

品質基準:
- 仮説は具体的なメソッド名・変数名で示す
- デバッグコードは即コピペで動くこと
- 「おそらく」「たぶん」のような曖昧表現を使わない
```

なぜこの設計か: コンパナの開発原則「デバッグファースト・推測で修正しない」を徹底させる。修正提案を禁止することで、原因特定に集中させる。

#### A-03. リファクタリング計画

```
あなたはレガシーコードリファクタリングの専門家です。

対象:
[ファイル名・現在の行数・問題点]

タスク:
- このファイルを段階的に分割するロードマップを作る
- 各段階で何を切り出すか、何が残るかを明示

- 既存の挙動を絶対に変えない(機能とビジュアルの両方)

- バックアップ手順を最初に明示する

出力構成:
1. 現状分析(行数の内訳・責務の重複箇所)
2. 分割計画(Phase 1〜N)
3. 各Phaseの詳細
4. 検証手順
5. ロールバック方法
```

なぜこの設計か: 事件14「Phase R-3分割時の定数移動漏れ」のような事故を再発させないため、検証コマンドと既存挙動の保全を強制する。

#### B-02. 投稿品質診断

```
あなたはT-CAX投稿の品質審査官です。

直近5投稿を以下の観点で診断してください。

判定軸:
1. ヘッドニュース有無(必須)
2. 因果(なぜ動いたか)有無
3. ダウ理論(SH/SL含む)有無
4. 次の見方有無

6. 鉤括弧の開閉一致
7. アナリスト調混入の有無

9. 数字の根拠(時間限定主張の整合性)
10. ハッシュタグ数(2個以下)

各投稿について:
- 上記10項目のチェック結果
- 不合格項目の具体的な箇所(本文を引用して指摘)
- 修正後の文(全文書き直し)

最後に5投稿全体の傾向と、promptBuilder.gs のどのセクションを修正すべきかを提案してください。
```

なぜこの設計か: 事件8「投稿タイプ設計の根本誤り」、事件13「時間限定数値主張の捏造」のような問題を機械的に検出できる。

## 6. 使い始めの5本(まずここから)

40本(本書では30本)を一気に使う必要はない。まずこの5本を仕事に組み込む。

### 1. C-01:朝の相場ノート(T-CAX投稿改善)

→ 既に毎日生成しているものの品質ベンチマークとして使う。生成された投稿が基準を満たしているか自分で採点する習慣をつけると、promptBuilder.gs の改善ポイントが見える。

### 2. D-04:失敗談記事(note集客)

→ 失敗談は最も共感を得やすく、コンパナの真骨頂(7年の経験)が活きる。月1本書く習慣にする。

### 3. A-02:デバッグ診断コード生成(TC開発)

→ 「推測で修正しない」原則を徹底するための型。バグ調査のときは必ずこれを通す。

### 4. E-01:ステップ配信(LINE関係構築)

→ LINEを単発配信ではなくシステム化する起点。一度作れば長期的に効く。

### 5. F-02:ターゲット深掘り(マーケティング)

→ 全施策の精度がここで決まる。3ヶ月に1回は必ずやる。ペルソナの解像度が上がると、note記事もLINE配信も投稿も全て深くなる。

## まとめ

良いプロンプトには、役割・目的・前提・出力形式・品質基準の5要素が必要。

コンパナの仕事に当てはめるなら:
- TC開発では Phase 構成と検証手順を強制
- T-CAX改善では事件簿を踏まえた失敗回避
- 投稿ではコンパナ口調と数字根拠を強制
- noteでは note記事スタイル v2.1 準拠
- LINEでは1メッセージ1論点
- マーケでは評価軸を固定して比較

すべての共通点は「曖昧な依頼では曖昧な成果しか返らない」こと。

このドキュメントは育てるもの。使いながら、コンパナ専用のテンプレ集として更新していく。
