### 審査落ちの根本原因（リサーチ結果）

審査落ちは3層構造の問題による：

1. カードブランド（Visa/MC/JCB等）のMCC分類で「FX＝高リスク業種」として自動判定
2. PAY.JPは「投資関連サービス（ツールやコンサルティング等）」を審査困難商材として明記
3. FX情報商材への取り締まり強化の余波で「FX」キーワードが一括拒否される構造

TCは法的には金融商品取引法の規制対象外（記録・分析ツールであり投資助言ではない）。
しかしこの法的事実は自動審査では考慮されない。

教訓：**1つの決済会社に全部賭けない。多層構造で構築する。**

### PAY.JP Checkoutの仕組み（重要）

PAY.JPのCheckoutはpostMessageでトークンを送信する仕様。
data-on-createdやform submitは発火しない。

```javascript
window.addEventListener('message', async (e) => {
    if (e.origin !== 'https://checkout.pay.jp') return;
    const data = JSON.parse(e.data);
    if (data.action !== 'applyResponse') return;
    // data.response.id がトークンID
});
```

## 5. 多層決済戦略（v3.0 新規）

### 基本方針の転換

v2.1までは「1つの決済会社に申請→通ったらリリース」という直列型だった。
v3.0では「複数ルートを同時進行し、最も早く通ったものから順次稼働」する**多層型**に転換する。

```
多層決済戦略（v3.0）

  Layer 1: PayPal（即時導入・まずリリースを可能にする）
      ↓ 並行して
  Layer 2: MoR型（Lemon Squeezy / FastSpring）
      ↓ 並行して
  Layer 3: 国内ゲートウェイ（KOMOJU / fincode by GMO / Square）
      ↓ 並行して
  Layer 4: PAY.JP再申請
      ↓
  通ったものから順次追加 → 複数決済手段を提供
```

### Layer 2: MoR型プラットフォーム（カード決済の本命）

MoR（Merchant of Record）方式では、プラットフォームが「販売者」になるため、
コンパナが直接カードブランドの審査を受ける必要がない。
さらに税金の計算・徴収・納付を自動化してくれる。

### Squareの3段階処理とPayPalの1段階処理の違い（v3.7新規）

PayPal（実装済み）は **PayPal側でCustomer・Card・Subscriptionを一括管理**するため、フロント→Edge Functionの呼び出しが1回で済む。

一方Squareは**SaaS開発者側で3段階の処理を連鎖**させる必要がある：

```
Square 3段階処理フロー
─────────────────────────────────────

[フロントエンド]
  1. Web Payments SDK でカード情報をトークン化
     └─ 生のカード番号はSquareサーバーに直接送信（SaaS側は触らない）
     └─ nonce（単回使用トークン）を受け取る
     
  2. Edge Function呼び出し（nonce + ユーザー情報を送信）
     
[Edge Function: square-create-subscription]
  Step1. Customers API: CreateCustomer
         └─ Square Customer を作成（email, 名前）
         
  Step2. Cards API: CreateCard
         └─ nonce を使って Card on file を保存
         └─ 失敗時: Customer削除でロールバック
         
  Step3. Subscriptions API: CreateSubscription
         └─ plan_variation_id + customer_id + card_id で定期課金開始
         └─ 失敗時: Card削除+Customer削除でロールバック
         
  Step4. Supabaseに保存
         └─ subscriptions テーブル（provider='square'）
```

## 11. Phase 11: サーバー側プラン制限の詳細設計（v3.8新規）

### 11.1 背景と問題点

現状の Trading Complete のプラン制限は **PaymentModule.js（フロントエンド）に実装されている**：

```javascript
// 現状の実装（PaymentModule.js v3.0.0）
canAddTrade(totalTradeCount) {
    const limits = this.getPlanLimits();
    return totalTradeCount < limits.totalTrades;
}
```

これはブラウザで動作するJavaScriptコードのため、**ユーザーの完全な支配下にある**：

- F12（DevTools）でメソッドを直接書き換え可能
- `this.#currentPlan = 'pro'` のように内部状態を直接変更可能
- localStorage や IndexedDB の改ざん可能

**結果**: 技術力のあるユーザーは Free プランのまま無制限に使い続けられる。

### 11.3 設計方針

#### A. 多層防御（Defense in Depth）

| 層 | 役割 | 突破された場合 |
|----|------|-------------|
| 1. フロントエンド（PaymentModule） | UX向上・即座のフィードバック | サーバーが守る |
| 2. **サーバー側（RLS+トリガー）** | **最終防衛ライン** | **守れる** |
| 3. データベース整合性 | 不正データの混入防止 | データ削除トリガーで自動修復 |

**フロント側のチェックを廃止するのではなく、サーバー側のチェックを「追加」する**。フロント側はUX向上目的（保存ボタンを押す前に「上限到達」と表示する等）として残す。

#### B. プラン情報の信頼源

```
信頼すべきプラン情報の源:
  ✅ Supabase の subscriptions テーブル（DB）
  ❌ ブラウザの PaymentModule.#currentPlan
  ❌ localStorage / IndexedDB
  ❌ HTTPリクエストヘッダー
```

サーバー側のチェックでは、**毎回 subscriptions テーブルを参照**してプランを判定する。

### 11.4 実装案: trades テーブルの INSERT トリガー（最重要）

```sql
-- ユーザーのアクティブなプランを取得する関数
CREATE OR REPLACE FUNCTION get_user_active_plan(p_user_id UUID)
RETURNS TEXT AS $$
DECLARE
  v_plan TEXT;
BEGIN
  SELECT plan INTO v_plan
  FROM subscriptions
  WHERE user_id = p_user_id
    AND status = 'active'
    AND (current_period_end IS NULL OR current_period_end > NOW())
  ORDER BY created_at DESC
  LIMIT 1;
  
  -- アクティブなサブスクがなければ 'free'
  RETURN COALESCE(v_plan, 'free');
END;
$$ LANGUAGE plpgsql STABLE SECURITY DEFINER;

-- INSERT 前にプラン制限をチェックするトリガー関数
CREATE OR REPLACE FUNCTION enforce_trade_limit()
RETURNS TRIGGER AS $$
DECLARE
  v_plan TEXT;
  v_count INTEGER;
  v_limit INTEGER := 20;  -- Free プラン上限
BEGIN
  -- ユーザーのプラン取得
  v_plan := get_user_active_plan(NEW.user_id);
  
  -- Pro/Premium は無制限
  IF v_plan IN ('pro', 'premium') THEN
    RETURN NEW;
  END IF;
  
  -- Free プランの件数チェック
  SELECT COUNT(*) INTO v_count
  FROM trades
  WHERE user_id = NEW.user_id;
  
  IF v_count >= v_limit THEN
    RAISE EXCEPTION 'Free plan limit exceeded: maximum % trades allowed', v_limit
      USING ERRCODE = 'P0001',
            HINT = 'Please upgrade to Pro plan';
  END IF;
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- トリガーをテーブルに紐付け
DROP TRIGGER IF EXISTS check_trade_limit ON trades;
CREATE TRIGGER check_trade_limit
BEFORE INSERT ON trades
FOR EACH ROW
EXECUTE FUNCTION enforce_trade_limit();
```

**動作確認**:
- ✅ Free プランのユーザーが 21 件目を保存 → DB側で拒否
- ✅ Pro プランのユーザーは無制限
- ✅ プラン昇格後は即座に制限解除（毎回 subscriptions テーブルを参照するため）

### お金に関わる処理の鉄則

1. カード情報を絶対に触らない → 決済サービスのCheckoutに任せる
2. 課金状態の判定はサーバーサイドで → WebhookからDBを更新

4. Webhookの署名検証は必ず行う → なりすまし防止

### 審査対策の教訓（v3.0追記含む）

1. 決済審査AIは「ホームページの文言・画像」で業種を自動判定する
2. 「FX」「金融」「損益」「チャート」等のキーワードや画像は審査に悪影響
3. アプリ本体の内容は審査対象外（ホームページのみが対象）
4. 業種は「SaaS」「ソフトウェア」「情報サービス」で申請する
5. tokutei.htmlの決済プロセッサ名は汎用表現にしておくと切り替え時に便利
6. ホームページ画像はNotebookLM+実スクショのハイブリッド方式が効果的
7. 通貨ペア名・FX業者名が大きく見えるスクショは避ける or ぼかす
8. **1つの決済会社に依存しない多層構造が最も安全**（v3.0追記）
9. **MoR型はカードブランド審査を直接受けないため、FX審査リスクが低い**（v3.0追記）
10. **事前相談してから申請するとムダな審査落ち履歴を避けられる**（v3.0追記）
11. **商品説明の英語表現は「Trade Journal Software」「Performance Analytics Tool」**（v3.0追記）
12. **Stripe買収されたサービス（Lemon Squeezy等）はStripe時代の閉鎖履歴が影響する可能性が高い**（v3.3追記・Lemon Squeezy否決から判明）
13. **「理由非開示の否決」は一般的。Stripe/PayPal/カード会社の規制によるものと説明される**（v3.3追記）
14. **Squareの業種選択は「専門サービス / システム、ソフトウェア開発、運用、保守」が最適**（v3.3追記）
15. **決済サービスのアカウント作成と実際のWeb決済API利用（Web Payments SDK等）は別段階。アカウント作成通過＝決済可能ではない**（v3.3追記）
16. **Square実装はPayPalの約1.5〜2倍の工数。3段階処理（Customer→Card→Subscription）とロールバック処理、自前カードフォームUIが必要**（v3.7追記）
17. **手数料の見かけ上の数字（どちらも3.6%）より、40円固定料金の有無が手取り率の差を生む。Squareは固定40円なしのため¥1,980/月で¥40安い**（v3.7追記）
18. **「日本の30代サラリーマン」がターゲットの場合、PayPalよりカード直接入力のほうが離脱率が低い可能性。Squareを「サブ」ではなく「UI上重要な選択肢」として設計すべき**（v3.7追記）
19. **フロントエンドのプラン制限は「UX」、サーバー側のプラン制限は「セキュリティ」。両方必要。フロント側だけだとF12で突破される**（v3.8追記）
20. **多層防御（Defense in Depth）の原則: フロント→サーバー→データベースの3層で守る。1層だけ守る発想は危険**（v3.8追記）
21. **Supabaseの強み: RLS（Row Level Security）+ トリガーで、サーバーサイドコードを書かずにプラン制限を強制できる。これは個人開発SaaSの大きなアドバンテージ**（v3.8追記）
22. **JPY指定はSquare独自仕様: 通常Squareは最小通貨単位（USD=セント）で金額指定するが、JPYだけは円単位で指定する。amount=1980→¥1,980（amount=198000ではない）**（v3.9追記）
23. **Subscription PlanのID階層構造: SUBSCRIPTION_PLAN（親）+ SUBSCRIPTION_PLAN_VARIATION（子・月額/年額）の2階層。決済時に使うのは「Variation ID」のほうで、親IDは管理用**（v3.9追記）
24. **Webhook Edge Functionは `--no-verify-jwt` 付きデプロイが必須: Webhookは外部（Square）から呼ばれるためJWTを持たない。PAY.JPと同じパターン**（v3.9追記）
25. **Webhook側エラーは200で返す: SquareなどはHTTP 2xx以外を返すとリトライしてくる。エラーが起きてもログに残しつつ200で「受け取った」と返すのが鉄則**（v3.9追記）
26. **3段階処理 + 逆順ロールバック: Square 決済は Customer→Card→Subscription の3段階で、各段階の失敗時に「すでに作成したものを削除」する逆順ロールバックを実装。これによりゴミデータが残らない**（v3.9追記）

### Sandbox E2Eテストで判明した既存サブスクリプション削除手順

PayPal 等で既存サブスクが残っている場合、Square テスト時に UNIQUE制約違反で 3段階処理の Step4（DB INSERT）が失敗する。Sandbox 環境では以下で対処:

```sql
-- ⚠️ Sandbox/開発環境専用。本番では絶対に実行しない
DELETE FROM subscriptions;
```

リリース後の本番運用では、Edge Function の INSERT を UPSERT 化することで対応する（改善項目 #2）。
