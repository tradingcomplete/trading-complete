## 1. 背景・目的

### 1.1 背景

Trading Complete は本番リリース直前（2026年夏予定）の最終調整段階。Phase 10（Square実装）と Phase 11（サーバー側プラン制限）は完了したが、コンパナ社長より「**計算と分析が合っていないとリリースできない**」との要望を受け、リリース前最終監査として計算ロジック全体を 4 並列エージェントで監査した。

### 1.2 目的

| # | 目的 |
|---|---|
| 1 | 計算ロジックの**矛盾**・**バグ**・**仕様不明確箇所**をすべて洗い出す |
| 2 | 本番リリース前に修正必須な項目を確定する |
| 3 | 仕様明確化が必要な項目を社長判断に上げる |
| 4 | 修正後の検証方法（テスト計画）を確立する |

### 1.3 監査基本方針

- 修正は一切しない（観察のみ）
- ドキュメント駆動（仕様書 vs 実装コードの差分）
- 具体的な数値シナリオでシミュレーション
- 重要度別に分類（🔴 CRITICAL / 🟡 WARNING / 🟢 OK / 💡 提案）

## 2. 監査範囲・実施方法

### 2.1 監査ドメイン（4つに分割・並列実施）

| エージェント | ドメイン | 主要対象ファイル |
|---|---|---|
| 1 | pip計算 + 円建て損益 | TradeCalculator.js / YenProfitLossManager.js / TradeEntry.js / TradeExit.js |
| 2 | 統計・分析・レポート | StatisticsModule.js / ReportModule.js / AISummaryModule.js |

### 2.2 検証手法

各エージェントは以下を実施:
1. 関連する要件定義書を読み込み（仕様の正を理解）
2. 実装コードを読み込み
3. 仕様 vs 実装の差分を識別
4. 具体的な数値シナリオでシミュレーション
5. バグ・矛盾・不明確箇所をレポート

### CRITICAL #1: XAU/USD（金）の pip 計算が 100倍 ズレる

| 項目 | 内容 |
|---|---|
| **症状** | XAU/USD で pips が **100倍 過大表示** |
| **根本原因** | `js/part2/TradeCalculator.js` L22 / `js/part5/risk-ui.js` L229 / `YenProfitLossManager.js` L127-143 で `pair.includes('JPY') ? 100 : 10000` と二択判定。XAU は10000倍乗算してしまう |
| **影響範囲** | 金（XAU）取引ユーザーの全 pips 計算・円建て損益計算・統計・チャートすべて |
| **シミュレーション** | XAU/USD long 0.1ロット entry=2000.00 exit=2010.00 → 実装結果: 100,000 pips ❌ / 正解: 1,000 pips ✅ |
| **修正方針** | 通貨ペア別 pip 設定の共通ユーティリティを作成。`getPipMultiplier(pair)` 関数を `js/utils/` 等に1箇所定義し、3ファイルで共通利用。XAU/USD = 100倍、XAG/USD（銀）も追加検討 |
| **仕様明確化** | 不要（金は0.01単位で確定） |

---

### CRITICAL #2: クロスペア（EUR/USD等）の円建て損益がハードコード仮値

| 項目 | 内容 |
|---|---|
| **症状** | EUR/USD・GBP/USD など円建てクロスペア以外の通貨ペアで、円損益がハードコード値（1500円/pip 等）で計算され、実レートを反映しない |
| **根本原因** | `js/yen-profit-loss/YenProfitLossManager.js` L127-143 の `getPipValue()` メソッド |
| **影響範囲** | EUR/USD・GBP/USD・AUD/USD 等のドルストレート取引者の円建て損益・統計・チャート |
| **シミュレーション** | EUR/USD short 1ロット (100k通貨) +20pips、想定 USD/JPY=150 → 正しい計算: `20 × 100000 × 0.0001 × 150 = 30,000円` / 実装は `quote_currency_rate` を使わずハードコード値で計算 |
| **修正方針** | `getPipValue()` を改修し、quote_currency_rate を使った正規実装に置換。フォーマット: クロス円 = `pip × ロット数 × 1000円/pip` / ドルストレート = `pip × ロット数 × quote_currency_rate × 0.0001` |
| **仕様明確化** | quote_currency_rate の取得方法を確定（次項参照） |

### CRITICAL #4: 期間フィルタの entry_date vs exit_date 混在

| 項目 | 内容 |
|---|---|
| **症状** | 同一トレードが**異なる月に計上される**可能性。月跨ぎトレード（例: 3/31 entry, 4/2 exit）は ReportModule のセッション分析では3月、AISummaryModule の月次集計では4月、StatisticsModule では exit基準 |
| **根本原因** | `ReportModule.js` L1516（entry_date）/ `AISummaryModule.js` L63-65（exit_date）/ `StatisticsModule.js` L481-485（exit基準ソート）で基準が異なる |
| **影響範囲** | タブ間数字不一致。月次レポートと AI サマリーの数字が合わない（過去のバグ「タブ間数字統一17,557円差分」が別経路で再発） |
| **修正方針** | グローバル定数 `STATISTICS_PERIOD_BASIS = 'exit_date'`（または 'entry_date'）を定義し、全モジュールで統一参照 |
| **仕様明確化** | 🚨 **社長判断必須**: entry_date 基準 / exit_date 基準 のどちらか（推奨: exit_date = 決済日 = 損益確定日） |

### CRITICAL #5: セッション分析 DST 判定計算式に誤り

| 項目 | 内容 |
|---|---|
| **症状** | 3月初旬の NY時間判定が 1週間ズレる可能性 |
| **根本原因** | `js/part8_modules/ReportModule.js` L50-73 `#isUSDaylightSaving()` メソッド。DST開始日（3月第2日曜）の計算式 `(7 - firstDay) + 8` が条件によって第3日曜になる |
| **影響範囲** | 3月のセッション分析・NY時間集計が1週間誤って分類される |
| **シミュレーション** | 3月1日が月曜日(1) → 計算: `(7-1)+8 = 14日` ❌（第3日曜）/ 正解: 3月8日（第2日曜） |
| **修正方針** | DST 開始日: `firstSunday + 7` または `(8 - firstDay) % 7 + 8`（年により14日になる場合あり）を Date オブジェクトで正規化。同様に DST 終了日（11月第1日曜）も再検証 |
| **仕様明確化** | 不要（米国の DST ルールは確定） |

### CRITICAL #6: 期待値計算式の Pips/Yen 矛盾

### 元の選択肢一覧（参考・履歴）

### Q1: quote_currency_rate の取得方法（CRITICAL #3）
- A) ユーザー手動入力（決済時に画面入力）
- B) 外部API連携（リアルタイムレート自動取得）
- C) デフォルト USD/JPY=150固定 + ユーザー上書き可能

### Q2: 期間フィルタ基準（CRITICAL #4）
- A) entry_date 基準（エントリー日でその月に計上）
- B) exit_date 基準（決済日でその月に計上）← 推奨
- C) ユーザー設定で切り替え可能

### Q3: 投入資金の定義（CRITICAL #9）
- A) 入金合計（出金を含めない）← 推奨
- B) 入金 - 出金（純拠出額）
- C) 期初口座残高（年度別利益率と同じ）

### Q4: 締め後編集ポリシー（CRITICAL #7）
- A) 締め後は編集不可（締め解除機能なし）
- B) 締め解除機能を提供（管理者操作で再オープン可能）
- C) 締め後の編集は警告表示のみ（操作は許可）

### Q5: ChartModule の高DPI対応（WARNING W7）
- A) 即対応（Retina ユーザーへの品質向上）
- B) 後回し（パフォーマンス影響を懸念）

### Q6: 期待値計算の定義（CRITICAL #6 v1.1で要追加調査に降格）
- A) 仕様書 v1.16 通り `期待値 = 勝率 × 平均利益 - (1-勝率) × 平均損失`（厳密な期待値）
- B) 現実装（StatisticsModule.js L1046）の `(総利益 - 総損失) / 総トレード数 = 平均/トレード`（事後の平均値）
- C) 両方を別フィールドで表示
- 解説: A と B は理論的には同値（大数の法則）だが、サンプル数が少ないと誤差が出る
