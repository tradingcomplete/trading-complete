### 1. Phase 9 v1.0: ハッシュタグ動的可変運用(2026-04-24)

**背景**: 2026年 X(旧Twitter)アルゴリズム最新動向レポートを調査した結果、以下が判明した。

- 3個以上のハッシュタグは **-40% のリーチペナルティ**(国内実測・業界ベンチマーク一致)
- 2026年1月の xAI オープンソース化により、ハッシュタグは手動特徴量から削除され、Grok Transformer が文脈理解する設計に変更
- Musk が 2024/12 に「ハッシュタグ使用停止」を明言、2025/6 から広告では全面禁止
- 日本の FX 系著名アカウント(JFX小林社長・羊飼い・西原・田向等)は **ほぼハッシュタグなしで運用**

TCAX は毎投稿に最大3個のハッシュタグを自動付与していたため、リーチを抑制している可能性大。

**採用方針**: 選択肢D「動的可変(コンテキスト連動)」

- **平常時**: ハッシュタグ 0個
- **重要イベント当日**: 1個のみ付与(#FOMC / #日銀 / #雇用統計 等)

**Q1〜Q4 判断の確定**:

| Q | 判断 | 概要 |
|---|---|---|
| Q1 | 情報源 = 経済カレンダーシート | `getEconomicCalendar_('today')` の「★重要」マーク行を唯一の真実とする |
| Q2 | 時間窓 = 当日全投稿 | 日付一致のみ(時刻判定・前後2時間等は v1.0 では不採用) |
| Q3 | 複数候補 = 本文主題+優先度テーブル | 本文言及回数 ≥2 かつ単独最多で主題採用、それ以外は優先度順 |
| Q4 | #FX 固定 = 完全撤廃 | 平常時は完全0個 |

**実装**:

- 新規関数: `getImportantEventTag_(today)` / `determinePrimaryEventTag_(bodyText, eventTags)`
- 書き換え: `generateDynamicHashtags_(text, postType)` 全面刷新(旧108行 → 新42行)
- 削除関数: `countPairTag_()`(通貨ペア検出廃止)
- DRY 化: `determinePrimaryEventTag_` 内の本文カウントは既存 `countAnyTag_` を再利用

**優先度テーブル**(v1.0):

```

```

**対応イベントタグ**(v1.0・10種類):

| タグ | 本文キーワード |
|---|---|
| #FOMC | FOMC, パウエル |
| #日銀 | 日銀, BOJ, 植田, 金融政策決定会合 |

**検証**:

## v13.0 での大変更点(v12.10 からの変更)

### 1. 検証アーキテクチャの統廃合(4段 → 2段)

**v12.10 までの問題**: 「問題が出るたびに検証層を追加」を繰り返した結果、4段検証が重複動作。

**v13.0 の解決**:
- Stage 1 総合レビュー(Claude + Web検索 最大5回・max_tokens 8192)
- Stage 2 修正適用(論理→事実→NG削除→WARN→品質)
- Claude API 呼び出し: 7回 → 4回(-43%)
- 実行時間: 4分 → 2分18秒(-45%)

### 2. プロンプト構造の根本再設計(Phase 6)

**v12.10 までの問題**: プロンプトが 24,000〜29,000字に膨張。Claude の attention が希薄化して口調・論理が甘くなる。

**v13.0 の解決**:
- 統合案C(因果関係3→1): -180字・-2セクション
- S1 Before/After 実例セクション新設: few-shot誘導で口調学習
- S2 【★★★最終確認】再構成
- S3 統合案D(事実論理3→1): -2セクション
- S4 統合案A(フォーマット5→1): -4セクション
- S6 ダウ理論(日足・週足 SH/SL)を投稿プロンプトに注入
- プロンプト: 24,663字 → 20,753字(-4,700字・-18.5%)

### 3. 構造整合性の3段防御(v13.0.8)

**v12.10 までの問題**: 鉤括弧「」『』の中に「→」や改行が入り論理破綻する投稿が生成される。検証でも検出されない。

**v13.0 の解決**: 3層防御
- 段階1(promptBuilder.gs): 鉤括弧は同じ行で開閉のルールを明示
- 段階2(validationV13.gs): Q8 構造整合性チェック新設 + detectStructureBreaks_ 機械チェック
- 段階3(qualityReview.gs): trimToCharMax_ に鉤括弧保護ロジック

### 4. 口調の根本改善(few-shot学習)

**v12.10 までの問題**: 禁止指示を大量に並べても Claude の口調が硬い(アナリスト調・報告調)。

**v13.0 の解決**: Before/After 実例8組 + コンパナらしさの核6原則
- 漢語より和語、名詞化より動詞化
- 「〜ですね」「〜かなと思います」「〜とこですね」で文末崩し
- 文末は必ず動詞で完結(「〜かなと。」「〜とこ。」禁止)

   - charMax を超えたら末尾から削る

   ↓

   ↓

## 5. v13.0 検証アーキテクチャの詳細

### 5-1. Stage 1 総合レビュー

**呼び出し**: Claude + Web検索最大5回・max_tokens 8192

**検出する項目**(Q1〜Q8・合計8観点):

```
A. 品質チェック(Q1-Q5):
   Q1 重複・同語反復
   Q2 冗長表現・AI感
   Q3 文章の自然な流れ
   Q4 文字数(charMin〜charMax)
   Q5 ★コンパナ口調(v13.0.7 厳格化):
     ・アナリスト調/AI翻訳調を error 検出
     ・文末未完結(「〜かなと。」「〜とこ。」)を error 検出

   Q6 事実検証(下記B)
   Q7 絵文字行の体言止め

   ★Q8 構造整合性(v13.0.8 新設):
     (a) 鉤括弧「」『』の対応チェック
     (b) 鉤括弧の中に改行・→・絵文字が入っていないか
     (c) →行が前の行から意味的に継承されているか

B. 事実検証(Q6):
   判定優先順位: (1)確定データ > (2)Web検索 > (3)本文 > (4)内部知識
   時間軸主張(「週中高値」「急落率」等)は検証不能なので factErrors に入れない

C. 論理整合性(Q6.5):
   事実は正しいが論理矛盾する投稿を検出
   例: 「予想は外れた」と「着地点は合っていた」の同居
   例: 「リスクオン」と「円高進行」の同居
```

**出力JSON構造**:
```json
{
  "quality": [...],       // Q1-Q5, Q7, Q8 の問題
  "factErrors": [...],    // Q6 事実誤り
  "logical": [...],       // Q6.5 論理矛盾
  "webClaims": [...]      // Web検証結果
}
```

### 5-2. Stage 2 修正適用

問題の種類ごとに異なる修正戦略:

| 修正種 | 対象 | 戦略 |
|---|---|---|
| 修正1 | 論理矛盾(logical) | Claude に指示して直させる |
| 修正2 | 事実誤り(factErrors) | 機械置換(原値で上書き) |
| 修正3 | Web検証NG(webClaims reject) | 該当行を削除 |
| 修正4 | Web検証WARN | Claude に指示して修正 |
| 修正5 | 品質 error(quality error) | Claude に一括修正依頼 |

**修正時のプロンプトには必ずキャラクターシート全文を末尾注入**(v13.0.2 の学び: 口調を守るため)。

### 5-3. キャラクター注入戦略の変遷

| 版 | 戦略 | 結果 |
|---|---|---|
| v13.0 初期 | 注入なし | 口調が硬い |
| v13.0.1 | 3関数すべての冒頭に注入 | **失敗**(468字・注釈混入・文脈崩壊) |
| v13.0.2 | 品質修正(applyQualityFix_)のみ・末尾に注入 | **成功**(306字・ブロック構造完結) |

学び: **注入箇所は1箇所のみ・末尾に・SSOT**

### 5-4. 二重実行防止(v13.0.4)

**問題**: Phase A → Phase B の再実行で v13.0 が2回動作

**解決**: ScriptProperties に `V13_LAST_EXECUTED_<postType>` を記録。180秒以内に再呼び出しされた場合はスキップ。

### 5-5. JSON堅牢化(v13.0.5)

**問題**: 日本語プロンプト内の鉤括弧が JSON の引用符と衝突してパースエラー

**解決**: プロンプトで JSON フィールド値の日本語を「〜」で囲むルール明示・Stage 1 失敗時は文字数保証のみ実行するフォールバック

### 5-6. Q5 口調判定厳格化(v13.0.6〜v13.0.7)

**v13.0.6 で追加**:
- アナリスト調・AI翻訳調を error 判定

**v13.0.7 で追加**:
- 「〜かなと。」「〜とこ。」の文末未完結を error 判定
- 「〜かなと思います。」「〜とこですね。」で動詞完結するよう修正指示

### 5-7. 構造整合性の3段防御(v13.0.8)

**段階1** promptBuilder.gs L1571〜:
```
■ ★鉤括弧の整合性(絶対遵守・構造破綻防止):
  ・鉤括弧「」『』は必ず同じ行の中で開いて閉じろ
  ・鉤括弧の中に「→」「改行」「絵文字」を入れるな
```

**段階2-a** validationV13.gs buildStage1Prompt_ に Q8 追加

**段階2-b** validationV13.gs 末尾に detectStructureBreaks_ 関数:
- 鉤括弧の数の対応チェック
- 鉤括弧の中身チェック(改行・→・絵文字)
- 問題があればログ警告

**段階3** qualityReview.gs trimToCharMax_ に鉤括弧保護:
- 行削除で鉤括弧が破綻する場合、削除を諦める
- または鉤括弧ブロック全体を削除

## 14. 設計の原則(今後開発する時の指針)

1. **小手先修正の罠を避ける**
   - 問題が出たら「もう1段検証を足す」「ルールを1行追加」は肥大化の元凶
   - 根本原因を突き止める・構造を再設計する

2. **生成時点で対処する**
   - 検証で直すより、生成時点で間違えないプロンプトを作る
   - Before/After 実例・明確な制約が検証より効く

3. **few-shot > 禁止指示**
   - LLMは「こう書け」の実例に強く反応する
   - 「〜するな」の禁止を並べるより、OK例を1つ見せる方が効果的

4. **プロンプト末尾が最強**
   - Claude は最後に見た指示を最優先する
   - 重要な指示は末尾に置く(★★★最終確認・Before/After)

5. **SSOT(Single Source of Truth)**
   - キャラクター定義はスプレッドシートの「キャラクター」シートが唯一
   - コード側で重複注入するとかえって薄まる

6. **並行稼働フラグで即ロールバック**
   - 新機能は必ずフラグで切り替え可能に
   - 問題発生時はフラグOFFで即座に前バージョンに戻せる

7. **疲れたら止まる**
   - 疲労した状態でコードに変更を加えるとバグを生む
   - 検証・シミュレーション・一晩寝かせる判断を優先

## v14.2 期限切れバグ修正(2026-04-28)

### 概要

sheetsManager.gs `expireOldDrafts()` の対象を「下書き」のみから「下書き」+「承認待ち」に拡張。manualモードで承認されない「承認待ち」がシート上に永久残留する中優先既知問題を解消。

### 背景

- v12.7 以来 `expireOldDrafts()` は「下書き」ステータスのみを対象に DRAFT_EXPIRY_MINUTES(90 分)経過で自動的に「期限切れ」化していた
- manualモードでは Phase C 完了後にステータスが「承認待ち」となり、コンパナさん承認まで滞留する設計
- 承認されないまま放置された「承認待ち」が判定対象外となり、シート上に永久残留する問題が発生
- 引き継ぎプロンプトの「既知の未対応問題」(中優先)として記録されていた

### 実装

- 変更ファイル: sheetsManager.gs(L515 付近)
- ステータス判定の拡張: `status === '下書き'` 単独 → 「下書き」または「承認待ち」を早期 continue 化で判定
- 閾値: DRAFT_EXPIRY_MINUTES(90 分)を両ステータス共通で適用

## v14.2 テーマブロック原則導入(2026-04-29)

### 概要

- 段落 5 で再び豪ドル円(テーマAに逆戻り)

テーマA が冒頭と末尾に分断され、テーマB/C が薄っぺら締めになる構造。コンパナさんから「軸数の問題ではなく、テーマごとに話が分かれていないことが本丸」との指摘。
