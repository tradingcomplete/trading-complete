#### 実装内容

**handleError_ 共通ヘルパー**(utils.gs に追加):

```javascript
/**
 * エラーを段階別に処理
 * @param {string} severity - 'critical' / 'warning' / 'info'
 * @param {string} context - 発生箇所(ファイル名:関数名)
 * @param {Error} error - エラーオブジェクト
 * @param {Object} [meta] - 追加情報
 */
function handleError_(severity, context, error, meta) {
  var timestamp = new Date().toISOString();
  var message = '[' + severity.toUpperCase() + '] ' + context + ': ' + error.message;
  
  if (severity === 'critical') {
    console.error(message);
    throw error;
  }
  
  if (severity === 'warning') {
    console.log('⚠️ ' + message);
    try {
      var ss = SpreadsheetApp.openById(getApiKeys().SPREADSHEET_ID);
      var sheet = ss.getSheetByName('エラー隠蔽ログ');
      if (sheet) {
        sheet.appendRow([timestamp, severity, context, error.message, JSON.stringify(meta || {})]);
      }
    } catch (logErr) {
      console.log('⚠️ handleError_ ログ書込失敗(続行): ' + logErr.message);
    }
    return;
  }
  
  // info
  console.log('ℹ️ ' + message);
}
```

**段階分類の指針**:

| 段階 | 該当例 |
|---|---|
| **Critical** | 確定データ取得失敗(collectAnchorData_ の根幹)・API 認証失敗・スプシ書込失敗・定数未定義参照エラー(事件14 類似) |
| **Warning** | 補助ログ書込失敗・アノマリー判定失敗・キャッシュ失敗・ニュース取得失敗(コア処理に影響しない) |
| **Info** | 想定内の例外(データなし・条件未達・ユーザー操作待ち) |

**移行対象の明示**(優先 3 ファイル):

```
validationV13.gs:
  - Stage 1 呼び出し失敗 → Critical
  - Stage 2 修正失敗 → Warning(他の修正は試みる)
  - 対話型検証ログ書込失敗 → Warning

main.gs:
  - Phase A の本体失敗 → Critical
  - Phase B/C スケジュール失敗 → Warning
  - 投稿後ログ記録失敗 → Warning

anchorDataCollector.gs:
  - 始値取得・通貨強弱取得失敗 → Critical
  - 政策金利・要人・経済カレンダー・継続中事象の取得失敗 → Warning(代替値で続行)
  - 自動検証エラー → Info
```

#### 動作確認シナリオ

| # | シナリオ | 期待 |
|---|---|---|
| 1 | Critical: 始値取得失敗 | エラー throw・Phase A 停止・Gmail に通知 |
| 2 | Warning: 経済カレンダー取得失敗 | 「エラー隠蔽ログ」シートに記録・代替値で続行 |
| 3 | Info: 継続中事象が空 | console.log のみ・続行 |
| 4 | 事件14 類似: 定数 PAST_TENSE_KEYWORDS 未定義 | Critical として throw |

#### 観察期間と昇格判定

初期運用で Critical 判定の妥当性を評価するため:

- Day 1-7(第 1 週): 全 Critical 候補を `handleError_('warning', ...)` で仮運用・挙動観察
- Day 8 に「エラー隠蔽ログ」を確認し、1 週間で実際に発生したエラーを確認
- 実運用で真に止めるべきだったエラーのみ Critical に昇格

#### ロールバック手順

- 本番停止発生時: Apps Script 画面で該当ファイルを Git の前バージョン(ブランチで管理)から復元
- 段階的撤退: Critical → Warning に戻すだけ(throw を削除)

#### リスク

- Critical 判定を間違えると本番が止まる
- **対策**: 第 1 週を Warning 仮運用 → 観察 → 昇格 の 2 段階プロセス
- 対象ファイル外のエラーは従来通り catch-skip のまま残る(残存リスク・v2.0 で検討)

#### 実装規模

- utils.gs: +60行(handleError_ 関数)
- validationV13.gs / main.gs / anchorDataCollector.gs: 各 5-10 箇所の catch 節を handleError_ に置換(合計約 70 行)

### 2-4. T1-D: 承認メール却下理由タグ化

#### 背景

現在の承認メール(manual モード)には「✅承認する / ❌中止する」のリンクがある。却下時の理由は記録されていない。Phase 5 のプロンプト自動進化は「人間が却下した理由」というラベル付き教師データが特に価値高いが、現状は情報が失われている。

- 変更ファイル: `approval.gs`(承認メール生成+ hook URL 処理)
- 追加シート: 「却下理由ログ」新規作成

#### 実装内容

承認メールの却下部分を以下のバリエーションに拡張:

| タグ | 意味 |
|---|---|
| ❌ 数値誤り | レート・変動率・pips 等の数値が間違っている |
| ❌ 固有名詞誤り | 要人名・機関名・イベント名が間違っている |
| ❌ 論理矛盾 | 因果・逆説・時系列の論理が破綻 |
| ❌ 事実誤り(その他) | 上記以外の事実間違い |
| ❌ トーン不適 | コンパナらしくない・説教臭い・AI っぽい |
| ❌ タイミング不適 | 投稿タイプと主題が合わない(MORNING なのに NY 前の話 等) |
| ❌ 重複 | 過去投稿と内容が重複している |
| ❌ 内容薄い | 主張が弱い・深みがない・独自性欠如 |
| ❌ その他 | 自由記述欄 |

各タグは URL パラメータで区別:
- 現状: `https://.../approve?id=xxx&action=reject`
- 新: `https://.../approve?id=xxx&action=reject&reason=tone`

approval.gs の doGet 拡張で reason パラメータを受け取り、「却下理由ログ」シートに記録:

| 列 | 項目 |
|---|---|
| A | 却下日時 |
| B | 投稿ID |
| C | 投稿タイプ |
| D | 却下理由タグ |
| E | 投稿本文(全文) |
| F | 投稿時に発生した修正件数 |
| G | 自由記述(任意) |

#### 動作確認シナリオ

| # | シナリオ | 期待 |
|---|---|---|
| 1 | 承認メールで「❌ 数値誤り」をクリック | 却下理由ログに reason=number で記録 |
| 2 | ❌ その他 + 自由記述 | コメント欄に本文が保存される |
| 3 | 複数の却下理由が該当する場合 | 主要 1 つを選択(UI は単一選択) |

#### リスク

- 却下理由のカテゴリが実態と合わない(実運用で必要カテゴリが異なる可能性)
- **対策**: 2 週間運用後にカテゴリ再調整の余地を残す(v1.1 で見直し)

function computePatternHash_(errorType, keywords) {

}

### 3-7. T2-F-2: 日内コンテキスト注入を投稿生成プロンプトにも拡張(★T2-F 部分効果の補強)

#### 背景

T2-F 実装直後の本番観察(2026-04-27 18:23 testDraftMode)で、**部分的な効果しか出ていない** ことが判明:

##### 観察結果

| 項目 | 結果 | 評価 |
|---|---|---|
| ホットトピック選定 | 「豪ドル独歩高の真因」(豪ドル米ドル) | ✅ 多様化 |
| 投稿本文 | 米イラン暗礁 + 豪ドル米ドル + **明日の植田総裁会見** | ❌ 日銀混入 |
| Stage 1 Q2 重複警告 | 「5 テーマすべてが既存 LUNCH/MORNING と重複」 | ❌ 再発 |

ホットトピック選定段階では多様化したが、**投稿生成段階で日銀情報が再注入され**、結果として最終本文は依然として既出主題と重複した。

##### 根本原因

投稿生成プロンプトには T2-F のヒント(本日の既出投稿)が **届いていなかった**。Claude は以下のセクションを読んで日銀を含めてしまう:

1. 【経済カレンダー】に日銀会合 ★重要 が注入
2. 【明日の重要イベント】に日銀 4/28 が明記
3. 【今日書くべきトピック】(豪ドル)と他セクション(日銀)の両方を読んだ結果、両方を本文に含める判断

つまり T2-F は selectHotTopic_ の prompt にだけヒントを入れたが、**投稿生成 prompt に同じヒントが入っていない**ため、二段階目で重複が再発する構造。

#### 変更対象

- ファイル: `promptBuilder.gs`
- 該当箇所: dynamicPart 構築時、formatHotTopicForPrompt_(hotTopic) 注入の直後

#### 実装内容

`_priorPosts` 変数(T2-F で既に取得済み)を再利用して、投稿生成プロンプトにも同じヒントを注入する。

```javascript
if (hotTopic) {
  dynamicPart += formatHotTopicForPrompt_(hotTopic);

  // ★v14.2 T2-F-2(2026-04-27): 投稿生成プロンプトにも priorPosts ヒント注入
  if (_priorPosts && _priorPosts.length > 0) {
    dynamicPart += '\n【本日の既出投稿(本文の主題重複を避けよ)】\n';
    for (var ppi = 0; ppi < _priorPosts.length; ppi++) {
      var pp = _priorPosts[ppi];
      if (!pp) continue;
      var ppSummary = String(pp.text || '').substring(0, 120).replace(/\n/g, ' ');
      dynamicPart += '- ' + (pp.time || '??:??') + ' ' + (pp.type || '?') + ': ' + ppSummary + '\n';
    }
    dynamicPart += '\n';
    dynamicPart += '★重要(投稿の主題重複回避・最優先ルール):\n';
    dynamicPart += '- 上記投稿で既に書いた主題・切り口・通貨ペアの繰り返しは禁止\n';
    dynamicPart += '- 当日最大材料(日銀・FOMC等)は 1 日 2 回まで触れる可・3 投稿目以降は必ず別の角度を探せ\n';
    dynamicPart += '- 「タカ派/ハト派」「円買い/円売り」の二項対立を 2 回以上書いた場合、3 回目以降は別フレーミング\n';
    dynamicPart += '- 経済カレンダーに日銀があっても、既に MORNING/TOKYO/LUNCH で書いた場合 LONDON では別主題が原則\n';
    dynamicPart += '\n';
  }
}
```

| # | シナリオ | 期待 |
|---|---|---|
| 1 | MORNING(初回) | priorPosts=[] でヒント注入なし・従来通り |
| 2 | TOKYO(MORNING で日銀)で日銀以外を選定 | 本文も日銀を避ける |
| 3 | 2026-04-27 を再現テスト(5 投稿目) | 本文に「植田会見」「タカ派/ハト派」が混入しない |
| 4 | 経済カレンダーに日銀 ★重要 ありの 4 投稿目 | 本文で日銀に触れない or 別フレーミング |

#### リスク

- **過度な多様化**: 当日の真の最大材料を完全に避けるリスク
  - **対策**: 「1 日 2 回まで触れる可」を明示

### 4-2. T3-B: Haiku による pre-screen 実験

#### 背景

Sonnet Stage 1(8192 tokens)は高コスト。Haiku(Sonnet 比 約 20% コスト)で「明らかにおかしい」箇所を先行検出し、Sonnet の負荷を下げる構造転換が可能か実験する。Attention 分散対策(事件11)としても、小さい範囲で見るので見落とし減の副次効果あり。

#### 変更対象

- 新規ファイル: `haikuPreScreen.gs`(新規)
- 変更ファイル: `validationV13.gs`(Stage 1 前段に Haiku 呼び出しを挟む)

#### 実装内容

処理フロー:

```
[従来] 投稿本文 → Sonnet Stage 1(大きく見る) → 結果
[新]   投稿本文 → Haiku pre-screen(疑問点抽出) 
       → Sonnet Stage 1(Haiku の疑問点のみ深く見る) → 結果
```

Haiku pre-screen 関数:

```javascript
function haikuPreScreen_(text, anchorData) {
  // Claude Haiku API 呼び出し
  // プロンプト: 「以下の投稿から数値・日付・固有名詞を抽出せよ」
  // 出力: {numbers: [...], dates: [...], names: [...], suspiciousClaims: [...]}
  // suspiciousClaims はさらに Sonnet で深堀り対象
}
```

Sonnet Stage 1 は「Haiku が抽出した suspiciousClaims のみ」を確認モードに変更(max_tokens を 4096 に削減)。

#### 動作確認シナリオ(A/B テスト形式)

| # | シナリオ | 期待 |
|---|---|---|
| 1 | 通常投稿で従来フロー | 誤り検出 X 件 |
| 2 | 同じ投稿で新フロー | 誤り検出 X 件 + 処理時間・コストを記録 |
| 3 | 2 週間で 50 投稿を A/B 比較 | 検出漏れ率・誤検出率・コストを比較 |

#### 判定基準(v1.1 で緩和)

| 指標 | 合格条件 |
|---|---|
| 検出漏れ率 | 従来比 +5% 以内 |
| 誤検出率 | 従来比 -10% 以上 |
| コスト | 現状維持または削減(v1.0 の -10% 必須から緩和) |

v1.0 では「コスト -10% 以上」を必須としていたが、Sonnet の max_tokens を 8192 → 4096 に半減することで理論上はコスト削減可能なものの、実測次第で Haiku 追加分との相殺で変動する。改善できれば良く、必ずしも大幅削減が必要ではないため「現状維持または削減」に緩和。

検出漏れ率・誤検出率が合格かつコスト悪化なしなら本番採用、それ以外は撤退。

#### リスク

- Haiku の性能不足(日本語理解・構造抽出)
- **対策**: A/B テストで定量評価・未達なら即撤退

#### 実装規模

- haikuPreScreen.gs: +150行
- validationV13.gs: +40行(呼び出し追加)
- A/B テスト: 2 週間観察

## 7. リスクと対策

### リスク 1: T1-B の Critical 判定で本番停止

**内容**: 段階化で Critical 判定を間違えると Phase A/B が止まり投稿できなくなる。

**対策**:
- 初期の 1 週間は全て Warning として扱う観察期間を設ける
- 観察期間後に実データで Critical 判定の妥当性を再評価
- 本番停止時のロールバック手順を事前準備

### リスク 2: T1-C/T2-D のログ書込で処理が重くなる

**内容**: Phase A/B の各ステップで appendRow が追加され、処理時間が延びる。

**対策**:
- 各 appendRow は約 100ms 未満・1 投稿で 5-6 件程度
- 合計 500-600ms の増加・Phase A/B の 2-3 分処理時間には影響軽微
- GAS 6 分上限には十分余裕あり

### リスク 3: T3-B の Haiku が性能不足で撤退

**内容**: Haiku の日本語理解・構造抽出が不十分で、検出漏れが増える。

**対策**:
- A/B テスト 2 週間で定量判定
- 撤退時は haikuPreScreen.gs を削除・validationV13.gs の呼び出し部分を revert
- 撤退判断のコスト: 実装 2 日 + 観察 2 週間 = 低コスト

### リスク 4: Phase 5 着手条件が予定より遅れる

**内容**: ログ蓄積が日 5 件ペースより遅い場合、2026-05-20 目標に間に合わない。

**対策**:
- 日次ログ件数をダッシュボード(T2-C)で可視化
- 遅延なら Tier 3 開始を先延ばし(Phase 5 本体着手日をスライド)
- 焦って Phase 5 を前倒しするのは避ける(IV-14 肥大化ループ回避)

### リスク 5: 要件定義書の文章とコードの矛盾(IV-15 の再発)

**内容**: 本書自体が IV-15 教訓の対象になりうる。

**対策**:
- 本書の「実装内容」セクションは関数名・ファイル名・行数の具体表記を徹底
- 「保持」「使用」と書いた関数は必ず仕様コードでも呼び出す形に統一
- 実装後の grep チェックを必須化
