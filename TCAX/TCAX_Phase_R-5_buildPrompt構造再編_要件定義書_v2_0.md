## 1. 背景と目的

### 1-1. 解決したい問題

Phase R-4 完了後も残る以下の問題。

1. **静的/動的の交互配置**: プロンプトが「静的→動的→静的→動的」の順で、Prompt Caching で長い連続静的ブロックを確保できない
2. **最終確認の位置**: 末尾(attention最強位置)にあり、静的ブロックへの移動に工夫が必要
3. **Phase 8 の前提未整備**: キャッシュヒット率50%程度に留まる(理想は70%以上)

### 1-2. 目的

- プロンプトを「静的プレフィックス → 動的サフィックス」の順に再編
- Phase 8(Prompt Caching)導入時のキャッシュヒット率を72%以上に引き上げる土台整備
- 末尾「最終確認」の attention を維持しつつ静的化する工夫の実装

### 1-3. 非目的

- プロンプト内容の変更(Phase R-4 で完了済み)
- Claude API呼び出しロジックの変更(Phase 8 で対応)
- ローカル変数の命名変更や関数分解(Phase R-2 の方針に従い内部分解しない)
- キャラクターシートへの言及(SSOT 遵守)

## 2. 現状分析(v1.0 から継承)

### 2-1. buildPrompt_ の現状規模(Phase R-4 完了後)

- 総行数: 624行
- prompt += 出現回数: 81箇所
- ローカル変数宣言数: 75個
- buildPrompt_ 内から呼ばれる外部関数: 約30個

### 2-2. ローカル変数の依存関係(v2.0 で追加確認)

75個のローカル変数を全て精査した結果、**全ての動的データは「動的データ収集フェーズ」に集められ、その後「動的文字列組立フェーズ」で使える**ことが確認できた。静的/動的の分離は可能。

## 3. 設計方針(v2.0 で最適解確定)

### 3-1. アプローチの採用(Q1 確定)

**アプローチA: 二段階組み立てを採用**

```javascript
var staticPart  = '';  // Prompt Caching 対象
var dynamicPart = '';  // 毎回変わる部分

// [1] 静的プレフィックス構築(約13,000字)
// [2] 動的データ収集(既存ロジックのまま)
// [3] 動的サフィックス構築(約7,500字)
// [4] 末尾再確認マーカー(約80字)
// [5] 連結

var prompt = staticPart + dynamicPart;
return prompt;
```

**採用理由**
1. ロジック変更最小(データ収集はそのまま、文字列組立の順序だけ変える)
2. ローカル変数75個の参照タイミングリスク低(変数は生成タイミングのまま)
3. ロールバック容易
4. Phase 8 への接続明確(staticPart がそのままキャッシュ対象)

### 3-2. 原則

1. プロンプト内容を変えない(Phase R-4 で完了済みのため)
2. 文字列組立の順序だけを変更する
3. ローカル変数の生成タイミングは変えない
4. Claude API 呼び出しロジックには触れない
5. テストは全5タイプで完走確認

### 4-3. 末尾再確認マーカーの設計(Q2 確定)

v1.0 の4項目版を廃し、C案(2項目版)を採用。

**採用する内容**

```
【★再確認(最終チェック)】
1. 冒頭のルール・トーン・フォーマットを守れ
2. 文末は動詞完結(「〜かなと。」で切るな)
```

**変更理由**
- 4項目は attention 分散の可能性
- このマーカーの役割は「動的データに引っ張られた Claude を冒頭ルールに引き戻す watch 役」
- 詳細指示は冒頭【★★★最終確認】にあるため、ここでは抽象的な watch で十分
- 項目2は事件9(文末未完結「〜かなと。」の再発)の教訓を踏まえ具体的に残す

### 4-4. 情報ソースの配置(Q3 確定)

【情報ソース】(getReferenceSources_ の出力)を動的サフィックスから静的プレフィックスに移動する。

**移動理由**
- 内容は「Bloomberg、日経、みんなのFX等のサイト一覧」で完全固定
- Sheets「参照ソース」シート編集頻度は低い
- 論理的に「参考URL一覧」として投稿タイプ指示の近くにある方が自然

### 4-5. postPrompt の配置(Q4 確定)

getPostPrompt_() は Sheets「投稿プロンプト」シートから取得する「投稿タイプ別プロンプト」。これを静的ブロックに配置する。

**配置理由**
- 1投稿内のリトライ(平均1.3回)でキャッシュヒット発生
- 30%の投稿で1回以上のキャッシュヒット期待
- Sheets編集頻度は低いため影響軽微
- 16タイプ分のキャッシュが自然に分かれる(問題なし)

## 5. 実装手順(v2.0 で確定)

**Task 4: 動的サフィックスの構築**

動的データを dynamicPart に順番に追加。

```javascript
dynamicPart += '\n【★再確認(最終チェック)】\n';

dynamicPart += '2. 文末は動詞完結(「〜かなと。」で切るな)\n';
```

**Task 6: 連結処理**

```javascript
var prompt = staticPart + dynamicPart;
return prompt;
```

**Task 7: 動作確認**

- 構文チェック
- 出力プロンプトの構造確認(静的/動的の分離)
- testDraftMode で全5タイプ完走確認

### 5-3. 実装の難所と対処

**難所1: postPrompt の取得タイミング**
- getPostPrompt_(postType) はstaticPart組立時に実行
- Sheets への1回のアクセスを前倒しする(問題なし)

**難所2: getTCOverview() の条件**
- 現状 `if (tcTypes.indexOf(postType) !== -1)` で分岐
- staticPart組立時に同じ条件判定を適用

**難所3: ホットトピック選定の Claude API 呼び出し**
- 動的データ収集フェーズで実行
- 結果は formatHotTopicForPrompt_() で整形して dynamicPart に追加

**難所4: 問いかけ誘発の確率的注入**
- Math.random() < 0.5 の条件判定
- 動的サフィックス内の条件分岐として維持

### 5-4. ロールバック計画

- Phase R-5 実装前のバックアップ保持
- 各 Task 完了時に構文チェック・中間コミット相当
- testDraftMode 完走失敗時は即ロールバック
- 品質低下時も即ロールバック

## 7. リスクと対策

### 7-1. リスク

1. **ローカル変数の参照タイミングがずれる**
   - 対策: データ収集フェーズを明確化し、dynamicPart への追加をその後に実行
2. **静的/動的の分離が完全でない**
   - 対策: 条件分岐含む全関数出力を「静的」扱いにし、Phase 8 でタイプ別キャッシュを許容
3. **末尾移動により「最終確認」の attention が弱まり品質低下**
   - 対策: 末尾再確認マーカー(2項目版)を設置、静的ブロック末尾付近に【★★★最終確認】を配置
4. **Phase R-4 との整合性**
   - 対策: Phase R-4 で統合した構造は維持、本フェーズはさらに順序再編のみ

### 7-2. ロールバック計画

- Phase R-5 実装前のバックアップを保持
- 各 Task 完了時に構文チェック・中間コミット相当で保存
- testDraftMode 完走失敗時は即ロールバック

## 8. 確定した判断事項(v2.0 で確定)

### 8-1. Q1: アプローチA採用 → **採用**

**理由**: B(prompt += 直接入れ替え)・C(関数分解)よりリスク低。ロジック変更最小で、ローカル変数75個の参照タイミングは維持される。Phase R-2 で見送った「内部分解」の方針とも整合。

### 8-2. Q2: 末尾再確認マーカー → **C案(2項目版)を採用**

**理由**: 4項目版は attention 分散の可能性。このマーカーの役割は watch 役で、詳細指示は冒頭【★★★最終確認】にあるため、抽象的な watch で十分。項目2は事件9(「〜かなと。」再発)の教訓を具体的に残す。

### 8-3. Q3: 情報ソース静的化 → **採用**

**理由**: 完全固定内容(Sheets「参照ソース」シート)。Sheets編集頻度は低い。論理的に「参考URL一覧」として投稿タイプ指示の近くが自然。

### 8-4. Q4: postPrompt 静的化 → **採用**

**理由**: 1投稿内リトライ(平均1.3回)で30%キャッシュヒット発生。Sheets編集頻度は低く影響軽微。16タイプ分のキャッシュが自然に分かれる(問題なし)。
