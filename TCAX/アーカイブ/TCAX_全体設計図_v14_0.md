## 1. システム概要

```
GAS + Claude + Twelve Data で1日6回の市場投稿を自動生成・投稿。
指標30分前に動的投稿。週末は心得枠4投稿。市場系にはAI画像自動生成。

サービス:
  GAS              自動実行エンジン(無料)
  ★Claude Sonnet 4.6  ★主力: 投稿生成 + 市場分析 + 総合検証(v13.0 統合検証)
  Gemini 2.5 Pro   画像生成のみ(v13.0 で役割縮小)
  Twelve Data API  為替レート(7ペア + BTC + Gold)
  Google Sheets    データ保存(23シート)
  X API v2         投稿

v13.0 の重要な変更:
  - Claude が「分析 + 生成 + 検証」の3役を一貫して担う
  - Gemini は画像生成のみ(因果関係の方向性ミス問題を回避)
  - 旧 qualityReview.gs の大部分は v13.0 で置換(Phase 5で削除予定)
```

④ ★v13.0 総合検証(validationV13.gs) ←★ここが最大変更★
   
   USE_V13_VALIDATION=true の場合:

## 4. v13.0 アーキテクチャの核心

### 4-1. 「Claude 一貫主導」の完成

v12.4 で「ライティングと修正は Claude」の方向性を出し、v13.0 で全面完成:

```
v12.10まで:
  生成: Claude / 分析: Claude / 検証: Claude + Gemini(factCheck)
  → Gemini が時々逆方向の指摘をして事故(因果関係のミス)

v13.0:
  生成: Claude / 分析: Claude / 検証: Claude のみ
  → 責任の所在が明確・Web検索で事実確認も Claude が実施
  → Gemini は画像生成のみ
```

### 4-2. 「生成時点で対処」の原則

```
v12.10まで: 検証で直す(後追い型)
  → 検証ロジックが肥大化 → 重複 → API呼び出し増 → 時間増

v13.0: 生成時点で間違えないプロンプト(予防型)
  → Before/After 実例で口調学習
  → 鉤括弧ルールで構造破綻防止
  → ダウ理論で時間軸明確化
  → 結果: 検証で修正する回数が減少
```

### 4-3. 「few-shot > 禁止指示」の発見

```
v12.10まで: 「〜するな」「〜禁止」を積み重ね
  → プロンプトが肥大化 → Claude の attention 希薄化 → 効果薄

v13.0: 「こう書け」の具体的実例(Before/After)
  → LLM は few-shot に強く反応
  → プロンプトが減ってもかえって品質向上
```
