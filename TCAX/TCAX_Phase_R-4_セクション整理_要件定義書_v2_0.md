## 5. Stage 1.5: 口調ルールの独立化(v2.0 新規)

### 5-1. 現状

buildFormatRules_ 1585行:

```
9. 口調:冒頭【ペルソナ】【発信の原則と口調】が唯一の基準。
   アナリスト調(〜である/〜と見られる/推察される)は禁止。
   同じ語尾2回連続禁止。
```

これは「最優先ルール」セクションの9番目として固定されている。

### 5-2. 問題点

1. **番号依存**: 将来の統合で番号ズレが発生する可能性
2. **埋没位置**: 口調は最重要要素なのに9番目(末尾)に位置
3. **attention 減衰**: Claude が 9項目を順に処理する中、9番目では注意が薄れる

### 5-3. 変更案

```
★口調の唯一の基準(最優先ルールの上位原則)
  冒頭【ペルソナ】【発信の原則と口調】が唯一の基準。
  アナリスト調(〜である/〜と見られる/推察される)は禁止。
  同じ語尾2回連続禁止。
```

番号なしで「最優先ルールの上位原則」として表記。既存のルール1〜8・10以降(表記ルール統合せず)の番号は変更しない。

### 5-4. 削減効果

- セクション削減: なし(位置変更のみ)
- 品質向上: 口調指示の視認性向上(期待)
- 将来への投資: 番号依存脱却

### 5-5. 実装箇所

promptPolicies.gs の buildFormatRules_ 内 1585行

## 9. リスクと対策(v2.0 更新)

### 9-1. リスク

1. **統合時に微妙なニュアンスが失われる**

   - 対策: needsMarketRules / isMarketType / tcNoTypes 等の条件は必ず維持

## 10. 確定した判断事項(v2.0 で確定)

v1.0 の第10章で「承認を求めたい判断事項」として挙げた Q1〜Q4 について、シミュレーション結果に基づき以下のとおり確定した。

### 10-1. Q1: 統合案A(ルール統合)→ **しない**

**理由**
- 【最優先ルール】は「投稿全体の振る舞い」(行動規範)
- 【表記ルール】は「記法」(書式規則)
- 両者は性質が違い、統合するとセマンティックの分離が失われる
- 削減効果も -1セクション・-80字と軽微
- `if (needsMarketRules)` 条件を持つ表記ルールの統合は条件分岐を複雑化する

### 10-2. Q2: 統合案F(トーン統合)→ **しない**

**理由**
- 【Before/After】は 9例 800字の few-shot learning(例から学ばせる)
- 【最終確認】は 7項目 300字の指示文(ルールを守らせる)
- 役割が根本的に違い、統合すると attention が分散する
- buildBeforeAfterExamples_ は独立関数として Phase R-2 で分離した(設計思想と矛盾する)
- Phase 8(Caching)の観点でもキャッシュ効率は同じ(どちらも静的)

### 10-3. Q3: Stage 2(構造再編)→ **Phase R-5 として切り出す**

**理由**
- buildPrompt_ は 620行・109箇所の prompt += ・50+のローカル変数
- 静的/動的の再編は Phase R-2 で見送った「内部分解」と同じ大手術
- 具体的リスク: ローカル変数参照タイミング変更・動的データ注入順序の変更・attention ピーク位置の変化
