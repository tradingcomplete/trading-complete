## 10. Phase R-5: buildPrompt_ 構造再編(v3.0 新規・完了)

### 10-1. 目的

プロンプトを「静的プレフィックス → 動的サフィックス」の順に再編し、Phase 8(Prompt Caching)の前提条件を整備する。

### 10-2. 要件定義の経緯

| バージョン | 内容 |
|---|---|
| v1.0 | アプローチA(staticPart/dynamicPart 二段階組み立て) + 4項目再確認マーカー |
| v2.0 | Q1〜Q4 シミュレーション結果反映 |

Q1〜Q4 の最適解:
- Q1(アプローチA)→ **採用**
- Q2(末尾マーカー)→ **C案(2項目版)に変更**(attention 分散回避)
- Q3(情報ソース静的化)→ **採用**
- Q4(postPrompt 静的化)→ **採用**(リトライ時キャッシュヒット期待)

### 10-3. 実装上の重要な発見

当初は「buildPrompt_ の 109箇所の prompt += を並び替える大手術」と想定していたが、実装中に決定的な発見があった。

> **コードの「実行順」を変えなくても、staticPart/dynamicPart の2変数に振り分けるだけで、最終的な `staticPart + dynamicPart` で「静的プレフィックス → 動的サフィックス」の順序が得られる**

これにより、ローカル変数75個の参照タイミングリスクを完全回避。実装は純粋な「prompt += の置換 + 末尾連結処理」という単純作業に落とし込めた。

### 10-4. 実装内容

| 変更 | 詳細 |
|---|---|
| ヘッダコメント更新 | Phase R-5 の構造と履歴を明記 |
| var staticPart / var dynamicPart 追加 | 冒頭 characterPrompt を staticPart に初期化 |
| prompt += の振り分け | 静的 34箇所 / 動的 78箇所 |
| 末尾再確認マーカー | Q2 C案(2項目版) → dynamicPart 末尾 |
| 連結処理 | var prompt = staticPart + dynamicPart |
| console.log 位置修正 | var prompt 宣言後に移動(初回実行時のエラー修正) |

## 11. Phase R-3 追加修正: 移動漏れ定数の復元(v3.0 新規・完了)

### 11-1. 問題発覚の経緯

Phase R-4/R-5 本番動作確認時に2件のエラーが発生。

1. **1回目のエラー(Phase R-5 実装直後)**
   - `TypeError: Cannot read properties of undefined (reading 'length')`
   - buildPrompt_ 641行目の console.log が var prompt 宣言前に実行されていた
   - 即時修正(console.log を var prompt = staticPart + dynamicPart; の後ろに移動)

2. **2回目のエラー(Phase R-3 時点からの残存バグ)**
   - `⚠️ 市場ニュース取得エラー（スキップ）: FUTURE_INTENT_KEYWORDS is not defined`
   - Phase R-3 分割時(postProcessor.gs → postFinalizer.gs)に定数の移動漏れ

3. **3回目のエラー(同じパターン)**
   - `⚠️ 市場ニュース取得エラー（スキップ）: PAST_TENSE_KEYWORDS is not defined`
   - 2回目と同じパターンの移動漏れ

### 11-2. 修正内容

postFinalizer.gs の isFutureDatePastTenseLine_ 関数の直前に以下2つの定数を追加:

```javascript
/** 削除対象となる過去形/完了形のキーワード */
var PAST_TENSE_KEYWORDS = [
  'が発言', 'と発言', 'を発言', ...（約30キーワード）
];

/** 保持対象となる未来予定/予想の文脈キーワード */
var FUTURE_INTENT_KEYWORDS = [
  '予定', '予想', '見込み', '見通し', ...（約21キーワード）
];
```

### 11-3. 教訓

1. 分割対象ファイル全体で `^var [A-Z_]` パターンの定数定義を先にリストアップする
2. 移動先の関数が参照する大文字識別子(`[A-Z_]{3,}` パターン)を洗い出し、定義がセットで移動しているか確認する
3. 元の postProcessor.gs には2つの定数のみ(今回全て対処済み)
