## v13.0 での大変更点(v12.10 からの変更)

### 1. 検証アーキテクチャの統廃合(4段 → 2段)

**v12.10 までの問題**: 「問題が出るたびに検証層を追加」を繰り返した結果、4段検証が重複動作。

**v13.0 の解決**:
- Stage 1 総合レビュー(Claude + Web検索 最大5回・max_tokens 8192)
- Stage 2 修正適用(論理→事実→NG削除→WARN→品質)
- Claude API 呼び出し: 7回 → 4回(-43%)
- 実行時間: 4分 → 2分18秒(-45%)

### 2. プロンプト構造の根本再設計(Phase 6)

**v12.10 までの問題**: プロンプトが 24,000〜29,000字に膨張。Claude の attention が希薄化して口調・論理が甘くなる。

**v13.0 の解決**:
- 統合案C(因果関係3→1): -180字・-2セクション
- S1 Before/After 実例セクション新設: few-shot誘導で口調学習
- S2 【★★★最終確認】再構成
- S3 統合案D(事実論理3→1): -2セクション
- S4 統合案A(フォーマット5→1): -4セクション
- S6 ダウ理論(日足・週足 SH/SL)を投稿プロンプトに注入
- プロンプト: 24,663字 → 20,753字(-4,700字・-18.5%)

### 3. 構造整合性の3段防御(v13.0.8)

**v12.10 までの問題**: 鉤括弧「」『』の中に「→」や改行が入り論理破綻する投稿が生成される。検証でも検出されない。

**v13.0 の解決**: 3層防御
- 段階1(promptBuilder.gs): 鉤括弧は同じ行で開閉のルールを明示
- 段階2(validationV13.gs): Q8 構造整合性チェック新設 + detectStructureBreaks_ 機械チェック
- 段階3(qualityReview.gs): trimToCharMax_ に鉤括弧保護ロジック

### 4. 口調の根本改善(few-shot学習)

**v12.10 までの問題**: 禁止指示を大量に並べても Claude の口調が硬い(アナリスト調・報告調)。

**v13.0 の解決**: Before/After 実例8組 + コンパナらしさの核6原則
- 漢語より和語、名詞化より動詞化
- 「〜ですね」「〜かなと思います」「〜とこですね」で文末崩し
- 文末は必ず動詞で完結(「〜かなと。」「〜とこ。」禁止)

## 1. 設計思想

```
「魚ではなく、釣り方を」

AI自動投稿であることを公開。プロンプトを見せて信頼と期待感を生む。

【公開するもの(スプレッドシート)】
  キャラクター / トレードスタイル / 投稿プロンプト / 参照ソース / TC概要
  → 「何をAIに聞いているか」が見える

【非公開のもの(コード内)】
  フォーマットルール / 後処理チェーン / 文字数制限 / 構造整合性チェック
  → 見せても意味がない&見栄えが悪い

【使うほど賢くなる仕組み】
  投稿成功時にGeminiが学びを自動抽出 → 学びログに蓄積
  6カテゴリ: 相場心理 / テクニカル / ファンダメンタルズ / リスク管理 / 指標分析 / 市場構造

★v13.0 検証アーキテクチャ(v12.0〜12.4 の発展形)
  旧: 4段検証(Geminiファクトチェック + Claude品質 + Gemini修正 + trimToCharMax_)
  新: 2段検証(Stage 1 総合レビュー + Stage 2 修正適用)
  理由: 4段は重複動作・責任不明確。2段にすることで
    - API呼び出し削減(7→4回)
    - 実行時間短縮(4分→2分18秒)  
    - ロジックの見通し改善

★v13.0 プロンプト再設計
  旧: 指示・禁止事項を積み上げる方式(結果的に肥大化)
  新: few-shot(Before/After実例)で誘導・セクション統合で密度UP
  学び: ルールを減らすと、Claude の attention が解放されて質が上がる

★v13.0.8 構造整合性の3段防御
  1. プロンプトでの予防(生成時点で破綻させない)
  2. 検証での検出(破綻したら Claude に直させる)
  3. 圧縮での保護(trimToCharMax_ が鉤括弧を壊さない)
```

## 2. システム全体フロー(v13.0)

```
投稿生成(generatePost)
   ↓
1. buildPrompt_(promptBuilder.gs)
   - キャラクターシート全文
   - 確定レート・通貨強弱・市場分析
   - ★ダウ理論(日足・週足 SH/SL)
   - 市場ニュースTOP5
   - 経済カレンダー
   - 投稿タイプ別指示
   - 仮説・学びログ
   - ★Before/After 実例(末尾直前・few-shot)
   - ★★★最終確認(最末尾)
   ↓
2. Claude 市場分析(geminiApi.gs)
   - ダウ理論 + 通貨強弱 + ニュースで市場を読む
   - 約 500字の分析結果をプロンプトに追加
   ↓
3. Claude 投稿生成
   - プロンプト + 市場分析 → 投稿本文
   - 約 300字の投稿
   ↓
4. 後処理チェーン(postProcessor.gs)
   - 改行整形・Markdown除去・禁止表現置換
   - ハッシュタグ自動生成
   ↓
5. v13.0 総合検証(validationV13.gs)
   Stage 1: Claude総合レビュー(Q1-Q8)
     Q1-Q5: 品質(硬さ・重複・文字数・口調)
     Q6: 事実検証(確定データ > Web検索 > 本文 > 内部知識)
     Q6.5: 論理整合性
     Q7: 絵文字行の体言止め
     ★Q8: 構造整合性(鉤括弧「」『』の対応・改行/→の混入)
   Stage 2: 修正適用(必要な種類のみ実行)
     修正1: 論理矛盾
     修正2: 事実誤り(機械置換)
     修正3: Web検証NG行削除
     修正4: WARN Claude修正
     修正5: 品質 error Claude一括修正
   ↓
6. 文字数保証(trimToCharMax_・qualityReview.gs)
   - charMax を超えたら末尾から削る
   - ★鉤括弧保護: 鉤括弧の途中で切らない
   ↓
7. 構造警告チェック(detectStructureBreaks_)
   - 機械的に鉤括弧・→・絵文字の破綻を検出
   - 問題があればログ警告
   ↓
8. アダプタ層 → 既存フォーマットでログ保存
   ↓
完成投稿(Sheets下書き → 承認メール → X投稿)
```

## 5. v13.0 検証アーキテクチャの詳細

### 5-1. Stage 1 総合レビュー

**呼び出し**: Claude + Web検索最大5回・max_tokens 8192

**検出する項目**(Q1〜Q8・合計8観点):

```
A. 品質チェック(Q1-Q5):
   Q1 重複・同語反復
   Q2 冗長表現・AI感
   Q3 文章の自然な流れ
   Q4 文字数(charMin〜charMax)
   Q5 ★コンパナ口調(v13.0.7 厳格化):
     ・アナリスト調/AI翻訳調を error 検出
     ・文末未完結(「〜かなと。」「〜とこ。」)を error 検出

   Q6 事実検証(下記B)
   Q7 絵文字行の体言止め

   ★Q8 構造整合性(v13.0.8 新設):
     (a) 鉤括弧「」『』の対応チェック
     (b) 鉤括弧の中に改行・→・絵文字が入っていないか
     (c) →行が前の行から意味的に継承されているか

B. 事実検証(Q6):
   判定優先順位: (1)確定データ > (2)Web検索 > (3)本文 > (4)内部知識
   時間軸主張(「週中高値」「急落率」等)は検証不能なので factErrors に入れない

C. 論理整合性(Q6.5):
   事実は正しいが論理矛盾する投稿を検出
   例: 「予想は外れた」と「着地点は合っていた」の同居
   例: 「リスクオン」と「円高進行」の同居
```

**出力JSON構造**:
```json
{
  "quality": [...],       // Q1-Q5, Q7, Q8 の問題
  "factErrors": [...],    // Q6 事実誤り
  "logical": [...],       // Q6.5 論理矛盾
  "webClaims": [...]      // Web検証結果
}
```

### 5-2. Stage 2 修正適用

問題の種類ごとに異なる修正戦略:

| 修正種 | 対象 | 戦略 |
|---|---|---|
| 修正1 | 論理矛盾(logical) | Claude に指示して直させる |
| 修正2 | 事実誤り(factErrors) | 機械置換(原値で上書き) |
| 修正3 | Web検証NG(webClaims reject) | 該当行を削除 |
| 修正4 | Web検証WARN | Claude に指示して修正 |
| 修正5 | 品質 error(quality error) | Claude に一括修正依頼 |

**修正時のプロンプトには必ずキャラクターシート全文を末尾注入**(v13.0.2 の学び: 口調を守るため)。

### 5-3. キャラクター注入戦略の変遷

| 版 | 戦略 | 結果 |
|---|---|---|
| v13.0 初期 | 注入なし | 口調が硬い |
| v13.0.1 | 3関数すべての冒頭に注入 | **失敗**(468字・注釈混入・文脈崩壊) |
| v13.0.2 | 品質修正(applyQualityFix_)のみ・末尾に注入 | **成功**(306字・ブロック構造完結) |

学び: **注入箇所は1箇所のみ・末尾に・SSOT**

### 5-4. 二重実行防止(v13.0.4)

**問題**: Phase A → Phase B の再実行で v13.0 が2回動作

**解決**: ScriptProperties に `V13_LAST_EXECUTED_<postType>` を記録。180秒以内に再呼び出しされた場合はスキップ。

### 5-5. JSON堅牢化(v13.0.5)

**問題**: 日本語プロンプト内の鉤括弧が JSON の引用符と衝突してパースエラー

**解決**: プロンプトで JSON フィールド値の日本語を「〜」で囲むルール明示・Stage 1 失敗時は文字数保証のみ実行するフォールバック

### 5-6. Q5 口調判定厳格化(v13.0.6〜v13.0.7)

**v13.0.6 で追加**:
- アナリスト調・AI翻訳調を error 判定

**v13.0.7 で追加**:
- 「〜かなと。」「〜とこ。」の文末未完結を error 判定
- 「〜かなと思います。」「〜とこですね。」で動詞完結するよう修正指示

### 5-7. 構造整合性の3段防御(v13.0.8)

**段階1** promptBuilder.gs L1571〜:
```
■ ★鉤括弧の整合性(絶対遵守・構造破綻防止):
  ・鉤括弧「」『』は必ず同じ行の中で開いて閉じろ
  ・鉤括弧の中に「→」「改行」「絵文字」を入れるな
```

**段階2-a** validationV13.gs buildStage1Prompt_ に Q8 追加

**段階2-b** validationV13.gs 末尾に detectStructureBreaks_ 関数:
- 鉤括弧の数の対応チェック
- 鉤括弧の中身チェック(改行・→・絵文字)
- 問題があればログ警告

**段階3** qualityReview.gs trimToCharMax_ に鉤括弧保護:
- 行削除で鉤括弧が破綻する場合、削除を諦める
- または鉤括弧ブロック全体を削除

## 8. ダウ理論(SH/SL)の活用(v13.0 重要改修)

### 8-1. SH/SL 判定ロジック

**場所**: priceSummary.gs updateSwingHighLow_

### 8-4. 時間軸の原則(コンパナのスタイル)

コンパナは**デイ〜スイングトレーダー**。投稿の時間軸は**基本日足**で語る。
- 短期(数時間)は補足で触れる程度
- 週足と日足が一致: 確度高
- 週足と日足が逆行: 転換の可能性として言及

## 9. 口調の設計(v13.0 最大の革新)

### 9-1. Before/After 実例8組

| # | 変換テーマ |
|---|---|
| 1 | 「〜を確認」の堅さ → 口語化 |
| 2 | AI翻訳調・漢語過多 → 崩した話し言葉 |
| 3 | 報告調・名詞化 → 動詞化 |
| 4 | 堅い文末「〜はずです」→ 「〜話」 |
| 5 | 堅い文末「〜ましょう」→ 「〜とこ」 |
| 6 | 漢語多用 → 和語化 |
| 7 | 堅い体言止め+カタカナ → 崩し |
| 8 | 「〜します」の謙譲 → 「〜したいとこですね」 |

### 9-2. コンパナらしさの核6原則

```
1. 漢語より和語
   「複合的」→「絡んでる」
   「局面」→「ところ」
   「確認」→「確かめる」

2. 名詞化より動詞化
   「上値の重さ」→「上が重い」
   「未確定ながら」→「まだわかんないけど」

3. 「〜です/ます」だけで終わらせず、
   「〜ですね」「〜かなと思います」「〜とこですね」「〜感じですね」で崩せ

4. ★文末は必ず動詞で完結(v13.0.7 新設)
   NG: 「〜かなと。」「〜とこ。」「〜感じ。」
   OK: 「〜かなと思います。」「〜とこですね。」「〜感じです。」

5. 「正直」「ちょっと」「ぶっちゃけ」を自然に挿入

6. 同じ語尾2回連続は禁止
```

## 14. 設計の原則(今後開発する時の指針)

1. **小手先修正の罠を避ける**
   - 問題が出たら「もう1段検証を足す」「ルールを1行追加」は肥大化の元凶
   - 根本原因を突き止める・構造を再設計する

2. **生成時点で対処する**
   - 検証で直すより、生成時点で間違えないプロンプトを作る
   - Before/After 実例・明確な制約が検証より効く

3. **few-shot > 禁止指示**
   - LLMは「こう書け」の実例に強く反応する
   - 「〜するな」の禁止を並べるより、OK例を1つ見せる方が効果的

4. **プロンプト末尾が最強**
   - Claude は最後に見た指示を最優先する
   - 重要な指示は末尾に置く(★★★最終確認・Before/After)

5. **SSOT(Single Source of Truth)**
   - キャラクター定義はスプレッドシートの「キャラクター」シートが唯一
   - コード側で重複注入するとかえって薄まる

6. **並行稼働フラグで即ロールバック**
   - 新機能は必ずフラグで切り替え可能に
   - 問題発生時はフラグOFFで即座に前バージョンに戻せる

7. **疲れたら止まる**
   - 疲労した状態でコードに変更を加えるとバグを生む
   - 検証・シミュレーション・一晩寝かせる判断を優先

## v14.0 Phase 1 追記(2026-04-20・ホットトピック事前選定機構)

### 背景

v13.0 決定版リリース後の観察期間(Phase 4)で「投稿が面白くない」という根本的な評価があった(TCAX_REFERENCE.md 事件8参照)。分析の結果、「Claude に全情報を投げて自由に選ばせる」設計だと毎日同じ静的データ(ゴトー日・通貨強弱%)を選びがちだと判明。

**「材料選定を先に機械的に実行する」方式に転換**。

### 新規実装

#### `selectHotTopic_()` 関数

- **所在**: promptBuilder.gs 末尾(約170行)
- **役割**: Claude API で「今この瞬間のホットな材料1つ」を選定
- **入力**: ニュースTOP5 + 通貨強弱 + 方向性 + ダウ理論 + 継続中重大事象 + 投稿タイプ
- **出力**: 構造化JSON
  - `headline` (20文字以内)
  - `background` (80文字程度)
  - `causalChain` (60文字程度)
  - `mainPair` (通貨ペア日本語名)
  - `dowContext` (40文字程度)
  - `nextView` (60文字程度)
  - `confidence` (high/medium/low)
- **API呼び出し**: +1回(約10秒程度)
- **フォールバック**: null を返す → 呼び出し元で従来動作に戻る

#### `formatHotTopicForPrompt_()` 関数

- **所在**: promptBuilder.gs 末尾
- **役割**: selectHotTopic_ の JSON を投稿生成プロンプト用テキストに整形
- **出力例**:
```
【★今日書くべきトピック1つ(これを核にしろ・ここから逸脱禁止)】
ヘッドライン: ...
背景: ...
因果チェーン: ...
今日の主役ペア: ...
ダウ理論の文脈: ...
次の見方: ...

【書く手順(この順番で構成しろ)】
1. 冒頭1行: ヘッドラインをキャッチーに言い換え
2. 背景を簡潔に説明
3. 因果チェーンで相場への波及を語る
4. 主役ペアのダウ理論文脈を具体値で
5. 次の見方を示す
```

### buildPrompt_ への改修

上位スコープ変数4つを追加し、既存のデータ取得で「ついでに保存」:

var _hotTopicCurrencyStrength = '';  // hotPairResult.text

- MORNING / TOKYO / LUNCH / LONDON / GOLDEN(市場系のみ)
- 他タイプ(RULE系・WEEKLY_LEARNING等)は従来動作

### 初テスト結果(2026-04-20 17:31)

- selectHotTopic_ 実行時間: 約8秒
- プロンプト増加: +約600字 / +2セクション

### 設計原則(v14.0 Phase 1 から追加)

#### 8. 材料選定を投稿生成から分離する
- 「何を書くか」と「どう書くか」は別のタスク
- Claude に両方同時にやらせると attention が分散する
- 材料選定は事前に済ませて、投稿生成には「選定済みの構造化データ」を渡す

#### 9. トレーダーの4つの疑問に答える
- 新しい材料はないか?
- 今のポジションは正しいか?
- 持ち続けるか利確するか?
- 他のトレーダーはどう見ているか?

この4つのどれか(または複数)に答える投稿 = 価値ある投稿。

#### 10. 面白い投稿の4要素
- ヘッドニュース
- 値動きの背景・なぜ
- ダウ理論
- トレンド転換のライン

この4要素が揃っている投稿が「面白い」。
