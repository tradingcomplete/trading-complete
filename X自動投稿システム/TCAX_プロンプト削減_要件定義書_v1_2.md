## 1. 背景と目的

### 背景

2026-04-20 に v14.0 Phase 1 (ホットトピック事前選定機構) を実装。
`selectHotTopic_()` が以下の要素を事前に決定するようになった:

- ヘッドライン(冒頭を決める)
- 背景(2段目を決める)
- 因果チェーン(本文の骨格を決める)
- 主役ペア(話題の焦点を決める)
- ダウ理論の文脈(チャート視点を決める)
- 次の見方(結論を決める)

この結果、**投稿生成時のClaudeは「何を書くか」でほぼ迷わない状態**になった。
「どう書くか(コンパナ口調)」だけに集中できる。

### 目的

従来の「Claude に全情報を投げて自由に選ばせる」設計を前提にしていた
膨大な指示セクション群を、**現在の新方式に最適化された形**に再構成する。

**目標**: 現状 約 24,000字 → 目標 約 20,000字前後 (削減幅はシート現状依存)

### 注意

**これは v14.0 Phase 1 が 5投稿タイプ全てで安定動作することを確認した後に実行する**。
観察期間中に実行してはならない(品質低下の原因切り分けが困難になるため)。

## 5. フェーズ別削減プラン

### ★★★ Phase 1: GOLDEN 緊急タイムアウト対策 (v1.2 新規・最優先) ★★★

#### 目的

**GOLDEN 投稿が再び 6分タイムアウトで失敗しないようにする**。
他の Phase を待たず **明日の GOLDEN(2026-04-21 20:47頃)までに手を打つ**必要あり。

#### 緊急度

**最高**。これができなければ、明日以降 GOLDEN が生成されない可能性が高い。

#### 対策の選択肢(組み合わせ可能)

##### 選択肢A: TC関連6セクションの統合(推奨・最効果)

**現状**: LUNCH と GOLDEN のプロンプトに以下 6セクションが追加される:
1. 【TC導線】
2. 【Trading Complete(TC)について - 投稿に自然に織り込むための参考情報】
3. 【TCが解決するユーザーの課題(ペイン)】
4. 【TCのターゲット層(ペルソナ)】
5. 【TCの核心的価値(提供する本質)】
6. 【AIへの厳格な記述ルール(トーン&マナー)】
7. 【TC開発の背景(自然な導入用ストーリー)】
8. 【Trading Completeの主要機能(文脈に合わせて自然に1つだけ触れること)】
9. 【TC導線(AI自律判断・週1〜2回・投稿の20%以下)】

※合計 9セクション・約 3,000字

**統合案**: これらを **1セクション【TC導線(週1〜2回・自然に織り込め)】に圧縮**。
コア指示と1〜2のOK/NG例のみ残す。約 -2,500字削減。

**作業箇所**:
- スプレッドシート「投稿プロンプト」シート(該当タイプの行で TC 関連記述を圧縮)
- または promptBuilder.gs 内で TC 関連を動的に絞る処理を追加

##### 選択肢B: GOLDEN のニュース取得キャッシュ活用(即効性あり)

**現状**: GOLDEN 発火時(19:50頃)、LONDON(17:31頃)のキャッシュ(1時間)は既に切れている。
新規に Claude web_search 5クエリを実行 = 約1分40秒消費。

**対策案1**: ニュースキャッシュ時間を 1時間 → **2時間** に延長
→ LUNCH(12:11頃)のキャッシュを LONDON / GOLDEN で再利用可能
→ ただし「新鮮な情報」という本来の価値を損なう(要議論)

**対策案2**: ニュース取得失敗時のフォールバックを拡充
→ キャッシュミス時の Claude web_search を 5クエリ → 3クエリに削減
→ 約60秒削減見込み

##### 選択肢C: Stage 1 検証のタイムアウト監視

**現状**: Stage 1 が修正1〜5 を順次実行中、時間切れになると投稿全体が失敗。

**対策案**: Stage 1 開始時点の経過時間をチェックし、**5分を超えていたら修正をスキップして保存のみ実施**する。
→ 「不完全な投稿でも、下書きを作って承認待ちに入れる」が最優先。
→ コンパナが手動でチェックして承認すればよい。

**作業箇所**: validationV13.gs の `executeValidationV13_` 冒頭

##### 選択肢D: ランダム遅延の短縮

**現状**: ランダム遅延 0〜43秒(Bot判定回避)。

**対策案**: GOLDEN のみランダム遅延を 0〜15秒に短縮。
→ 最大28秒の削減。
→ ただし Bot判定回避の効果が下がる可能性(要議論)。

#### 推奨する組み合わせ

**明日までに必達**: 選択肢A (TC関連統合) + 選択肢C (Stage 1 タイムアウト監視)

- 選択肢A だけで約 -2,500字 = 約 -30秒の API 応答時間短縮
- 選択肢C で万が一の保険を追加(修正が途中で止まっても下書きは残る)

この2つで **所要時間を 4分台に抑える** のが目標。

#### ロールバック計画

- **選択肢A**: スプレッドシート編集前にバックアップを取る
- **選択肢C**: validationV13.gs の修正前コミットからリセット可能

- 4分30秒以内に完了する
- 下書きが作られる
- 承認メールが届く

v8.6 統合版と実シートの内容が乖離している可能性があるため。

1. **GAS エディタで以下の関数を実行**し、実シートの内容をログに出力する:

```javascript
/**
 * キャラクターシートの実体を確認する(Phase 0 用)
 * v1.1: プロンプト削減要件定義書で追加
 */
function phase0_dumpCharacterSheet() {
  var ss = SpreadsheetApp.openById(getApiKeys().SPREADSHEET_ID);
  var sheet = ss.getSheetByName('キャラクター');
  
  if (!sheet) {
    console.log('❌ キャラクターシートが存在しない');

  }
  
  var data = sheet.getDataRange().getValues();
  console.log('============================================');
  console.log('📋 キャラクターシート実体 (' + data.length + '行)');
  console.log('============================================');
  
  for (var i = 0; i < data.length; i++) {
    var sectionName = String(data[i][0] || '').trim();
    var content = String(data[i][1] || '').trim();
    
    if (!sectionName && !content) continue;
    
    console.log('');
    console.log('【行' + (i+1) + '】A列(セクション名): "' + sectionName + '"');
    console.log('        B列(文字数): ' + content.length + '文字');

    // B列に【】サブセクションが含まれているか検出
    var subSections = content.match(/【[^】]+】/g) || [];
    if (subSections.length > 0) {
      console.log('        ★ B列内のサブセクション数: ' + subSections.length);
      console.log('        ★ サブセクション一覧: ' + subSections.join(', '));
    }

  console.log('');
  console.log('============================================');
  console.log('📊 要約');
  console.log('============================================');
  
  var totalChars = 0;
  var totalSubSections = 0;
  for (var i = 0; i < data.length; i++) {
    var content = String(data[i][1] || '');
    totalChars += content.length;
    totalSubSections += (content.match(/【[^】]+】/g) || []).length;
  }
  console.log('総行数: ' + data.length);
  console.log('B列の総文字数: ' + totalChars);
  console.log('【】サブセクション総数: ' + totalSubSections);
```

- B列内に【】サブセクションが多数ある → これが口調15セクションの正体

```
B列内のサブセクション数は?
```

## 6. 検証方法

### 各Phase終了時の検証手順

1. **テスト実行**: `testProMorning()` を3回実行
2. **ログ確認**:
   - プロンプト文字数が目標通り減っているか
   - セクション数が目標通り減っているか
   - ホットトピック選定が正常動作しているか
   - Stage 1 検出数が異常増加していないか
3. **投稿品質確認**: コンパナが目視で「面白い」と判断できるか
4. **4要素チェック**(事件8より):
   - ヘッドニュース ✓
   - 因果 ✓
   - ダウ理論 ✓
   - 次の見方 ✓
5. **4疑問チェック**(事件8より):
   - 新しい材料 ✓
   - ポジション検証 ✓
   - 利確判断 ✓
   - 多数派の見方 ✓

### 合格基準(各Phase共通)

- 削減目標の ±10% 以内で削減できている
- 投稿品質が削減前と同等以上(コンパナ評価)
- 文末未完結・絵文字修正失敗・役割逸脱が再発していない
- Stage 1 検出数が削減前と比べて 1.5倍以内

## 7. ロールバック基準

以下のいずれかに該当したら **即座にロールバック**:

### 即ロールバック

- 投稿品質がコンパナ評価で「削減前より悪い」
- 文末未完結が直近 5投稿で 2回以上再発
- 4要素のうち 2つ以上が欠落した投稿が出た
- ホットトピック選定失敗率が 20% を超えた

### 慎重に継続

- Stage 1 検出数が 1.3〜1.5倍に増加(修正で収束しているなら続行)
- 文字数圧縮発動率が 1.5倍程度に増加(内容に影響なければ続行)

### ロールバック手順

1. **スプレッドシート編集の場合**: バックアップシート「キャラクター_backup_YYYYMMDD」からデータをコピーして戻す
2. **promptBuilder.gs 編集の場合**: 修正前のバックアップから復元
3. ロールバック理由を REFERENCE.md 事件簿に記録
