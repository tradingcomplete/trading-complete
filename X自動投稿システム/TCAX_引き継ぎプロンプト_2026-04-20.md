## 🎯 TCAX とは

**T-CAX(Trading Complete Auto X)**: FX投稿を X(Twitter)に自動投稿するGASシステム

- **スケジュール**: 平日5回+週末8回(v12.7 で NY削除)
- **生成**: Claude Sonnet 4.6(分析・生成・検証の3役)
- **データ**: Twelve Data API(7通貨ペア + BTC + Gold)
- **ニュース**: Claude web_search(5クエリ・1時間キャッシュ)
- **規模**: GASファイル28本・約27,500行

### 平日のスケジュール(月曜の例)

| 時刻(ランダム+0〜5分) | タイプ | 上限文字数 |
|---|---|---|
| 08:03 | MORNING | 150-300字 |
| 09:18 | TOKYO | 100-180字 |
| 12:08 | LUNCH | 100-180字 |
| 17:22(サマータイム: 16:22) | LONDON | 150-250字 |
| 20:47(サマータイム: 19:47) | GOLDEN | 200-350字 |

### 目標

**「FXトレーダーがマストで登録しとけと言うアカウント」**

コンパナの本質言語化(事件8より):
> 「事実を収集する → それをコンパナ風にユーザーに有益に伝える。対話では1〜2行渡すだけで済んでたが、それをシステムで自動にしたい」

## 🎓 v14.0 Phase 1 の設計思想(最重要・絶対に読め)

### 背景(事件8より)

**従来の設計**:
- Claude に全情報(スプレッドシート23枚+ニュース50件+通貨強弱+ダウ理論+...)を投げて、その中から自由に選ばせる
- 結果: 毎日同じ静的データ(ゴトー日・通貨強弱%)を選びがちで「面白くない」投稿に

**コンパナが本当に求めている設計**:
- 「今この瞬間のホットな材料」が先
- 時間帯(MORNING/TOKYO/LUNCH等)は従属的な情報
- 材料 → 因果チェーン → ダウ理論 → 次の見方 の順で組み立てる
- 対話時コンパナが1〜2行で渡してくれた「材料の選定」を、システムが代行すべき

### selectHotTopic_() の役割

1. ニュース取得後、通貨強弱・ダウ理論・方向性・継続中重大事象を総合して判断
2. Claude API で「今最も語るべきトピック1つ」を選ぶ
3. 構造化JSON(headline/background/causalChain/mainPair/dowContext/nextView/confidence)を返す
4. GAS側で `formatHotTopicForPrompt_` を使って投稿生成プロンプトに注入
5. 投稿生成 Claude は「選ぶタスク」から解放され、「書く(コンパナ口調)」に集中できる

### トレーダーが常に抱える4つの疑問(事件8より)

**価値ある投稿 = この4疑問に新鮮な材料を起点に答えること**:

1. 新しい材料はないか?(最新情報)
2. 今のポジションは正しいか?(検証したい)
3. 持ち続けるか利確するか?(判断したい)
4. 他のトレーダーはどう見ているか?(多数派を知りたい)

### コンパナが「面白い」と感じる投稿の4要素

1. **ヘッドニュース**(今日の相場を動かした具体的なトピック)
2. **値動きの背景・なぜ**(因果関係)
3. **ダウ理論**(チャートの構造)
4. **トレンド転換のライン**(「このラインを抜けたら転換」という具体的なレベル)

### ★Claude が改修判断する時の4つの自問(絶対守れ)

- その修正は「材料の鮮度」を上げるか?
- その修正は「なぜトレーダーは X を見るか」の4疑問に答えるか?
- その修正は「時間帯が先」の古い発想に戻っていないか?
- コンパナが実際に「面白い」と感じる投稿に近づくか?

**プロンプトを削ること・セクションを整理すること・文字数を守らせることは二次的。本質は材料の鮮度と因果の明確さ**。

## 🛠️ 技術的な引き継ぎ事項

### selectHotTopic_() の所在と依存関係

- **定義**: promptBuilder.gs L2345 付近
- **呼び出し元**: promptBuilder.gs の `buildPrompt_` 内(ニュース取得直後)
- **依存関数**:
  - `callClaudeApi_()` (geminiApi.gs L1214)
  - `getOngoingEvents_()` (sheetsManager.gs L1735)
- **失敗時**: null を返す → 呼び出し元で従来動作にフォールバック

### buildPrompt_ 内の上位スコープ変数(v14.0 Phase 1 で追加)

```javascript
var _hotTopicCurrencyStrength = '';  // hotPairResult.text を保存
var _hotTopicRateDirection    = '';  // calculateRateDirection_ の結果
var _hotTopicDowTheory        = '';  // getDowTheorySummary_ の結果
var _hotTopicOngoingEvents    = '';  // getOngoingEvents_ の結果
```

これらは既存のデータ取得処理で「ついでに保存」して、後に selectHotTopic_ に渡す。

### 既存機構への影響

- **ゼロ**。新関数は promptBuilder.gs 内で完結している
- geminiApi.gs / validationV13.gs / その他 GS ファイルは変更不要
- 既存の buildPrompt_ の戻り値(文字列)の形式は変わっていない

## 💬 コンパナとの対話のコツ

### 対話スタイルの特徴

- 質問は `ask_user_input_v0` ツールで選択肢提示(モバイル入力を減らす)
- 答えが出たら **すぐ動く**(議論に時間をかけない)
- 進捗はコマンド単位で報告(「○○実行しました・△△が判明しました」)
- 悪い結果も正直に報告する(誤魔化しは信頼を損なう)

### コンパナが喜ぶリアクション

- 「コンパナの指摘が正しかった」と認めること
- 実データでの検証結果を出すこと
- 自分の推測と事実を区別して話すこと
- シミュレーションしてから実装すること

### コンパナが怒るリアクション

- 「あとで確認します」「次回で」
- 確認なしに大きく変更する
- 曖昧なまま進める
- 「動くはず」と言って動かない
