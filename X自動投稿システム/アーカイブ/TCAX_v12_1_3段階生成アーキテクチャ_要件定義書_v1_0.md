## 1. なぜ変えるのか

### 1-1. v12.0の問題

v12.0では3段防御（ハルシネーション対策）を導入し、数値の捏造は大幅に減った。
しかし**方向性の矛盾**という新たな問題が顕在化した。

```
【実際に起きた事例（2026-04-06 LONDON投稿）】

  通貨強弱データ: EUR(+0.81%) > JPY(+0.13%) > USD(-0.11%)
  → EURが最強、USDが弱い。EUR/USDは上昇。

  Geminiが書いた投稿:
  「ロンドン勢はユーロ売りで反応」「ECBハト派スタンスが意識」
  → 完全に逆。EURは最強通貨。ECBは利上げ転換観測。

  ファクトチェック: 通過（数値の誤りではないため検出不能）
  Claude Q6: 通過（レートの数値自体は正確なため）
```

### 1-2. 根本原因

Geminiに「分析もライティングも全部やれ」と1回で指示していた。
22,000文字のプロンプトに60以上のセクションを詰め込み、
ニュース選定・データ分析・文章作成・文字数管理・構造遵守を同時に要求。

人間でもこれだけ同時にやったら間違える。
Geminiは「一般論」（リスク回避=円買い、ECB=ハト派等）に逃げてしまう。

### 1-3. 解決の方針

**役割を分ける。**

```
旧（v12.0）: Gemini（全部やれ）→ 間違える → Claude（ダメ出し）
新（v12.1）: Claude（データを読め）→ Gemini（この分析で書け）→ Claude（品質チェック）
```

コンパナの言葉: 「自分が楽を選ぶな、ユーザーが求めるものを第一に考えよ」

## 2. v12.1 アーキテクチャ全体像

### 2-1. 3段階生成パイプライン

```
【Stage 1: 分析役（Claude Sonnet 4.6）】← ★v12.1新設
  入力: 為替レート7ペア + 通貨強弱 + ペア別変動 + 商品価格 + ニュースTOP5
  処理: データを正しく読み、今の市場で何が起きているかを分析
  出力: 400文字以内の市場分析テキスト
    - 主要テーマ（1文）
    - 通貨の強弱（データからの事実）
    - 主要ペアの方向（上昇/下落/横ばい）
    - 背景の推測（ニュース・金融政策から）
    - 注意（データと矛盾しやすい一般論への警告）

【Stage 2: ライティング役（Gemini 2.5 Pro）】← 既存を改良
  入力: 既存プロンプト22,000文字 + ★Claude市場分析（最優先セクションとして注入）
  処理: Claude分析に従って投稿テキストを生成
  制約: 「Claude市場分析と矛盾する記述は絶対に書くな」
  出力: 投稿テキスト

【Stage 3: レビュー役（Claude Sonnet 4.6）】← 既存（Q1〜Q7）
  入力: 生成テキスト + 確定データ + 過去投稿
  処理: 品質チェック7項目（タイプ整合・重複・完成度・文字数・口調・事実・絵文字行）
  出力: 指摘事項 → Gemini修正 → 最終テキスト
```

```

  │     └── 22,000文字のプロンプト完成

  ├── ★Step 2.5: Claude市場分析（analyzeMarketWithClaude_）← v12.1新設
  │     ├── 通貨強弱・ペア変動を再取得（detectHotPair_）
  │     ├── 商品価格（キャッシュヒット）

  │

  ├── Step 4: 後処理チェーン（12段階）
```

## 4. 未実装（v12.1 Phase 2）: パイプライン分割

### 4-1. 背景

GAS 6分制限は絶対。v12.1で時間配分が改善されたが、
リトライが多い場合や画像生成込みだと5.5分に達することがある。
今後Phase 5（プロンプト自動進化）等で処理が増えると限界に達する。

### 4-2. 設計

generatePostを2段階に分割し、各フェーズに6分ずつ使えるようにする。

```
【Phase A: テキスト生成（1回目のトリガーで実行）】
  Step 1: レート取得
  Step 2: buildPrompt_
  Step 2.5: Claude市場分析
  Step 3: Geminiテキスト生成
  Step 4: 後処理チェーン
  Step 5: リトライ群
  → 下書きシートに保存（ステータス: 「Phase A完了」）
  → Phase Bトリガーを1分後に設定

【Phase B: 品質チェック+画像+送信（2回目のトリガーで実行）】
  Step 6: ファクトチェック（Gemini + Grounding ON）
  Step 7: 品質レビュー（Claude Q1〜Q7）→ Gemini修正
  Step 8: 画像生成（対象タイプのみ）
  Step 9: 承認メール送信
```

### 4-3. メリット

```
・各フェーズ6分 → 合計12分使える
・Phase Aでリトライが多くても、Phase Bに影響しない
・Phase Bで画像生成に時間がかかっても安全
・将来のPhase 5（プロンプト自動進化）もPhase Bに組み込める
```

### 4-4. 実装上の注意点

```
・Phase A→Bのデータ受け渡し: 下書きシート + ScriptProperties
・Phase Bトリガーの設定: ScriptApp.newTrigger().timeBased().after(60000).create()
・トリガーのクリーンアップ: 使い終わったトリガーを自動削除
・エラー時: Phase Aで失敗 → Phase Bは実行しない（エラーメール送信）
・テスト: testPro_でPhase A→Bが連続実行されることを確認
```

```
優先度: 中（現在はBTC/ゴールド事前取得で60秒削減済み。即座に必要ではない）
```

## 8. 設計の鉄則（v12.1追加分）

```
【v12.1で追加された鉄則】

・Geminiに「分析と執筆の両方」を1回でやらせるな。分析はClaudeに任せよ。
・Claude市場分析は「データが全て。一般論ではなく数字が示す事実」を原則とする。
・Geminiへの修正指示は「削除するな、書き直せ」。削除は楽だが読者の価値を損なう。
・charMin未満の投稿は「雑に見える」ため絶対禁止。
・TOKYO/LUNCHは「短くてもためになる」が軸。中身を薄くするための短さではない。
・BTC/ゴールドはscheduledFetchRatesで事前取得。generatePost内での60秒待ちを回避。
・キャッシュ時間はトリガー間隔より余裕を持たせる（毎時更新 → 75分キャッシュ）。
・GAS 6分制限に対する安全策: Claude分析は経過3分以上でスキップ。
```
