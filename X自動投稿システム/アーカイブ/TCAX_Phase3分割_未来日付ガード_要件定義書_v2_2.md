### 画像少なすぎ問題への対応

v1.5で画像生成を月43枚まで絞った結果、「通常の平日はMORNINGの1枚だけ」という状況に。コンパナさんの直感「少なすぎたな」という違和感は、実運用者の感性として大事な指摘。

### 新方針: 「1日を画像で挟む」

朝と夜の2点に画像を配置:

| 時間帯 | 投稿 | 画像 | 役割 |
|--------|------|------|------|
| 朝（7-8時） | MORNING | ✅ | 1日の始まり・方向性提示 |
| 午前（東京オープン） | TOKYO | なし | |
| 昼 | LUNCH | なし | |
| 夕方（欧州オープン） | LONDON | なし（v1.5維持） | |
| **夜（NY前）** | **GOLDEN** | ✅ **新規追加** | **1日の集大成・次の展開予告** |

### GOLDEN画像の根拠

1. **読者の閲覧時間帯と合致**: 帰宅後の夜は最も画像を見る余裕がある時間帯
2. **GOLDEN本来の役割**: 仮説振り返り+次の展開。情報密度が高く、視覚整理の価値が大きい
3. **1日のリズム作り**: 朝の「始まり」と夜の「集大成」を画像で強調することで、フォロワーの認知に残りやすい
4. **LONDON時間帯より効果が期待**: 夕方欧州参入時は読者が仕事中/移動中。夜のNY前は帰宅後の最活性時間帯

### 画像生成枚数の更新

| シナリオ | 平日画像（月） | 週末画像（月） | 合計/月 |
|---------|--------------|--------------|--------|
| 現状（v12.6.1） | MORNING 20 + LONDON 20 + NY 20 + INDICATOR 15 = 75 | WEEKLY_REVIEW 4 + NEXT_WEEK 4 = 8 | **83枚** |
| v1.5（NY+LONDON削除） | MORNING 20 + INDICATOR 15 = 35 | 8 | **43枚** |
| **v1.7（GOLDEN追加）** | **MORNING 20 + GOLDEN 20 + INDICATOR 15 = 55** | **8** | **63枚** |

### 実装上の変更

imageGenerator.gs の `IMAGE_TYPE_COLORS` に GOLDEN を追加:

```
var IMAGE_TYPE_COLORS = {
  MORNING:       { accent: 'サイアン（水色/cyan）',  label: 'MORNING BRIEF' },
  GOLDEN:        { accent: '未定（ネオンオレンジ or 金色系を推奨）', label: 'GOLDEN TIME' },  // ★v1.7新規
  INDICATOR:     { accent: 'ネオン赤（neon red）',   label: 'INDICATOR ALERT' },
  WEEKLY_REVIEW: { accent: 'ゴールド（gold）',       label: 'WEEKLY REVIEW' },
  NEXT_WEEK:     { accent: 'ネオン緑（neon green）', label: 'NEXT WEEK' }
};
```

GOLDENのアクセントカラーは実装時に決定。候補:
- ネオンオレンジ: 夜のNY前の緊張感・集中モードを表現
- 金色系: 名前「GOLDEN」と合致するが、WEEKLY_REVIEWと被る懸念
- ネオン紫（NYから引き継ぎ）: 深夜感を残すが、GOLDENは深夜ではないので違和感

IMAGE_ARCHETYPESの bestFor にも GOLDEN を追加（DUAL_CONTRAST, DYNAMIC_ACTION 等が候補）。

## 3. 設計方針

### 3.1 3つのPhaseに分割

現状のPhase A/Bを3段階化する。切れ目を「下書き保存」に設ける。

```
【現状 Phase A/B】
  Phase A（6分）: テキスト生成 + 全検証 + 修正 + 下書き保存
  Phase B（1分後）: 画像生成 + 承認メール

【新 Phase A/B/C】
  Phase A（目標3〜4分）: テキスト生成 + 簡易ファクトチェック → 下書き保存
  Phase B（1分後・目標3〜4分）: 深い品質整形 + 未来日付ガード + 最終事実検証
  Phase C（更に1分後）: 画像生成 + 承認メール送信
```

### 3.2 設計思想: Write-Then-Review

「生成」と「検証」を独立したPhaseに分離する。これは以下の理由で理にかなっている:

- 生成直後のテキストは書き手バイアスが強い
- 下書き保存を挟むことで「他人の文章」として再検証できる
- Claude自身の推奨設計（複雑タスクはAPI呼び出しを分割）
- システム作成者が既に「疑ってチャットに送る」という形で無意識に実践していたパターンのシステム化

### 3.3 未来日付ガードの追加

今日の事故を根絶するため、2段のガードを設ける:

**ガード1（ニュース収集直後）**: fetchMarketNews_でGeminiが取得したニュースから未来日付を含む項目を強制除外

**ガード2（投稿生成後）**: applyPostProcessingChain_に未来日付除去関数を追加。投稿本文の「4/17」「4月17日」等を検出したら該当文を削除

### 3.4 API呼び出し回数は増やさない

A案では既存処理を3つのPhaseに振り分けるだけで、Claude/Gemini APIの呼び出し回数は変わらない。**コスト増加はゼロ**。

未来日付ガードはコード側の正規表現処理で完結するためAPIコスト不要。

## 4. Phase分割の詳細設計

### 4.1 Phase A: 生成Phase（目標3〜4分）

**役割**: 初稿の生成と確定データによる事実チェックだけ実施。品質修正は次Phaseに委譲。

**実行処理**:
1. レート取得（Twelve Data API）
2. Claude市場分析（analyzeMarketWithClaude_）
3. 投稿テキスト生成（callClaudeGenerate_）
4. 基本的な後処理チェーン（applyPostProcessingChain_の前半）
5. ★**ガード2**: 未来日付除去（新規）
6. factCheckPost_（Gemini Grounding）で事実検証 → 結果を記録のみ
7. 下書き保存（saveDraft）
8. **Phase B トリガー設定**（1分後に実行）

**終了時点で下書きシートに保存されるもの**:
- 投稿テキスト（初稿）
- ファクトチェック結果（JSON）
- 使用したレート・通貨強弱データ
- factCheckSkippedフラグ
- Phase A完了時刻

### 4.2 Phase B: 品質整形Phase（新設・目標3〜4分）

**役割**: 下書きを読み込み、深い品質レビューと事実誤り修正を実施。

**実行処理**:
1. 下書きを読み込み（getDraftById_）
2. Phase Aで記録したファクトチェック結果を処理（autoFixPost_ or forceRemoveIssueLines_）
3. qualityReviewPost_（Claude Web検索付き）でQ1〜Q7品質レビュー
4. Q6事実誤りがあれば元投稿ベース修正（v12.6.1の方式）
5. 他Q指摘のパッチ修正
6. finalFactVerify_（Claude JSON検出+コード置換）で最終事実検証
7. ★**ガード2再適用**: 修正後に再度未来日付チェック
8. 完成版テキストを下書きに上書き保存
9. **Phase C トリガー設定**（更に1分後に実行）

**フォールバック時の安全策**:
- factCheckSkipped=true（Gemini 503等）の場合、Phase Bで強制的にmanualモードに切り替え
- 修正処理が時間切れになった場合も、Phase Aの初稿をそのまま維持して次Phaseに渡す（投稿自体は止まらない）

### 4.3 Phase C: 画像+通知Phase（現Phase Bと同等）

**役割**: 完成テキストに基づき画像を生成し、モードに応じて投稿または承認メールを送信。

**実行処理**:
1. 下書きから完成版テキストを読み込み
2. 画像対象タイプ（MORNING, GOLDEN, INDICATOR, WEEKLY_REVIEW, NEXT_WEEKの5タイプ）のみ画像生成（generatePostImage）
3. 画像をDriveに保存
4. ★**異常フラグ再確認**: factCheckSkipped=true または ハートビート警告時は強制manualに降格
5. POST_MODEに応じて分岐:
   - manual: 承認メール送信
   - validate: バリデーション通過で自動投稿 / 失敗でメール
   - auto: 直接投稿
6. 投稿成功時は下書きのステータスを「完了」に更新

### 4.4 ★NEW: autoモードでも下書きが必ず作られる（v1.2で明示化）

#### 4.4.1 現状の問題

現状の autoモード（main.gs handleAutoMode_）は、下書きを作成せず generatePost → executePostToX_ を1トリガー内で完結する設計になっている。これには以下の致命的な弱点がある:

- **中止不可**: 生成と投稿が同一実行内で起きるため、事故投稿を途中で止められない
- **事後分析困難**: 下書きが残らないため、何が生成されて何が投稿されたかのスナップショットがない
- **デバッグ不能**: 中間状態（ファクトチェック結果・品質レビュー指摘）が残らない
- **異常投稿の自動ブロック不可**: factCheckSkippedフラグ等の異常検知時も即投稿される

#### 4.4.2 Phase 3分割後の動作

Phase 3分割後は、**全モード（manual/validate/auto）で Phase A 終了時に必ず saveDraft が呼ばれる**。モードによる差は Phase C での処理だけになる。

```
Phase A（全モード共通）
  └─ saveDraft（下書き保存）← ★モード問わず必ず実行
     ステータス: 「Phase A完了」

Phase B（全モード共通）
  └─ 下書きを上書き更新
     ステータス: 「Phase B完了」

Phase C（モードで分岐）
  ├─ manual   → 承認メール送信。ステータス: 「承認待ち」
  ├─ validate → 通過→投稿+完了 / NG→メール+承認待ち
  └─ auto     → 投稿+ステータス「完了」
```

#### 4.4.3 新旧比較

| 項目 | 旧auto（v12.6まで） | 新auto（v12.7・Phase分割後） |
|------|---------------------|------------------------------|
| 下書き作成 | なし（skip） | **必ず作成** |
| 緊急中止ウィンドウ | なし（即投稿） | **約2分間（Phase A終了〜Phase C実行）** |
| 事後分析 | 投稿履歴のみ | 下書き+投稿履歴の両方 |
| ファクトチェック結果の保全 | ScriptProperties一時保存 | 下書きシート列に永続保存 |
| 異常フラグによる自動ブロック | なし | Phase Cで再確認→manualに降格 |
| 1トリガー内の処理時間 | 最大5-6分（逼迫） | 各Phase 3-4分（余裕あり） |

#### 4.4.4 緊急中止ウィンドウの活用

Phase 分割後、**Phase A 完了 〜 Phase C 投稿実行までの約2分間**は「緊急中止可能期間」となる。以下の手段で中止できる:

1. 下書きシートのステータスを「中止」に手動変更 → Phase C で検知して投稿をスキップ
2. GASトリガー一覧から Phase B/C のトリガーを手動削除 → 以降のPhaseが実行されない
3. カスタムメニュー「緊急停止（全トリガー削除）」を実行

この仕組みは完全自動運用の「最後の安全弁」となる。人間が常に監視する必要はないが、異常投稿を偶然発見した時に止める手段が確保される。

#### 4.4.5 既存 handleAutoMode_ の扱い

Phase 3分割を実装すると、main.gs の handleAutoMode_ / handleValidateMode_ / handleManualMode_ の役割が変わる:

- **旧**: generatePost の戻り値を直接受け取り、その場で投稿判断
- **新**: Phase C内でモード分岐を実装。旧関数はPhase Cから呼ばれるヘルパーとして残すか、Phase Cに統合する

実装時はリグレッション防止のため、旧関数を残してPhase Cから呼び出す方針が安全。

**処理内容**:

2. 各行に対して日付パターンを抽出:
   - `(\d{1,2})月(\d{1,2})日`
   - `(\d{1,2})\/(\d{1,2})`
   - `April (\d{1,2})`, `Apr (\d{1,2})`
3. 抽出した日付が今日より後なら、**その段落全体を削除**
4. ログ出力: `⚠️ 未来日付「4月17日」を検出 → ニュース段落削除`

### 5.2 ガード2: 投稿本文チェック

**実装場所**: postProcessor.gs に新関数 `removeFutureDateLines_` を追加

**処理内容**:
1. 投稿本文を改行で分割
2. 各行を日付パターンで検査
3. 未来日付が含まれ、かつ「過去形文脈」なら該当行を削除
4. 削除後の本文を再組み立て

### 5.3 ガード3: 最終事実検証の強化

**実装場所**: geminiApi.gs の finalFactVerify_ 内

**処理内容**:
- Claudeへのプロンプトに「今日は2026年4月16日。明日以降の日付の出来事を過去形で書いていないか」を明示
- 確定データに「本日: YYYY-MM-DD」を追加
- Claudeが検出した未来日付エラーもJSONで返却される

### 5.4 ★NEW: 過剰反応（False Positive）防止ロジック

NotebookLM指摘2を反映。未来日付を全て削除するのではなく、**過去形文脈のみを削除対象にする**判定ロジックを厳密に定義する。

**削除対象（過去形・完了形文脈）**:
- 「〜した」「〜された」「〜した」「〜を表明」「〜と発言」「〜が起きた」
- 「〜が示された」「〜を受けて」「〜を発表」
- 例: 「4月17日、トランプが解任示唆」「4/17、米CPIが予想上振れ」

**削除対象外（未来予定・予想文脈・ホワイトリスト）**:
- 「〜を控える」「〜が予定」「〜の予想」「〜を見込む」
- 「〜に注目」「〜が焦点」「〜の可能性」「〜を控えた」
- 「〜発表予定」「〜に公表」
- 例: 「4月17日、米CPI発表を控える」「4/17のFOMCに注目」

**判定フロー**:
```
1. 日付パターン抽出
2. 抽出日付が今日より後か？
   No → スキップ
   Yes → 次へ
3. その文脈に過去形/完了形動詞が含まれるか？
   Yes → 削除
   No → 未来予定として保持
4. 判定結果をログ出力（削除/保持の理由を明示）
```

**testPostProcessor.gs テストケース**:

| 入力 | 期待結果 |
|------|---------|
| 「4/17、トランプが解任示唆」 | 削除 |
| 「4月17日に米CPI発表予定」 | 保持 |
| 「4/17のFOMCに注目」 | 保持 |
| 「4月17日、パウエルが発言」 | 削除 |
| 「来週4/17〜4/18のイベント」 | 保持（「予定」文脈） |
| 「4/17、米雇用統計の結果が上振れ」 | 削除 |
| 「昨夜、パウエルが」 | 保持（日付依存なし） |

### 5.5 ★NEW: ガード発動ログの蓄積

削除された内容を**「未来日付ガード発動ログ」シート**に記録し、週次でレビューする。

- 保存内容: 日時・投稿タイプ・削除された文・判定理由
- 過剰反応（保持すべき文が削除された）を発見したら判定ロジックを修正

### 5.6 ★NEW: 継続中重大事象ガード（v1.3追加）

#### 5.6.1 なぜ必要か（2026-04-17 INDICATOR事故）

未来日付ガード（Phase 1）は「未来の出来事を過去形で書く」ハルシネーションには効くが、**逆方向のハルシネーション**には効かない。

2026-04-17のINDICATOR投稿で発覚した新種の事故:

```
誤: 「英国2月GDPが本日発表。→関税ショック前の数字ですが、...」
正: 関税ショックは2025年3月から継続中。2月GDPは関税ショックの真っ只中のデータ
```

これは「過去から継続中の事象を未発生扱いする」類のハルシネーション。3段のファクトチェック全てが見逃した。

#### 5.6.2 根本原因

**「継続中の重大事象」が確定データに含まれていない**。factCheckPost_ / qualityReviewPost_ / finalFactVerify_ のいずれもが、これらの事象を知識として持っているが「今も継続中」という事実を明示的に確認するステップがない。

AIの内部知識では「関税発動」は2025年の出来事として記憶されているが、それを「現在も継続中」と結びつけて検証するロジックがない。結果、「関税ショック前の数字」という現在形の誤記を検出できない。

#### 5.6.3 対策: 確定データに「継続中事象シート」を追加

スプレッドシートに新シート「**継続中重大事象**」を作成:

| 列 | 内容 | 例 |
|---|-----|-----|
| A | 事象名 | 米国関税措置 |
| B | 開始日 | 2025-03-12 |
| C | 最新更新日 | 2026-02-21 |
| D | 現状 | 継続中 / 終了 |
| E | 概要 | アルミ・鉄鋼25%（2025/3/12）→相互関税10-50%（2025/4/2）→最高裁判決（2026/2/20）→15%引き上げ（2026/2/21） |
| F | 影響する投稿タイプ | INDICATOR, LUNCH, LONDON, NY, NEXT_WEEK |
| G | 注意キーワード | 関税前 / 関税ショック前 / 関税発動前 |

#### 5.6.4 運用フロー

```
Phase A: テキスト生成時
  ↓
Phase B: 最終事実検証（finalFactVerify_）
  ├─ 通常の確定データ（レート・要人・政策金利・カレンダー）
  └─ ★v1.3追加: 継続中重大事象シートを確定データとして注入
       + Claudeに「投稿本文に『〜前』『〜以前』等の表現があれば、
          その事象が確定データで継続中と記されていないか確認せよ」を指示
  ↓
  矛盾検出 → 修正 or 削除
```

#### 5.6.5 プロンプト強化（finalFactVerify_）

geminiApi.gs finalFactVerify_ のプロンプトに以下を追加:

```
【継続中重大事象チェック（最重要）】
以下の事象は投稿時点で継続中であることを確認せよ:
[継続中重大事象シートの全行を注入]

★投稿本文に以下のような表現があれば要警戒:
- 「〜前」「〜以前」「〜発動前」「〜ショック前」「〜が起きる前」
- 「〜が来る前」「〜が始まる前」
継続中事象を「まだ来ていない」「起きていない」と書いていたら
ハルシネーション。correctには「〜が続く中」「〜の影響下で」
等に書き換えよ。
```

#### 5.6.6 継続中重大事象シートの初期データ（2026-04-17時点）

Phase 2実装時に以下のデータで初期化する。**週次でコンパナさんが手動更新**（AIに任せない・一次ソースで確認した情報のみ）:

| 事象 | 開始 | 現状 | 注意キーワード |
|------|------|------|--------------|
| 米国関税措置 | 2025-03-12 | 継続中 | 関税前, 関税ショック前 |
| 米イラン対立 | 2026-年前半 | 継続中 | イラン危機前, 中東危機前 |
| ホルムズ海峡関連 | 2026-年前半 | 継続中 | 海峡封鎖前, 原油ショック前 |

※ データの正確性は別途コンパナさんと一次ソース確認で詰める

#### 5.6.7 メンテナンス方針

- 継続中事象シートは**AIが自動更新しない**（ハルシネーションの原因を入れたら本末転倒）
- 週次レビューでコンパナさんが一次ソース（Reuters, Bloomberg, 政府発表）を確認して手動更新
- 事象が終了したら「現状」列を「終了」に変更。注入対象から外れる
- 新しい継続中事象が発生したら行を追加

#### 5.6.8 False Positive対策

「関税ショック前の景気後退」等、**本当に関税発動前のことを指している**場合もありうる。誤削除を防ぐため:

- 時期を明示する表現（「2024年当時」「パンデミック前の」等）がある場合は保持
- Claudeに「時期の明示がない『〜前』は警戒」と指示
- 削除ではなく「書き換え提案」を第一選択に

```

  ↓

  ↓

  └─ auto     → 「完了」
  
中止ケース:
  ├─ 人間が手動で「中止」に書き換え → Phase Cが検知してスキップ
  └─ Phase Bエラー → 「エラー」→ 管理者メール通知
```

**Phase Cが下書きを読む時は必ずステータスを確認する**。「中止」なら即 return。「完了」なら重複実行防止のため return。

## 7. ★NEW: 対話型検証（v1.4の核心）

### 7.1 なぜ必要か

現状の3段検証（factCheckPost_ / qualityReviewPost_ / finalFactVerify_）は**チェックリスト型**で、投稿本文から独自に検証質問を抽出していない。想定外の観点を見逃す構造的欠陥がある。

2026-04-17の2事故（INDICATOR「関税ショック前」/ MORNING「RBA副総裁発言」）は、既存3段検証を全て通過または検出漏れ。一方、コンパナさんが手動でやる検証は毎回見抜いてきた。

### 7.2 コンパナさんの手動検証プロセス（=これをシステム化する）

```
Step A: Claudeが投稿を生成
Step B: コンパナさんが投稿本文を読む
Step C: 違和感のある主張を抽出する
        例:「関税ショック前って本当?」「RBA副総裁が発言した? いつ?」
Step D: Claudeに「これ正しい?」と質問
Step E: ClaudeがWeb検索で事実確認
Step F: 矛盾があれば修正案提示
Step G: コンパナさんが承認/再考
```

### 7.3 システム化: 3ステップの対話型検証

Phase B（品質整形Phase）内で以下を実行:

#### 7.3.1 Step 1: 検証質問抽出

Claudeに投稿本文を読ませ、事実確認が必要な主張を最大5個抽出:

```
抽出対象:
- 日付・時刻を含む主張（例: 「本日15:00発表」）
- 具体的な数値（例: 「約8割」「4.10%」）
- 要人発言（例: 「〇〇が発言」）
- 継続的事象への言及（例: 「関税ショック前」）
- 時系列の因果（例: 「昨夜の流れを受けて」）

抽出しない対象:
- 相場観・感想（「注目ですね」「〜が怖い」）
- 一般的なトレード知識
- 投稿タイプの役割（MORNING/INDICATOR等の導線）

出力形式（JSON）:
{"claims": [
  {"id": 1, "text": "主張の原文", "question": "検証質問（Web検索用）"},
  {"id": 2, ...}
]}
```

**API呼び出し**: Claude 1回（Web検索なし、軽量）

#### 7.3.2 Step 2: 一括Web検証

抽出した質問を1回のClaude呼び出しで全て検証:

```
以下のN個の検証質問を、Web検索で確認せよ。

質問1: [抽出された質問]
質問2: [抽出された質問]
...

【各質問の判定】
- ✅: Web検索で裏付けあり
- ❌: Web検索結果と矛盾（事実誤認）
- ⚠️: 裏付け取れず（削除推奨）

出力形式（JSON）:
{"results": [
  {"id": 1, "verdict": "❌", "evidence": "...", "fix_suggestion": "..."},
  {"id": 2, "verdict": "✅", "evidence": "..."},
  ...
]}
```

**API呼び出し**: Claude 1回（Web検索ツール有効）。Web検索は質問数に応じてClaudeが内部で実行するため追加APIコストなし。

**重要**: N個の質問を1回にまとめることで、N回の個別呼び出しより圧倒的に効率的。

#### 7.3.3 Step 3: 修正

❌または⚠️判定の主張がある場合、修正を実行:

```
以下の投稿本文を、指摘された事実誤りを修正せよ。
ただし、口調・文字数・構造は維持し、事実部分のみ書き換える。

投稿本文:
[原文]

修正指示:
1. 「関税ショック前」→「関税ショックの最中の」に書き換え（関税は2025年3月から継続中）
2. 「RBA副総裁が発言」→ 削除（具体的発言の裏付けなし）

出力: 修正後の本文のみ
```

**API呼び出し**: Claude 1回（修正不要なら0回）

### 7.4 対話型検証の合計コスト

| ケース | API呼び出し |
|-------|-----------|
| 修正不要（全✅） | Claude 2回（抽出 + 検証） |
| 修正必要 | Claude 3回（抽出 + 検証 + 修正） |

## 7. エラーハンドリング

### 7.1 Phase A失敗時

- エラーメール送信
- Phase B/Cトリガーは設定しない（実行されない）
- 下書きは保存されない

### 7.2 Phase B失敗時

- Phase Aの初稿を使って Phase C を実行
- エラーメール送信（「Phase Bエラー。初稿のまま進行」）
- 品質は落ちるが、投稿自体は継続

### 7.3 Phase C失敗時

- エラーメール送信
- 下書きは残るため、手動で再実行可能

### 7.4 ファクトチェック失敗時（factCheckSkipped=true）

- Phase Bで強制的にPOST_MODEを一時的にmanualに切り替え
- 承認メールで「⚠️ ファクトチェック未実行」を大きく表示
- ユーザーが目視確認するまで投稿しない

**問題**: Phase Aで例外ハンドラも含めて落ちた場合、エラーメールも送られず、Phase Bトリガーも設定されず、**何も起きないまま朝を迎える**。

**対策**: 別トリガーで15分おきに状態チェック:

```
function heartbeatCheck() {
  今日の投稿予定時刻を走査

    - 下書きがない
    - かつエラーメールも送られていない（本日分）
  ↓
  → 「システム沈黙検知」として異常メール通知
}
```

**実装場所**: scheduler.gs に新関数 `heartbeatCheck_` 追加。トリガーは scheduleTodayPosts 時に15分おきで設定。

### 18.2 改善方針: `collectAnchorData_` 共通関数の新設

**構造化オブジェクトを返す関数**を新設する:

```javascript
/**
 * 確定データ(Anchor Data)を構造化オブジェクトとして収集する共通関数
 *
 * @param {Object} rates - Twelve Data APIから取得した現在レート
 * @param {Object} keys - getApiKeys()の戻り値
 * @param {Object} [options] - { includeCalendar: true, includeAnomaly: false, calendarScope: 'today' }
 * @return {Object} 構造化された確定データオブジェクト
 */
function collectAnchorData_(rates, keys, options) {
  var opts = options || {};

  // === 1. 基本情報 ===
  var now = new Date();
  var today = {
    iso: Utilities.formatDate(now, 'Asia/Tokyo', 'yyyy-MM-dd'),
    jp: Utilities.formatDate(now, 'Asia/Tokyo', 'yyyy年M月d日(E)'),
    timestamp: now.getTime()
  };

  // === 2. 確定レート ===
  var rateData = {
    current: rates || {},
    open: null,              // 本日始値(取得失敗時はnull)
    direction: null          // { usdjpy: 'up' | 'down', ... }
  };

  try {
    var ss = SpreadsheetApp.openById(keys.SPREADSHEET_ID);
    var openRates = getTodayOpenRates_(ss);
    if (openRates) {
      rateData.open = openRates;
      rateData.direction = computeRateDirection_(rates, openRates);
    }
  } catch (e) {
    console.log('⚠️ collectAnchorData_: 始値取得失敗(続行): ' + e.message);
  }

  // === 3. 通貨強弱ランキング ===
  var currencyStrength = [];
  try {
    var hotPairResult = detectHotPair_(rates, keys.SPREADSHEET_ID);
    if (hotPairResult && hotPairResult.csRanking) {
      currencyStrength = hotPairResult.csRanking;
    }
  } catch (e) {
    console.log('⚠️ collectAnchorData_: 通貨強弱取得失敗(続行): ' + e.message);
  }

  // === 4. 政策金利(テキスト形式、既存関数流用) ===
  var policyRates = '';
  try {
    policyRates = getPolicyRatesText_();
  } catch (e) { }

  // === 5. 要人リスト(テキスト形式、既存関数流用) ===
  var worldLeaders = '';
  try {
    worldLeaders = getWorldLeadersText_();
  } catch (e) { }

  // === 6. 経済カレンダー(optionalで取得) ===
  var calendar = null;
  if (opts.includeCalendar !== false) {
    try {
      calendar = getEconomicCalendar_(opts.calendarScope || 'today');
    } catch (e) { }
  }

  // === 7. 継続中事象(v12.7のタスク13-15で実装済み) ===
  var ongoingEvents = [];
  try {
    ongoingEvents = getOngoingEvents_();
  } catch (e) { }

  // === 8. 構造化オブジェクトとしてreturn + フォーマッター付き ===
  return {
    today: today,
    rates: rateData,
    currencyStrength: currencyStrength,
    policyRates: policyRates,
    worldLeaders: worldLeaders,
    calendar: calendar,
    ongoingEvents: ongoingEvents,

    // ===== フォーマッターメソッド =====

    /**
     * 「確定データ」形式の文字列に変換(finalFactVerify_, factCheckPost_スタイル)
     */
    toFactString: function() {
      var txt = '';
      txt += '【本日の日付(最重要の基準)】\n';
      txt += this.today.jp + '(ISO: ' + this.today.iso + ')\n\n';

      if (this.rates.current && Object.keys(this.rates.current).length > 0) {
        txt += '【確定レート(APIリアルタイム値)】\n';
        for (var i = 0; i < CURRENCY_PAIRS.length; i++) {
          var p = CURRENCY_PAIRS[i];
          if (this.rates.current[p.key]) {
            txt += p.symbol + '(' + p.label + '): ' + formatRate_(this.rates.current[p.key], p.key, 'verify') + '\n';
          }
        }
        txt += '\n';
      }

      if (this.rates.open && this.rates.direction) {
        txt += '【本日始値(方向判定の基準)】\n';
        txt += 'USD/JPY始値: ' + formatRate_(this.rates.open.usdjpy, 'usdjpy', 'verify') + '\n';
        if (this.rates.direction.usdjpy === 'up') {
          txt += '→ 本日のドル円は始値より上昇(= 円安方向)\n\n';
        } else {
          txt += '→ 本日のドル円は始値より下落(= 円高方向)\n\n';
        }
      }

      if (this.currencyStrength.length > 0) {
        txt += '【通貨強弱(実測値)】\n';
        for (var ci = 0; ci < this.currencyStrength.length; ci++) {
          var cs = this.currencyStrength[ci];
          txt += cs.currency + ': ' + (cs.score >= 0 ? '+' : '') + cs.score.toFixed(2) + '% ' + cs.direction + '\n';
        }
        txt += '\n';
      }

      if (this.policyRates) { txt += '【主要中銀の政策金利】\n' + this.policyRates + '\n'; }
      if (this.worldLeaders) { txt += '【主要国の首脳・要人】\n' + this.worldLeaders + '\n'; }

      if (this.calendar) {
        txt += '【経済カレンダー】\n' + this.calendar + '\n';
      }

      if (this.ongoingEvents.length > 0) {
        txt += '【継続中の事象(誤った時制で書かないこと)】\n';
        for (var oi = 0; oi < this.ongoingEvents.length; oi++) {
          txt += '- ' + this.ongoingEvents[oi].event + ' (' + this.ongoingEvents[oi].since + '〜)\n';
        }
      }

      return txt;
    },

    /**
     * 対話型検証 Step2(Web検証)に最適化した文字列
     * 「これを基準に主張を検証せよ」の指示に使う
     */
    toVerifyPrompt: function() {
      var txt = '【検証の基準となる確定データ】\n';
      txt += 'Web検索結果がこの確定データと矛盾する場合は、確定データを優先せよ。\n\n';
      txt += this.toFactString();
      return txt;
    },

    /**
     * 対話型検証 Step3(修正)に最適化した文字列
     * 「この確定データを守って修正せよ」の指示に使う
     */
    toFixPrompt: function() {

    }
  };
```

### 18.3 ヘルパー関数 `computeRateDirection_`

```javascript
/**
 * 現在レートと始値から、各通貨ペアの方向(up/down)を算出
 */
function computeRateDirection_(currentRates, openRates) {
  var direction = {};
  for (var key in openRates) {
    if (currentRates[key] && openRates[key]) {
      direction[key] = Number(currentRates[key]) > Number(openRates[key]) ? 'up' : 'down';
    }
  }
  return direction;
}
```

## 19. ★NEW(v1.9): チェックリスト型検証の段階的削減方針

### 19.1 現状の問題: 3層検証が「同じ仕事を3回やっている」

| 層 | 関数 | 検証方式 | 検証対象 |
|----|------|---------|---------|
| 1 | `factCheckPost_` | Gemini Grounding | 確定データ照合 + Web検索 |
| 2 | `qualityReviewPost_ Q6` | Claude Web検索 | 事実の信憑性 |
| 3 | `finalFactVerify_` | Claude JSON + コード置換 | 確定データガード |

**3層とも「事実を確認する」仕事**だが、**どれも投稿本文から検証質問を抽出していない**。チェックリスト(確定データ一覧)と照合するだけ。

→ 「関税ショック前」「RBA副総裁発言」のような**想定外の観点**を根本的に検出できない構造。

### 19.2 対話型検証導入後の理想形

対話型検証は**「本文から検証質問を抽出して、Web検索で確かめる」**という、今まで誰もやっていなかった仕事をする:

```
【チェックリスト型】 vs 【対話型】

チェックリスト型:
  「このデータと矛盾してないか?」
  → データにない観点は検出不能

対話型:
  「この本文で検証が必要な主張は何か?」
  → まず質問を作る段階で、想定外の観点もカバーされる
```

### 19.3 削減ロードマップ

**現状(v12.6.1)**:

## 21. ★NEW(v1.9): テスト戦略 - 3レイヤー設計

### 21.1 テスト設計の基本方針

**「本番投稿を待たずにロジックを検証できる」**ことを最優先とする。対話型検証は:
- 本番稼働時は1日5-6回しか動かない
- 各実行で確定データ(APIクレジット)を消費する
- 問題が出ても次の実行まで1時間以上待つ必要がある

→ **単体テスト + モックデータで瞬時に検証できる仕組み**が必要。

### 21.2 3レイヤー構成

**レイヤー1: 単体テスト(各ステップを独立検証)**

| テスト関数 | 目的 | 所要時間 |
|-----------|------|---------|
| `testExtractClaims()` | Step 1単体: 質問抽出のJSON応答が正しいか | 約10秒(Claude API 1回) |
| `testBatchWebVerify()` | Step 2単体: Web検証のverdict判定が正しいか | 約20秒(Claude + Web) |
| `testFixPost()` | Step 3単体: 修正が口調・文字数を維持するか | 約10秒(Claude API 1回) |
| `testCollectAnchorData()` | `collectAnchorData_` の全フィールド取得検証 | 約5秒(API不要) |

**レイヤー2: 既知事故の再現テスト(回帰テスト)**

| テスト関数 | 事故#(v1.8対応) | 期待結果 |
|-----------|-------------------|---------|
| `testCase_RBAFukuSosai()` | 事故#3: MORNING「RBA副総裁発言」 | Step2で ⚠️ 判定、Step3で削除 |
| `testCase_TariffShockBefore()` | 事故#2: INDICATOR「関税ショック前」 | Step2で ❌ 判定、Step3で書き換え |
| `testCase_TrumpPowellDismissal()` | 事故#1: MORNING「パウエル解任示唆」(未来日付) | Step1で抽出、Step2で ❌、Step3で削除 |
| `testCase_CleanPost()` | 正常な投稿(誤検出防止) | Step1で抽出は発生、Step2で全 ✅、Step3スキップ |

**これがポイント**: **過去の事故のテキストを固定文字列として保存**し、毎回実行する。新しい修正で既存の検出力が落ちたら即検知できる。

```javascript
// 例: 事故#3の再現テスト
function testCase_RBAFukuSosai() {
  var incidentPost = '☕ おはようございます\n' +
    '📝RBA副総裁が「利上げが必要」と発言。豪ドル買いが入りやすい展開に。\n' +
    '→ オージー円は注目ですね\n\n' +
    '💡今日の仮説: 日銀会合を控え、JPYはやや神経質な動き\n\n' +
    '#FX #豪ドル円';

  console.log('=== 事故#3 再現テスト開始 ===');
  var result = executeInteractiveVerify_(incidentPost, 'MORNING', {}, {});

  // 検証
  var passed = true;
  if (result.extractedCount === 0) {
    console.log('❌ Step1失敗: claimが抽出されなかった');
    passed = false;
  }
  if (result.warnCount === 0 && result.ngCount === 0) {
    console.log('❌ Step2失敗: RBA副総裁発言が裏付けなし/矛盾と判定されなかった');
    passed = false;
  }
  if (!result.fixApplied) {
    console.log('❌ Step3失敗: 修正が実行されなかった');
    passed = false;
  }

  console.log(passed ? '✅ 事故#3 再現テスト: 合格' : '❌ 事故#3 再現テスト: 不合格');
  return passed;
}
```

**レイヤー3: 統合テスト(End-to-End)**

| テスト関数 | 目的 |
|-----------|------|
| `testIntegratedDirty()` | 意図的に汚染した投稿を食わせて、修正後が改善しているか |

**全モード共通の目標設計** (要件定義セクション4.4.2):
```
Phase A（全モード共通）
  └─ saveDraft（下書き保存）← モード問わず必ず実行
     ステータス: 「Phase A完了」

Phase B（全モード共通）
  └─ 品質レビュー + 対話型検証
     ステータス: 「Phase B完了」

Phase C（モードで分岐）
  ├─ manual   → 承認メール送信。ステータス: 「承認待ち」
  ├─ validate → 通過→投稿+完了 / NG→メール+承認待ち
  └─ auto     → 投稿+ステータス「完了」
```
