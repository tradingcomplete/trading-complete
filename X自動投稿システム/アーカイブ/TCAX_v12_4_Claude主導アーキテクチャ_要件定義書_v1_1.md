## 1. v12.1→v12.4で何が変わったか

### 1-1. v12.1の設計（3段階生成）

```
Stage 1: Claude → 市場分析（データを読む）
Stage 2: Gemini → 投稿生成（書く）          ← ここが問題
Stage 3: Claude → 品質レビュー（チェック）
         Gemini → 品質修正（直す）           ← ここも問題
```

### 1-2. v12.1〜v12.3で発覚した問題

```
【問題1: Geminiが方向を無視する】
  通貨強弱: AUD +2.95%（最強）
  Geminiが書いた投稿: 「豪ドル売り加速、本日大きく下落」
  → Geminiは「地政学リスク→リスクオフ→資源国通貨売り」という一般論のロジックで
    確定データを上書きする。Claude市場分析が正確でも、Geminiが無視する。

【問題2: 修正ステップが正しい投稿を壊す】
  Claude生成: 「AUD +2.92%の上昇」 ← 正しい
  Claude品質レビュー: 「過去投稿に『下落』とある。矛盾」← ハルシネーション
  Gemini修正: 「上昇→下落」に書き換え ← 正しい投稿を壊した

【問題3: Gemini 503長時間化】
  2026年4月13-14日、Gemini Pro 503が半日以上継続。
  Claudeフォールバックで生成は完走するが、修正ステップもClaude429で失敗。
  修正なしの投稿が出力される。

【問題4: クロスチェックの限界】
  理論: 異なるモデルでチェックすれば品質が上がる
  現実: 問題を検出しても修正できなければ意味がない。
        検出率は高いが修正率がゼロの状態が発生。
```

### 1-3. 根本的な学び

```
クロスチェック（異モデル検証）より、正しいデータ（確定データ）を握っていることの方がはるかに重要。
通貨強弱+2.89%という数字があれば、どのモデルが何を言っても「AUDは上昇」が覆らない。
```

## 2. v12.4 アーキテクチャ（Claude主導）

### 2-1. 設計思想

```
旧: Geminiが書く → Claudeがダメ出し → Geminiが直す（クロスチェック）
新: Claudeが最初から正しく書く → 確定データで守る（確定データガード）
```

### 2-2. パイプライン

```
【Step 1: ニュース取得（Gemini + Grounding）】← Geminiが唯一得意なこと
  Gemini + Google検索Grounding でTOP5ニュースを取得
  1時間キャッシュで再利用

【Step 2: 市場分析（Claude）】← 変更なし
  為替レート + 通貨強弱 + 商品価格 + ニュースTOP5
  → データを正しく読み、何が起きているかを分析

【Step 3: 投稿生成（Claude）】← ★v12.4変更: Gemini→Claude
  buildPrompt_の22,000文字プロンプト + Claude市場分析
  → Claudeがデータに忠実にライティング
  末尾にキャラクター口調リマインダー配置

【Step 4: 後処理チェーン（コード）】← 変更なし
  14段の機械的テキスト整形

【Step 5: ファクトチェック検出（Gemini + Grounding）】← 変更なし（Groundingの価値）
  確定データ6（通貨強弱）による方向チェック追加（★v12.4）
  「AUD +2.95%なのに豪ドル下落と書いている」→ ❌

【Step 6: ファクトチェック修正（Claude + 確定データガード）】← ★v12.4変更: Gemini→Claude
  通貨強弱データを修正プロンプトに注入
  「確定データの方向と矛盾する修正は絶対にするな」ルール

【Step 7: 品質レビュー（Claude + Web検索）】← 変更なし
  Q1〜Q7（★Q7に体言止め判定追加）

【Step 8: 品質修正（Claude + 確定データガード）】← ★v12.4変更: Gemini→Claude
  確定データ（レート + 通貨強弱）を修正プロンプトに注入
  「確定データ > 品質レビュー指摘」ルール
```

### 2-3. Geminiの役割（v12.4以降）

```
Geminiが担当するもの（2つだけ）:
  ① ニュース取得（fetchMarketNews_）: Google検索Grounding
  ② ファクトチェック検出（factCheckPost_）: Google検索Grounding

Geminiが担当しなくなったもの:
  ✗ 投稿テキスト生成 → Claude
  ✗ ファクトチェック修正（autoFixPost_）→ Claude
  ✗ 品質修正 → Claude
  ✗ リトライ（主役ペア等）→ Gemini（レガシー。Phase 3で変更予定）
```

### 2-4. 確定データガード（v12.4の核心）

```
スプレッドシートから取得される確定データが「真実のアンカー」として、
全ての修正ステップで方向の矛盾を防ぐ。

注入される確定データ:
  確定データ1: 経済カレンダー（スプレッドシート）
  確定データ2: リアルタイムレート（Twelve Data API）
  確定データ3: 政策金利（スプレッドシート「確定データ」）
  確定データ4: 首脳・要人（スプレッドシート「確定データ」）
  確定データ5: 商品価格（Twelve Data API）
  確定データ6: 通貨強弱ランキング（★v12.4新規）← 方向チェックの核

ルール:
  「確定データの方向と矛盾する修正はするな」
  「品質レビューが『上昇→下落に直せ』と言っても、確定データで上昇なら上昇のまま」
```

## 3. 投稿品質の改善（v12.4）

### 3-1. 体言止めルール

```
場所: promptBuilder.gs + qualityReview.gs（Q7）
内容: 絵文字行は速報ヘッドライン調の体言止め・動詞止めで書く

OK: 「📝米イラン交渉が決裂。」「🛢原油、100ドル突破。」
NG: 「📝米イラン交渉が決裂しました。」「🛢原油が100ドルを突破しています。」

Q7判定基準:
  ■ 感想禁止（従来）: 「驚きました」「マジで」等の感想混入
  ■ 体言止め必須（★v12.4追加）: 「〜しました。」「〜しています。」は冗長
```

### 3-2. キャラクター口調リマインダー

```
場所: promptBuilder.gs（プロンプト最末尾に配置）
問題: 24,000文字のデータ・ルールを処理するうちにキャラクターの口調が薄まる
対策: プロンプトの最後に具体的なNG/OK例付きのリマインダーを配置

NG: 「市場では〇〇が後退。」← ニュース記事
OK: 「利上げの話、正直ちょっと遠のいた感じですね。」← コンパナ
```

### 3-3. 画像の時間表現

```
場所: imageGenerator.gs
問題: 投稿テキストの「本日15:15」を画像生成時にGeminiが「今夜」に変換
対策: 「時間表現を原文から変えるな」ルールを追加
```

### 3-4. 月曜MORNING時間変更

```
場所: config.gs
問題: 月曜7:28は原油先物開場（7:00）から28分後。週末ニュースが未インデックス
対策: 月曜MORNINGを08:03に変更（原油先物開場+1h確保）
```

## 6. 設計の鉄則（v12.4追加分）

```
【v12.4で追加された鉄則】

・確定データ > AI推論。通貨強弱+2.89%があれば、どのモデルが何を言っても「上昇」が事実。
・クロスチェック（異モデル検証）より、正しいデータを持っていることが重要。
・修正ステップには必ず確定データを注入せよ。データなしの修正は方向を壊す。
・Geminiは「検索（Grounding）」専用。「ライティング」と「修正」はClaude。
・プロンプトの最末尾にキャラクター口調リマインダーを配置（LLMは最後の指示を最重視）。
・絵文字行は体言止め・動詞止め。「〜しました。」「〜しています。」は冗長で禁止。
・「今日はここまで」にしない。テストで問題が見つかったら最後まで修正しきること。
・やってみて初めて分かることがある。ダメだったことが分かったことが完成に近づく。
```
