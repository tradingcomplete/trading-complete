## 3. Phase 1.5: ファクトチェック + 自動修正 ✅実装完了

### 3-1. 概要

投稿テキスト生成後、事実の正確性を自動検証し、誤りがあれば自動修正する。
★v12.0: 3段構成（Geminiファクトチェック → Claude事実検証 → Gemini修正）

### 3-2. 処理フロー

```
(1) テキスト生成（Grounding OFF）→ 後処理 → リトライ

(2) factCheckPost_（Gemini + Grounding ON）← チェック1
    → Layer 1: システム確定データと照合
      ・為替レート（7ペア）: 3%超の乖離で❌
      ・★v12.0: 商品価格（WTI/BTC/ゴールド/天然ガス）: 5%超の乖離で❌
      ・政策金利・カレンダー・要人・アノマリー
    → Layer 2: Grounding検索で事実確認
    → JSON形式で構造化された検証結果を返す

(3) 判定分岐:
    ✅全て正確 → (5)へ
    ❌誤り → (4)へ

(4) autoFixPost_（Gemini、Grounding OFF）
    → 誤りを修正 → 再度後処理を適用

(5) qualityReviewPost_（Claude Sonnet 4.6）← チェック2+3
    → Q6: 確定データ（レート+商品価格）との矛盾を検出
    → Q7: 絵文字行の感想混入を検出
    → Q1〜Q5: 品質チェック
    → Claude指摘 → Gemini修正 → コード側文字数保証

(6) 承認メール送信
    → 修正済みテキスト + ファクトチェック結果 + 品質レビュー結果を表示
```

### 3-3. 検証対象

```
【検証する事実】
  ・要人発言: 誰が、いつ、何を言ったか
  ・経済指標: 名称、日付、予想値、前回値、結果値
  ・レート水準: 具体的な数値の正確性
  ・★v12.0: 商品価格: WTI・BTC・ゴールド・天然ガスの具体値
  ・日付の整合性: 「今日」「昨日」の出来事が本当にその日か
  ・中央銀行の政策: 決定内容の正確性

【検証しない】
  ・個人の感想（「マジで注目」等）
  ・相場観（「上がりそう」等）
  ・RULE_1〜4（個人の経験談が主体）
```

### 3-4. 対象ファイル・関数

```
factCheck.gs:
  factCheckPost_()   → ファクトチェック（Gemini+Grounding ON）
                       ★v12.0: Layer1に商品価格追加（fetchDailyCommodityPrices_/fetchCommodityPrices_）
  autoFixPost_()     → 自動修正（Gemini、Grounding OFF）

qualityReview.gs:
  qualityReviewPost_() → 品質レビュー（Claude Sonnet 4.6）
                         ★v12.0: Q6事実検証+Q7絵文字行チェック追加（5→7項目）
                         ★v12.0: 確定データ（レート+商品価格）をClaudeに注入

geminiApi.gs:
  generatePost()     → フロー末尾にファクトチェック+品質レビュー呼び出し
                       ★v12.0: テキスト生成のGrounding OFF

approval.gs:
  sendDraftNotification()    → メール本文にファクトチェック+品質レビュー結果を追加
  processApprovedDrafts()    → ★v12.0: 投稿直前にスプレッドシートE列を再読み込み
  processPendingRegenRequests_() → ★v12.0: 画像再生成時もE列から最新テキスト取得

ScriptProperties:
  LAST_FACT_CHECK_{postType} → 一時保存（メール送信後に削除）
  SKIP_FACT_CHECK            → 一括テスト時のスキップフラグ
```

### 3-5. v12.0で解決したハルシネーション問題

```
【問題】
  テキスト生成時のGrounding ONで、Geminiが未検証の情報（WTI 113ドル台、
  トランプ発言等）を検索結果から拾い、それを膨らませてハルシネーションを生成。
  同じGeminiによるファクトチェックでは検出が不安定（同モデルバイアス）。

【対策: 3段防御】
  ① 発生防止: テキスト生成のGrounding OFF（検証済みデータだけで書く）
  ② Layer1強化: 商品価格を確定データに追加（WTI等の価格捏造を即検出）
  ③ 別モデル検証: Claude Q6で確定データとの矛盾を二重チェック

【結果】
  全16タイプテストでハルシネーション0件。
  数値の捏造（雇用統計17.8万人等）が残ってもQ6で毎回検出・修正。
```

## 11. 技術メモ

### 11-1. 3段防御アーキテクチャ（★v12.0）

```
【設計思想】
  Groundingの用途を明確に分離:
    リサーチ（fetchMarketNews_）: Grounding ON → ニュース検索
    ライティング（generatePost）: Grounding OFF → 検証済みデータで書く
    検証（factCheckPost_）: Grounding ON → 事実確認

【防御の流れ】
  防御1: 発生防止（Grounding OFFで未検証情報を拾わない）
  防御2: Gemini検出（Layer1確定データ + Layer2 Grounding検索）
  防御3: Claude検出（Q6で確定データとの矛盾を二重チェック）

【効果】
  修正前: テスト3回中1回がハルシネーション素通り
  修正後: テスト16タイプ全通過、ハルシネーション0件
```

### 11-2. 各Phaseのプロンプト注入上限

```
Phase 1.5（ファクトチェック）: 注入なし（検証結果はメールに表示）
Phase 2（仮説検証）: 最大500文字（直近5件のサマリー）
Phase 3（学び重み付け）: 既存枠内（追加文字数なし）
Phase 4（品質フィードバック）: 最大300文字（平均ER、ベスト/ワースト）
Phase 5（ルール更新）: プロンプト自体が変わる（注入ではない）
```
