T-CAX の運用を通じて得られた**事件の学び・ベストプラクティス・解決パターン**を集約した、開発者コンパナ + Claude の共有知識ベース。

T-CAX と Trading Complete は**別システム**。以下のように分けて管理:

| ドキュメント | 対象 | 内容 |
|-------------|------|------|

## 🚨 Part III: 事件簿(学びのある失敗と対処)

### 📌 事件1: runMorning 22:11 異常発火事件(2026-04-17)

#### 症状

- 2026-04-17(金) 22:11、金曜スケジュールに存在しないはずの `runMorning` が時間主導型トリガーで自動発火
- MORNING タイプの投稿が夜に生成された
- 下書き `20260417_221606_MORNING` として保存、Phase B/C まで完走、承認メール待機状態となった

#### 根本原因

**config.gs の SCHEDULE で月〜金の times / types 配列の要素数不整合**。

```javascript
// 事件発生時の状態(v12.7 NY削除時の作業漏れ)
5: { // 金曜
  times: ['07:22', '09:38', '12:11', '17:24', '20:49', '22:11'],  // 6要素
  types: ['MORNING', 'TOKYO', 'LUNCH', 'LONDON', 'GOLDEN']         // 5要素(NY削除済み)
}
```

scheduler.gs は `times[i]` と `types[i]` をペアで処理するが、`i=5` のとき `times[5]='22:11'` が存在するのに `types[5]=undefined` となる。

さらに `getFunctionName_()` のフォールバック:

```javascript
return map[postType] || 'runMorning';
// map[undefined] = undefined → 'runMorning' にフォールバック
```

これにより 22:11 に **意図しない runMorning がトリガー登録**され、金曜の夜に発火した。

#### 対処手順(完了)

1. **緊急対応**: `cleanupPostTriggers_()` + `scheduleTodayPosts()` でトリガー全リセット
   - 削除前: 11件(scheduleTodayPosts / runMorning×2 / runTokyo / runLunch / runLondon / runGolden / processApprovedDrafts / onOpen / scheduledFetchRates / scheduleDailyMaintenance)
   - 削除後: 5件(システム系のみ)
2. **下書き中止**: スプレッドシート「下書き」シートで該当行のステータスを「承認待ち」→「中止」に変更
3. **根本対策**: config.gs の SCHEDULE で月〜金の times 配列から 22:XX を削除。5要素に統一
4. **整合性確認**: `verifyScheduleIntegrity()` で全曜日の times/types 一致を確認

#### 学んだこと

##### ❌ 失敗の本質

**NY 削除を段階的に進める過程で、types 配列からは NY を削除したが、times 配列から 22:XX を削除し忘れた**。差分が小さく目視で気づきにくい箇所だった。

##### ✅ 教訓

1. **対応する配列は要素数を必ず揃える**: times/types のように「ペアで意味を成す配列」の不整合は、静的解析で検出されない実行時バグの温床
2. **無条件フォールバックは問題を隠蔽する**: `|| 'runMorning'` のフォールバックが undefined を隠し、runMorning の誤発火を引き起こした
3. **削除タスクは関連箇所を列挙してから実施**: 「NY 削除」のような横断的変更は、影響範囲を事前に洗い出すべき(types / times / 関数マップ / プロンプト / 画像定義 / トリガー)

##### 🔧 再発防止策

- **構造的検証テスト追加**: `verifyScheduleIntegrity()` (testFunctions.gs・v12.9) を常備
- **将来的なガード追加候補**: scheduler.gs の for ループ内に以下を追加
  ```javascript
  if (postType === undefined) {
    console.error('⚠️ types/times配列不整合: i=' + i + ', timeStr=' + timeStr);
    continue;
  }
  ```
- **削除チェックリスト化**: 投稿タイプ追加/削除時に触るべき箇所の一覧を残す(次回 NY 関連を完全削除する時の参考)

### 📌 事件2: Phase B Q6 RBA 利下げ誤判定事件(2026-04-17)

#### 症状

- 2026-04-17(金) GOLDEN 投稿で Phase A の最終事実検証が「RBAは利下げサイクル」→「利上げサイクル」と**正しく修正**
- しかし **Phase B の Q6(品質レビュー)** が Web 検索で「RBAは2025年2月に利下げ」を引いて**「利上げサイクルは誤り」と逆判定**
- 同じ Claude なのに Phase A と Phase B で真逆の判定が起きた

#### 根本原因

**qualityReview.gs の Q6 に確定データが不完全に注入されていた**。

Phase A の `finalFactVerify_()` には `collectAnchorData_()` 経由で以下が注入されていた:

- 政策金利(RBA 4.10%, 2026年3月17日利上げ)
- 通貨強弱(AUD+3.93% / USD-2.31% 等)
- 継続中重大事象(米国関税措置, 米イラン対立 等)

一方、qualityReview.gs の Q6 には以下しか注入されていなかった:

- 為替レート(7ペア)
- 商品価格(BTC, ゴールド)
- 経済カレンダー

**政策金利・通貨強弱・継続中重大事象が欠落**。その結果、Q6 は **Web 検索のみに依存**して判定。Claude の知識カットオフ(2025年1月頃)と SEO の影響で、**2025年の利下げ関連記事が検索上位**に出やすく、これを「直近の情報」と誤認した。

#### 対処手順(完了)

1. **qualityReview.gs 修正1(135行目付近)**: 経済カレンダー注入の直後に `collectAnchorData_(postType)` の呼び出しを追加。政策金利・通貨強弱・継続中重大事象を注入
2. **qualityReview.gs 修正2(約180行目付近)**: Q6 プロンプトに「判定優先順位」を明示
   ```
   1) 経済カレンダー確定データ
   2) 追加の確定データ(政策金利等)
   3) 本文内の数値データ
   4) Web検索
   5) Claudeの内部知識
   ```
3. **テスト**: `testQReviewRBA()` で意図通りの指摘を確認
   - 期待: Q6 が「RBAは2026年2月と3月に連続で利上げ、現在の政策金利は4.10%」を根拠に指摘
   - 結果: ✅ 期待通り

#### 学んだこと

##### ❌ 失敗の本質

**同じ設計思想(確定データ優先)が Phase A と Phase B で一貫していなかった**。Phase A の finalFactVerify_ では collectAnchorData_ を使っていたが、Phase B の qualityReview.gs では別の(部分的な)確定データ注入ロジックを使っていた。

##### ✅ 教訓

1. **Claude の時系列バイアス**: Claude の知識カットオフと Web 検索結果の新旧が混在する状態では、Claude は「カットオフ以前の情報」を「現在」と誤認する傾向がある
2. **Web 検索は SEO 影響で古い記事が上位に出やすい**: 特に金融政策のように「サイクル」単位で記事が大量生産される領域では、過去サイクルの記事が現在のサイクルを上書きして見える
3. **確定データの一元化**: 同じ情報(政策金利等)を複数箇所で個別実装すると、片方が欠けたときに Phase 間で判定矛盾が起きる。`collectAnchorData_()` のような集約関数を**全ての検証ステップで必ず使う**こと
4. **判定優先順位を明示化**: 「確定データ > Web検索」は言われないと Claude は対等に扱う。プロンプトで明示的に順位を書かないと効かない

##### 🔧 再発防止策

- **全検証ステップで collectAnchorData_ を使う**: 今回の修正で qualityReview.gs にも適用
- **優先順位を明示**: 「1) カレンダー > 2) 確定データ > 3) 本文 > 4) Web検索 > 5) 内部知識」を Q6 プロンプトに書いた
- **シナリオ別テスト常備**: testQReviewRBA() / testQReviewFOMC() / testQReviewOngoingEvents() で定期的に検証
- **対話型検証 Step 4 も同じ設計思想で動作**: Step 2 の Web検証でも確定データ優先が効いている(ログで確認済み)

### 📌 事件3: Gemini 503 多発事件(2026-04-16 以前から継続・観察中)

- Gemini 2.5 Pro API が `503 Service Unavailable` を返すことが頻発

- リトライで回避できるケースもあるが、致命的に詰まるケースもあった

#### 対処方針(v12.4 で部分解決)

**Claude 主導アーキテクチャへの移行**:

- テキスト生成: Gemini → **Claude** に切替(v12.4)
- 品質修正: Gemini → **Claude** に切替(v12.4)
- 最終事実検証: Claude JSON 検出 + コード置換(v12.6)

- ニュース取得(`fetchMarketNews_`) - Grounding 検索は Gemini が得意
- ファクトチェック検出(`factCheckPost_`) - タスク18で削除予定
- 画像生成(`imageGenerator.gs`) - gemini-2.5-flash-image

#### 残課題

- タスク18: factCheck 削除による Gemini 依存度さらに低減
- タスク19: Claude 統一(Gemini 503 フォールバック依存の解消)

#### 学んだこと

- **クロスチェックより「正しいデータを握ること」が重要**(v12.4 設計思想)
- **異モデル併用はコストとリスクを増やす**。確定データで勝負する方が堅牢
- **Gemini の得意領域(Grounding 検索・画像生成)だけ残す**のが現実的な落としどころ

### 📌 事件5: 投稿品質問題の総合改善(2026-04-18 v12.10)

#### 症状

コンパナから「毎投稿、正直おかしい文章が多く、承認時の判断負担が大きい。検証モードに切り替える確信が持てない」という報告。2026-04-18 08:22 の WEEKLY_REVIEW で以下が同時発生:

- 論理矛盾:「仮説は外れた」と「着地点は合っていた」が同居
- 事実誤認:Gemini が生成した「ホルムズ海峡開放」情報を Claude がそのまま採用(翌日覆った情報)
- finalFactVerify 過剰修正:週中の高値更新を現在レートで反論して「高値圏での推移」に劣化
- プロンプト肥大化:29,686文字・92セクション

#### 対処(8段階で実施)

1. **factCheck 無効化**(DISABLE_FACTCHECK=true)
   - Phase A の factCheckPost_ / autoFixPost_ を完全スキップ
   - Claude品質レビュー + 最終事実検証 + 対話型検証の3段構成に簡素化
   - Gemini ハルシネーション経由の悪化ルートを根絶

2. **Q6.5 論理整合性チェック追加**(qualityReview.gs)
   - 「事実は正しいが論理が矛盾する」パターンを独立判定
   - Q6(事実検証)と別観点で実施
   - 「外れた vs 合っていた」「円安進行 vs 円高方向」等を検出

3. **優先度付き修正プロンプト**(qualityReview.gs)
   - 修正指示を Q6.5 > Q6 > Q5/Q7 > Q1/Q2/Q3 > Q4 の順にソート
   - Q6.5 に「★最優先」ラベル、Q6 に「★次点」ラベル

4. **キャラクターシート末尾再注入廃止**(promptBuilder.gs)
   - 冒頭+末尾の2回注入 → 冒頭のみ注入 + 末尾は核心リマインダー3行
   - 12セクション重複を解消

5. **finalFactVerify 過剰修正防止**(geminiApi.gs)
   - 時間軸を含む主張(週中高値・〜年以来・急落率等)は修正対象外に
   - スナップショットレートで反論しない

6. **プロンプト重複削除**(promptBuilder.gs)
   - 【コンパナの口調(絶対遵守)】削除(キャラクターシートの【発信の原則と口調】と重複)
   - 【TCの主な機能】【TC導線の入れ方】ヘッダ削除(内容は保持)
   - buildMarketTypePolicy_ 内の【最重要】ラベル乱発削減(14→11箇所)
   - 【経済イベントの日付確認】と【未発表過去形禁止】を【経済イベントの日付・時制】に統合
   - MORNINGフレームの【今日の仮説】重複解消

7. **キャラクターシート調整**
   - 【共感】5セクションを1つに統合
   - 【軽さと勢い(感嘆符・行動促し・疑問形)】追加
   - 【禁止事項】に「具体エピソード詰め込み禁止」追加

8. **ニュース取得を Gemini → Claude web_search に切替**(marketAnalysis.gs)
   - 原因: Gemini Grounding がソース0件で内部知識(古い)のまま回答する現象
   - 対策: Claude web_search に全面移行・プロンプト強化(内部知識だけは禁止・3-5回検索推奨)
   - ソース0件時はキャッシュ保存せず再取得を試みる設計に変更
   - 実測: 50件のソースを自主的に取得・月12円程度のコスト増

| プロンプト文字数 | 29,686 | 22,677 | 22,677〜24,213 |
| セクション数 | 92 | 72 | 68 |
| Q6.5 論理矛盾検出 | なし | 1件検出→修正 | 1件検出→修正 |

| 投稿品質 | 論理矛盾あり | 改善 | 鮮度・正確性とも良好 |

#### 学んだこと

- **Gemini Grounding は気まぐれ**。「検索が必要」とモデル自身が判断した時だけ検索する仕様で、検索スキップが頻発する
- **Claude は Gemini より指示に忠実**。プロンプトで「必ず web_search を使え」と指示すれば実用レベルで動作する
- **tool_choice 強制は server-side tool(web_search 等) には非対応**。プロンプト強化で代替する
- **キャラクター定義の二重注入はアンチパターン**。LLM の attention 希薄化で逆効果
- **時間軸を含む主張はスナップショット値で反証するな**。週中ピークと現在値は別物
- **「事実は正しいが論理が矛盾」は Q6 では検出できない**。Q6.5 のような独立チェックが必要

**正しい考え方(コンパナの本質)**:

```

```

#### なぜトレーダーはXを見るか(コアインサイト)

**「乗り遅れたくない」「人より早く情報が欲しい」「儲かる材料を探している」**

トレーダーが常に抱える4つの疑問:
1. **新しい材料はないか?**(最新情報)
2. **今のポジションは正しいか?**(検証したい)
3. **持ち続けるか利確するか?**(判断したい)
4. **他のトレーダーはどう見ているか?**(多数派を知りたい)

**価値ある投稿 = この4つの疑問の全部または一部に、新鮮な材料を起点に答えること**

#### コンパナが「面白い」と感じる投稿の4要素

1. **ヘッドニュース**(今日の相場を動かした具体的なトピック)
2. **値動きの背景・なぜ**(因果関係)
3. **ダウ理論**(チャートの構造)
4. **トレンド転換のライン**(「このラインを抜けたら転換」という具体的なレベル)

今朝の LUNCH が「良さげ」と評価されたのは、①②③(条件分岐)の3要素が揃っていたから。MORNINGが「面白くない」と評価されたのは、①が弱く(ゴトー日だけ)②が数字羅列になり④がなかったから。

#### 対話と自動化の違い

対話時(人間コンパナ + Claude):
- コンパナが「今日は原油がこう動いた」と1〜2行で新鮮な材料を渡す
- Claude がコンパナ風に300字書く
- コンパナが修正指摘 → 完成
- **材料が先、Claude は書くだけ**

自動化(T-CAX):
- スプレッドシート23枚 + 全データを全部 Claude に投げつける
- Claude が情報処理に追われて「コンパナ風に書く」余力が減る
- 「全情報の中から自分で選ぶ」タスクがあるので材料の鮮度より網羅性優先になる
- **結果: 毎日似たような「ゴトー日」「通貨強弱%」投稿に**

#### 学んだこと

1. **時間帯を軸にした投稿タイプ設計は間違い**: 「LUNCH だから生活ネタ」のような発想は、読者のニーズ(最新材料への渇望)とズレる
2. **材料の鮮度が本質**: プロンプト削減やコード改修より、「今この瞬間のホットな材料を1つ選ぶ仕組み」が最重要
3. **Claudeがドキュメントを鵜呑みにする危険性**: 「プロンプト肥大化で attention 希薄化」という書籍的な説明を鵜呑みにし、本質(材料の鮮度)を見落とした
4. **コンパナの感覚を最優先**: Claude の分析が「LUNCH は役割逸脱」と判断しても、コンパナが「良さげ」と言えばそれが正解。設計書よりコンパナの評価が上
5. **投稿の評価軸を変える**: 「文字数内・役割通り・文末完結」よりも「新鮮な材料があるか・因果が明確か・次の見方があるか」

## 🎯 Part IV: ベストプラクティス集

### IV-1. データ優先順位の絶対ルール

T-CAX の全ての判定プロセスで、以下の優先順位を守る:

```
最優先 ①経済カレンダー確定データ(結果欄の有無で発表済み/未発表を判定)
       ↓
       ②追加の確定データ(政策金利・通貨強弱・継続中重大事象)
       ↓
       ③本文内の数値データ(APIから取得済みのレート等)
       ↓
       ④Web検索結果(SEO影響で古い記事が上位に出やすいので注意)
       ↓
最低  ⑤Claudeの内部知識(カットオフ以前の情報)
```

**理由**:
- ①〜③は「この瞬間の真実」として確定している
- ④は便利だが「現在」と「過去」の区別が曖昧
- ⑤は最もバイアスが乗りやすい

### IV-2. 確定データの一元化(collectAnchorData_ の全面活用)

**原則**: 確定データの収集は `collectAnchorData_()`(geminiApi.gs 定義)に一元化する。

**理由**:
- 同じ情報を複数箇所で個別実装すると、片方が欠けたときに Phase 間で判定矛盾が起きる
- データソース変更時の修正箇所が1箇所に集約される
- 新しい検証ステップを追加する時、既存の確定データを即座に再利用できる

**使用箇所(v12.9 時点)**:
- geminiApi.gs: `finalFactVerify_()`(Phase A 最終事実検証)
- qualityReview.gs: `qualityReviewPost_()` Q6(v12.9 追加)
- interactiveVerify.gs: Step 4 対話型検証(v12.7 タスク17-a で新設)

**NGパターン**:
```javascript
// ❌ 個別に取得している(再現性が崩れる)
var rates = getLatestRates();
var anchors = buildAnchorString(rates, strengthData, events);
```

**OKパターン**:
```javascript
// ✅ 既存の集約関数を使う
var anchorData = collectAnchorData_(postType);
```

### IV-3. Claude の時系列バイアス対策

**症状**:
- Claude の知識カットオフと Web 検索結果の新旧が混在すると、「カットオフ以前の情報」を「現在」と誤認する

**対策**:

1. **確定データの注入**: 「現在の政策金利は○○」という確定値を Claude に渡す
2. **判定優先順位の明示**: プロンプトに「確定データ > Web検索」を明記
3. **根拠付きで説明**: 「Web検索は SEO 影響で古い記事が上位に出やすい」という理由を書くと、Claude は優先順位を守りやすい
4. **具体例を提示**: 「RBA 4.10% と確定データにある時、Web検索で 2025年利下げ記事が出ても、それは古い情報として無視せよ」のような具体例を入れる

### IV-4. 4段防御 + Step 4対話型検証

T-CAX の現在の防御体系:

```
Phase A:
  防御1: 発生防止(Claude主導 + 確定データで未検証情報を混入させない)
  防御2: Gemini検出(Layer1 確定データ + Layer2 Grounding検索)
  防御3: Claude検出(Q1-Q7 + Q6 Web検索 + Q7 絵文字行チェック)
  防御4: finalFactVerify_(Claude JSON 検出 + コード置換)

Phase B:
  防御3': 品質レビュー再実行
  防御4': 最終事実検証再実行
  防御5: 対話型検証 Step 4(Claude + Web検索)

Phase C:
  画像生成 + モード分岐
```

**設計思想**:
- 各防御層の**役割を分離**: Claude は検出、コードは置換
- **確定データを真実のアンカー**として一貫して活用
- **段階的に絞り込む**: 広い検証(Gemini)→ 精密検証(Claude)→ 最終検証(対話型)

### IV-5. Phase 3分割の意味

**v12.7 以前**(Phase 未分割):
- 1つの関数で「生成→検証→画像→承認メール」を全て実行
- 6分の GAS タイムアウトでギリギリ(特に画像生成時)
- 途中でエラー発生すると、どこまで進んだか追跡が困難

**v12.7 以降**(Phase A→B→C 3分割):
- Phase A: 生成(目標3-4分)
- Phase B: 品質整形 + 対話型検証(目標3-4分)
- Phase C: 画像生成 + モード分岐(目標2-3分)
- 各 Phase 間は**1分後トリガー**で自動連携
- 下書きシートにステータスが記録されるため、**どこで止まったか一目瞭然**

**メリット**:
- 6分タイムアウト回避
- エラー発生時の影響範囲限定
- 緊急中止ウィンドウの確保(Phase A 完了〜Phase B 開始の1分間など)
- **全モード(manual / validate / auto)で下書き必須**(v12.8)→ autoでも事後確認可能

### IV-6. Phase 4 学習ループの鉄則

投稿後の自動学習(Phase 4)で守るべき原則:

1. **仮説の検証は本人の投稿文から抽出**: 人間が後から「これは仮説だった」と解釈するのではなく、投稿時点で「構造仮説」「今日の仮説」を明記させる
2. **エンゲージメントは 24 時間後に取得**: 投稿直後はエンゲージメントが未確定のため、24 時間のウィンドウを取る
3. **学びログの重み付け**: よく当たる学びは重み上位、外れる学びは重み下位に自動調整(70% 上位 + 30% ランダム注入)
4. **対話型検証ログは未使用の宝**: v12.7 で蓄積開始。将来のプロンプト改善の根拠データとして使う

### IV-7. 文字数保証の段階戦略

投稿本文の文字数制限を守るため、以下の段階を踏む:

```
1. プロンプトで文字数指定(150-350字 等)
2. Claude が生成(ほぼ守られるが超過することあり)
3. 品質レビュー Q4 で文字数チェック(超過時は削減指示)
4. Claude が修正(再生成ではなく元投稿ベース修正)
5. コード側で文字数保証(超過時は末尾文を削除して圧縮)
```

**ポイント**: **Claude 単独では 100% 守れない**。コード側の最終圧縮ステップが必要。

### IV-8. プロンプト構築の階層化

T-CAX のプロンプトは **111セクション** で構成されている(v12.9 時点)。肥大化を防ぐため:

- **スプレッドシートで定義**: キャラクター / 投稿プロンプト / トレードスタイル等
- **コードで注入**: 確定データ / 市場分析 / 学び / 仮説検証
- **動的生成**: 夜の材料ヒント(v12.9)など、条件によって変わる部分

**鉄則**: 1つの情報は1箇所で定義する。重複は混乱の元。

### IV-9. 対話型検証 Step 4 の費用対効果

**API呼び出し回数**:
- Step 1(質問抽出): Claude API 1回
- Step 2(Web検証): Claude API 1回(Web検索ツール使用)
- Step 3(修正): Claude API 0-1回(❌/⚠️ がある場合のみ)

**所要時間**: 30-60秒 / 投稿

**検出できる誤り**:
- 文脈依存の事実誤り
- 発言引用のニュアンス精度
- 既存のファクトチェック + Q6 では見逃す微妙な誤り

**運用判断基準**:
- 100件以上のログが蓄積されるまでは有効化継続
- ❌/⚠️ 判定率が 10% を超えるなら常時有効
- 10% 未満なら無効化を検討(コストカット)

### IV-10. Gemini 再読み込みの是非

過去に **「確認のため再読み込みする」のは事故の元**との教訓がある:

**悪いパターン**:
```
Grounding OFF で生成
→ 「確認のため Grounding ON でもう一度」
→ Grounding が拾う過去の情報で投稿が壊れる
```

**良いパターン**:
```
ニュース取得: Grounding ON(事前にニュースだけ取得)
テキスト生成: Grounding OFF(取得済みニュース+確定データで書く)
検証: 別モデル(Claude)で確定データと照合
```

**原則**: **用途ごとに Grounding ON/OFF を固定する**。途中で切り替えない。
