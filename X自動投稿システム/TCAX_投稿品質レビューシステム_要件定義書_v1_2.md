## 1. 背景と目的

### 1-1. 現在の問題

現在のT-CAXは「事実の正誤」はファクトチェック（factCheckPost_）で検証しているが、
「投稿としての品質」を検証する仕組みがない。

結果として、以下の問題が人間（コンパナ）の目視確認でしか検出できない:

| 問題カテゴリ | 具体例 | 現在の対応 |
|-------------|--------|-----------|
| A. タイプ不整合 | LONDON投稿なのに「東京市場」で始まる | なし |
| B. 1日の重複 | 6投稿全てが「休む勇気が大事」で締まる | なし |
| C. 文の未完成 | 体言止めで文が完結していない | なし |
| D. 時間帯の矛盾 | MORNING投稿が「今まさに〜」と現在進行形 | プロンプトで対応（不完全） |
| E. charMax超過 | 450文字制限で504文字の投稿が出る | 700文字ハードカットのみ |
| F. 同じニュース主題 | 1日6投稿が全て同じニュース・同じ切り口 | プロンプトで「最大2回」指示（不完全） |
| G. TC導線の不自然さ | 「Trading TC」等の誤表記、唐突なTC言及 | removeTCMention_（部分的） |

### 1-2. 目的

完全自動投稿（validateモード→autoモード移行）に必要な
「人間が承認前にチェックしていたこと」をシステム化する。

### 1-3. ゴール

品質レビューを通過した投稿は、人間の確認なしで投稿できるレベルになること。

## 2. なぜClaude APIを導入するのか

### 2-1. 同一モデルチェックの限界

T-CAXのパイプラインはGemini APIで統一されている:

```
テキスト生成:   Gemini 2.5 Flash
ファクトチェック: Gemini 2.5 Flash（Grounding ON）
自動修正:       Gemini 2.5 Flash（Grounding OFF）
```

ここに「品質レビュー」を追加する場合、再びGeminiを使うことは
「自分が書いた宿題を自分で採点する」のと同じ構造になる。

実際に2026-03-23のセッションで確認された問題:

| 問題 | Geminiの挙動 | 根本原因 |
|------|-------------|---------|
| ファクトチェックJSON解析失敗 | Geminiが自分で定めたJSON形式を守らない | 自分の出力フォーマットに対する甘さ |
| 「S&P500が9.6%下落」 | 生成→ファクトチェック「検証不能」 | 自分が書いたことを「確認できない」で通す |
| レート5桁表記 | プロンプトで4桁ルールを指示→無視 | 自分のルールに対する注意力不足 |
| LONDON投稿で「東京市場」始まり | ファクトチェックで検出されない | 事実検証の範囲外。品質チェック不在 |

### 2-2. クロスチェックの原則

異なるモデルでレビューすることで「バイアスなしの第三者チェック」が実現する。

人間の仕事に例えると:
- 同一モデル: 記者が書いた記事を同じ記者が校正する → 自分のミスに気づかない
- クロスチェック: 記者が書いた記事を編集者が校正する → 客観的なチェック

### 2-3. Claude Sonnet 4.6を選んだ理由

| 観点 | 詳細 |
|------|------|
| 指示遵守 | Claude Sonnet系はJSON出力の正確性が高い。Geminiで散発するJSON解析失敗が発生しにくい |
| コスト | 入力$3/MTok + 出力$15/MTok。1投稿あたり約0.2円、月179件で約36円 |
| Grounding不要 | 品質レビューは事実検証ではないためGoogle検索不要。Claude APIで完結 |
| 独立性 | Geminiとは完全に異なるモデル。同一モデルバイアスが構造的に排除される |
| TCとの将来統合 | Trading CompleteのPhase 2でClaude API導入を計画済み。ここで先行導入することでAPI運用ノウハウが蓄積される |

### 2-4. モデル使い分けの最終設計

```
【Gemini担当（Google検索が必要な仕事）】
  テキスト生成:    Gemini 2.5 Flash + Grounding ON（最新ニュース検索）
  ファクトチェック:  Gemini 2.5 Flash + Grounding ON（事実のGoogle検索検証）
  自動修正:        Gemini 2.5 Flash + Grounding OFF（correction情報で修正）
  
【Claude担当（Google検索が不要な仕事）】
  品質レビュー:    Claude Sonnet 4.6（投稿品質のクロスチェック）
  
分担の原則: Groundingが必要 → Gemini / 不要 → Claude
```

## 3. 現在のパイプラインと挿入位置

### 3-1. 品質レビュー挿入後のパイプライン

```
① テキスト生成（Gemini + Grounding）
② 後処理チェーン（18関数）
③ リトライ群（主役ペア/絵文字/リスクセンチメント/→複数ブロック）
④ ファクトチェック（Gemini + Grounding ON: 事実の正誤）
⑤ 検証不能❌即削除 + 修正可能❌をautoFix（Gemini）
⑥ 後処理チェーン再適用
⑦ ★品質レビュー（Claude Sonnet 4.6: 投稿としての品質）← 新規追加
⑧ 品質修正があれば後処理チェーン再適用
⑨ 今日の投稿キャッシュに保存（次の投稿の重複チェック用）
⑩ 結果保存（ScriptProperties）
```

挿入位置の理由:
ファクトチェック+autoFixで「正確な投稿」にした後に、「良い投稿か？」をレビューする。
順序が逆だと、品質レビューで修正したテキストがファクトチェックで壊される可能性がある。

## 4. レビュー項目の設計

### 4-1. レビュー項目一覧

| ID | カテゴリ | チェック内容 | 判定方法 |
|----|---------|------------|---------|
| Q1 | タイプ整合 | 投稿の冒頭・トーンがpostTypeの時間帯と合っているか | Claude + TYPE_DESCRIPTIONS |
| Q2 | 表現重複 | 今日の過去投稿と同じ締め文・フレーズが使われていないか | Claude + 過去投稿キャッシュ |
| Q3 | 文の完成度 | 体言止め・文脈の飛躍・途中で切れた文がないか | Claude |
| Q4 | 文字数 | charMin〜charMax内に収まっているか | Claude（現在の文字数をプロンプトで通知） |
| Q5 | 口調の一貫性 | コンパナの口調が維持されているか | Claude |

### 4-2. 修正方法

問題が見つかった場合、Claudeが修正済みテキスト（revisedText）を返す。

修正時の絶対ルール（プロンプトで指示）:
- 絵文字構造（📕📝💡☕⚠️✅で始まるブロック）は変えない
- →行の構造も変えない
- ハッシュタグ行はそのまま
- 事実やレートの数値は変更しない（事実検証はファクトチェックの仕事）
- 修正は最小限。問題のある箇所だけ直す

修正後は後処理チェーンを再適用してフォーマットを保証する。

### 5-2. qualityReview.gsの関数構成

```
qualityReviewPost_(postText, postType, typeConfig)
  品質レビューのメイン関数。
  Claude Sonnet 4.6にレビュープロンプトを送信し、JSON結果を返す。
  SKIP_FACT_CHECK=true（testAll時）はスキップ。

callClaude_(prompt, apiKey)
  Anthropic Messages API呼び出し。
  モデル: claude-sonnet-4-6
  リトライ: 最大2回（429時は5秒待機）

parseClaudeResponse_(text)
  Claude応答からJSONを抽出・パース。
  ```json ... ``` の除去にも対応。

getTodayPreviousPosts_()
  ScriptPropertiesから今日の過去投稿テキストを取得。
  Q2（表現重複）の判定に使用。

cacheTodayPost_(postType, postText)
  投稿テキストをScriptPropertiesにキャッシュ。
  generatePost成功時にgeminiApi.gsから呼ばれる。

clearTodayPostCache_()
  前日のキャッシュをクリア。
  scheduleTodayPosts()の冒頭でscheduler.gsから呼ばれる。

testQualityReview()
  手動テスト用。LONDON投稿（東京市場始まり）のサンプルで動作確認。
```

### 5-3. 投稿タイプ別のTYPE_DESCRIPTIONS

品質レビューでQ1（タイプ整合）を判定するために、各タイプの時間帯と役割を定義。

```
MORNING:  朝7:30頃。昨夜のNY市場おさらい→今日の東京への橋渡し。
TOKYO:    朝9:30頃。東京オープン後の「観察」。予想ではなく事実。
LUNCH:    昼12時台。午前の振り返り+欧州勢参入への橋渡し。
LONDON:   夕方17時台。「ここから欧州勢が本番」の緊張感。
GOLDEN:   夜20-21時台。1日を振り返る共感重視。居酒屋トーク。
NY:       夜22時台。NY市場オープン前の緊張感。
INDICATOR: 指標30分前の速報感。
```

### 5-4. 過去投稿キャッシュ（ScriptProperties）

```
キー: TODAY_POST_{postType}
値: JSON { type, text, time }
保存タイミング: generatePost成功時（geminiApi.gs内）
クリアタイミング: scheduleTodayPosts()冒頭（毎朝自動）
容量: 1投稿約700文字 × 16タイプ = 約11KB（制限500KBに対して余裕）
testAll時: SKIP_FACT_CHECK=trueの時はキャッシュしない
```

### 5-5. GAS 6分制限との兼ね合い

```
現在のGemini呼び出し回数（最悪ケース）:
① テキスト生成:              1回
② 主役ペアリトライ:          1回
③ 絵文字リトライ:            1回
④ リスクセンチメントリトライ:  1回
⑤ →複数ブロックリトライ:     1回
⑥ ファクトチェック:           1回（Gemini Grounding ON）
⑦ 自動修正:                  1回（Gemini）
⑧ 自動修正リトライ:          1回（Gemini）
⑨ ★品質レビュー:             1回（Claude）← 追加
合計: 最大9回（現在8回 + 1回追加）

各呼び出し約5-7秒。9回 = 最大63秒。GAS 6分（360秒）に余裕あり。
Claude APIはGrounding不要で軽量。応答速度はGeminiより速い傾向。
```

### 5-6. スキップ条件

- SKIP_FACT_CHECK=true（testAll時）→ 品質レビューもスキップ
- CLAUDE_API_KEY未設定 → スキップ（ログ出力のみ。システムは停止しない）
- Claude API呼び出し失敗 → スキップ（品質レビューはフェイルセーフ設計。失敗しても投稿は生成される）

### 5-7. 必要なスクリプトプロパティ

| キー | 値 | 取得先 |
|------|-----|--------|
| CLAUDE_API_KEY | Anthropic APIキー | https://console.anthropic.com/settings/keys |

### 5-8. Claude API呼び出しの仕様

```
エンドポイント: https://api.anthropic.com/v1/messages
モデル: claude-sonnet-4-6
max_tokens: 2000
ヘッダー:
  x-api-key: {CLAUDE_API_KEY}
  anthropic-version: 2023-06-01
  Content-Type: application/json
リトライ: 最大2回（429→5秒待機、その他→2秒待機）
```

## 6. コスト試算

### 6-1. 1投稿あたりのコスト

```
入力トークン: 約2,000トークン（投稿テキスト+過去投稿+レビュー項目）
出力トークン: 約500トークン（JSON結果+revisedText）

Claude Sonnet 4.6:
  入力: 2,000 / 1,000,000 × $3 = $0.006
  出力: 500 / 1,000,000 × $15 = $0.0075
  合計: $0.0135/投稿 ≒ 約2.0円/投稿（$1=150円換算）

※ 過去投稿が多い場合（6投稿分キャッシュ）は入力が増えるが、
   最大でも約5,000トークン → $0.015 + $0.0075 = $0.0225（約3.4円）
```

### 6-2. 月間コスト

```
平日: 6投稿 × 20日 = 120投稿
土日: 4投稿 × 8日 = 32投稿
指標連動: 約27投稿/月
合計: 約179投稿/月

月額コスト: 179 × 2.0円 = 約358円/月（最大でも約600円/月）

初回クレジット: $5.00購入済み（2026-03-24）
$5 ÷ $0.0135 = 約370投稿分 ≒ 約2ヶ月分
```

## 8. リスクと注意事項

### 8-1. Claude APIの外部依存
- Anthropic APIのダウンタイムで品質レビューが実行できない可能性
- 対策: フェイルセーフ設計。API失敗時はスキップして投稿を続行

### 8-2. 修正によるフォーマット崩れ
- revisedTextのフォーマットが崩れるリスク
- 対策: 修正後に後処理チェーン（18関数）を再適用

### 8-3. 過剰修正
- 問題ないのにClaudeが書き換えてしまうリスク
- 対策: プロンプトで「問題なければpassedをtrueにしろ」「修正は最小限」を強調
- Phase 2でチューニング

### 8-4. GASからの外部API呼び出し制限
- GASのUrlFetchApp日次制限: 20,000回/日（無料枠）
- 品質レビュー: 最大179回/月（約6回/日）。制限に対して余裕
