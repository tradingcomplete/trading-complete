## 3. Phase 1.5: ファクトチェック + 自動修正 ★v1.1追加 ✅実装完了

### 3-0. なぜPhase 2より先に必要か

```
Phase 2以降は「投稿の内容を学習する」仕組み。
もし誤った情報が投稿されていたら:
  → Phase 2: 間違った仮説を「的中」と判定してしまう
  → Phase 3: 間違った投稿に含まれる学びにスコアが加算される
  → Phase 4: 間違った投稿のエンゲージメントで品質評価が歪む
  → Phase 5: 歪んだデータでプロンプトが「改悪」される

つまり「学習データの品質保証」が全Phaseの前提条件。
```

### 3-0-1. 概要

投稿テキスト生成後、事実の正確性を自動検証し、誤りがあれば自動修正する。
「生成」と「検証」を別のGemini呼び出しで行う2段構え方式。

### 3-0-2. 処理フロー

```
(1) テキスト生成 → 後処理 → リトライ（既存）

(2) factCheckPost_（Gemini + Grounding）
    → 投稿内の事実を1つずつ抽出
    → Google検索で裏付けを確認
    → 各事実を ✅正確 / ⚠️要確認 / ❌誤り で判定
    → JSON形式で構造化された検証結果を返す

(3) 判定分岐:
    ✅全て正確 → そのまま承認メールへ
    ⚠️/❌あり → (4)へ

(4) autoFixPost_（Gemini + Grounding）
    → 誤りを指摘し「正確な情報に修正せよ」と指示
    → 修正後テキストに差し替え → 再度後処理を適用

(5) 承認メール送信
    → 修正済みテキスト + ファクトチェック結果（Before/After）を表示
    → コンパナが最終確認 → 承認/却下
```

### 3-0-3. 検証対象

```
【検証する事実】
  ・要人発言: 誰が、いつ、何を言ったか（発言者の混同がないか）
  ・経済指標: 名称、日付、予想値、前回値、結果値
  ・レート水準: 具体的な数値の正確性
  ・日付の整合性: 「今日」「昨日」の出来事が本当にその日か
  ・中央銀行の政策: 決定内容の正確性

【検証しない】
  ・個人の感想（「マジで注目」等）
  ・相場観（「上がりそう」等）
  ・RULE_1〜4（個人の経験談が主体）
```

### 3-0-4. 承認メールの表示例

```
【投稿テキスト】（修正済み）
☕ブロック総裁、3月利上げの可能性を示唆。
→RBAのブロック総裁が「every meeting is live」と発言...

━━━━━━━━━━━━━━━━━━━━━━
📋 ファクトチェック: ⚠️要確認あり → 自動修正済み
━━━━━━━━━━━━━━━━━━━━━━
1. 植田総裁が本日追加利上げに言及 → ❌ 本日ではなく1/23の発言
2. 豪1月貿易収支の予想39億豪ドル → ✅ 正確

🔧 自動修正が適用されました
  ❌ 植田総裁の発言日を修正 → 修正済み
━━━━━━━━━━━━━━━━━━━━━━
```

### 3-0-5. 対象ファイル・関数

```

  factCheckPost_()   → ファクトチェック（Gemini+Grounding）
  autoFixPost_()     → 自動修正（Gemini+Grounding）

```

### 3-0-6. 実装の難易度と工数

- 難易度: 中〜高
- 工数: 実績5時間（設計2h + 実装2h + テスト1h）

### 3-0-7. autoFixPost_メタ行除去の汎用化 ★v1.3追加

```
autoFixPost_がGeminiに修正を依頼すると、修正後テキストの前に
メタ行（「修正後の投稿テキスト：」「FX関連X投稿の修正を行います。」等）が
付与されるケースがある。

対策: stripAIPreamble_に汎用パターンを追加（個別対応→汎用化）
  → 先頭60文字に「修正後/修正を/投稿テキスト/以下の通り/投稿の修正」
    が含まれる行を自動除去
  → 「【修正後】」「【修正前】」「【元の】」等の角括弧ラベル行も除去
  → 本文中の「下方修正」「修正局面」等には反応しない（先頭行のみ対象）
```

## 4. Phase 2: 仮説の的中率自動集計 ✅実装完了

### 4-1. 概要

WEEKLY_HYPOTHESISで立てた仮説を、翌週のレートデータで自動検証する。

### 4-2. 新規シート: 仮説検証ログ

```
列  ヘッダー          内容                        データ型
─────────────────────────────────────────────────────────
A   仮説ID            一意のID                     yyyyMMdd_N
B   仮説日            仮説を立てた日               yyyy/MM/dd
C   仮説内容          テキスト                     テキスト
D   対象ペア          USD/JPY等                    テキスト
E   仮説方向          上昇/下落/レンジ              テキスト
F   仮説時レート       仮説時点のレート              数値
G   検証日            1週間後の日付                 yyyy/MM/dd
H   検証時レート       検証時点のレート              数値
I   変動幅            検証時 - 仮説時               数値（+/-）
J   判定              的中/不的中/判定不能           テキスト
K   的中理由          なぜ当たった/外れたか          テキスト（Gemini自動生成）
```

### 4-3. 処理フロー

```
(1) WEEKLY_HYPOTHESIS投稿後
    → extractPostInsights_で仮説を抽出（既存）
    → parseHypothesisDetails_で対象ペア・方向を自動パース ★v1.2追加
    → 仮説検証ログに新規行を追加（仮説ID, 仮説日, 仮説内容, 対象ペア, 仮説方向, 仮説時レート）
    → G〜K列は空欄のまま

(2) 翌週の月曜 朝5:00（scheduleTodayPosts内）
    → verifyWeeklyHypotheses_で未検証行を取得
    → レートキャッシュから検証時レートを取得
    → 変動幅を計算
    → 判定ロジック:
        仮説方向=上昇 かつ 変動幅 > +0.3円（JPYペア）/ +0.003（USDペア） → 的中
        仮説方向=下落 かつ 変動幅 < -0.3円 / -0.003 → 的中
        仮説方向=レンジ かつ |変動幅| < 0.5円 / 0.005 → 的中
        それ以外 → 不的中
    → Geminiで「なぜ当たった/外れた」を1行で自動生成
    → 結果をシートに書き込み

(3) 次のWEEKLY_HYPOTHESIS生成時
    → getHypothesisVerificationSummary_で直近5件の結果をプロンプトに注入
    → 例: 「直近5件の的中率: 3/5（60%）。上昇予測は2/3的中、下落予測は1/2不的中」
```

### 4-4. プロンプト注入例

```
【前回までの仮説検証（自動集計）】
直近5件の的中率: 3/5（60%）
  ✅ 2/28 ドル円は156円台を維持 → 的中（+0.45円）
  ❌ 2/21 ユーロドルは下落 → 不的中（+0.0021、むしろ上昇）
  ✅ 2/14 ドル円は155円を突破 → 的中（+1.23円）
  ...
傾向: 上昇予測の的中率が高い。下落予測は苦手。
→ この傾向を踏まえて仮説を立てよ。
```

### 4-5. 対象ファイル・関数

```
geminiApi.gs:
  parseHypothesisDetails_()            仮説テキストから対象ペア・方向を自動パース
  verifyWeeklyHypotheses_()            未検証仮説をレートデータで自動判定
  generateVerificationReason_()        的中理由をGeminiで1行自動生成
  getHypothesisVerificationSummary_()  直近5件の的中率サマリーを生成

  getUnverifiedHypotheses()            未検証仮説を取得
  updateHypothesisVerification()       検証結果を書き込み
  getRecentHypothesisResults()         直近N件の検証済み結果を取得

  scheduleTodayPosts()                 月曜朝にverifyWeeklyHypotheses_を呼び出し

## 5. Phase 3: 学びの重み付け ✅実装完了

### 5-1. 概要

### 5-2. 学びログシートの拡張

```

追加列（★v1.2で実装）:
E   使用回数        プロンプトに注入された回数        数値（初期値0）
F   有効度スコア    この学びが入った投稿の品質         数値（初期値0）
G   最終使用日      最後に注入された日                 yyyy/MM/dd
H   ステータス      active/retired                    テキスト

### 5-3. 処理フロー

```
(1) getLearningLog_をスコアベース選択に変更 ★v1.2実装済み
    → 70%: 有効度スコア上位から選択（実績重視）
    → 30%: ランダムに選択（新しい学びにもチャンスを与える）
    → 選択された学びの「使用回数」を自動インクリメント
    → 同スコアなら使用回数が少ないものを優先（新しい学びにチャンス）

(2) Phase 4連携（★v1.2で基盤実装済み、スコア加減算は今後）:
    → エンゲージメントの良い投稿で使われた学びのスコアを加算
    → エンゲージメントの悪い投稿で使われた学びのスコアを減算
    → 一定期間使われない学びは retired に変更

(3) 月1回の自動整理（Phase 5で実装予定）:
    → retired が多すぎる場合はアーカイブ
    → 同じ意味の学びを統合（Geminiで類似判定）
```

## 6. Phase 4: エンゲージメントフィードバック ✅実装完了

### 6-1. 概要

X API v2でインプレッション/いいね数を取得し、投稿品質を数値化する。

### 6-2. 新規シート: エンゲージメントログ

```
列  ヘッダー          内容                        データ型
─────────────────────────────────────────────────────────
A   投稿ID            投稿履歴のID                  テキスト
B   ツイートID        X APIのtweet_id               テキスト
C   取得日時          メトリクス取得日時              yyyy/MM/dd HH:mm:ss
D   インプレッション   表示回数                       数値
E   いいね数          いいね数                       数値
F   RT数              リツイート数                    数値
G   返信数            返信数                         数値
H   エンゲージメント率  (いいね+RT+返信)/インプレッション  数値（%）
I   品質スコア         正規化されたスコア（0〜100）    数値
J   投稿タイプ         MORNING等                     テキスト
```

### 6-3. 処理フロー

```
(1) 毎朝5:00にscheduleTodayPostsから自動実行 ★v1.2実装済み
    → 投稿履歴から24時間以上前かつ未取得のツイートIDを収集
    → X API v2 GET /2/tweets?ids={id}&tweet.fields=public_metrics でバッチ取得
    → メトリクスをエンゲージメントログに保存

(2) 品質スコアの計算 ★v1.2実装済み
    → 同じ投稿タイプの直近30件の平均エンゲージメント率を基準
    → ER/平均ER x 50 で正規化（平均=50点、2倍=100点、0=0点）
    → データ5件未満の場合は全タイプ平均を使用

(3) プロンプト注入 ★v1.2実装済み
    → 同タイプ5件以上蓄積後に自動注入開始
    → 平均ER、ベスト/ワーストの統計をプロンプトに追加
    → 品質スコアの高い投稿で使われた学びの有効度スコアを加算（Phase 3連携、今後実装）
```

### 6-4. プロンプト注入例

```
【投稿品質フィードバック（自動集計）】
MORNING 直近30件: 平均印象250回 / 平均ER0.45%
ベスト: 印象373回 ER2.2%
ワースト: 印象41回 ER0%
→ ER0.45%以上を目指せ。読者が反応する具体的な数値やニュースを含めよ。
```

### 6-5. X API仕様

## 7. Phase 5: プロンプト自動進化 ⬜待機中

### 7-1. 概要

蓄積されたデータ（仮説的中率、学びスコア、エンゲージメント）を元に、buildFormatRules_やキャラクタープロンプトのルールを自動更新する。

### 7-2. 処理フロー

```
(1) 月1回の自動分析（月初の月曜日に実行）
    → 直近1ヶ月の全投稿データを集計
    → Geminiに「改善提案」を生成させる

(2) Geminiへの分析プロンプト:
    → 直近30件の投稿テキスト + 品質スコア
    → 的中率の高い仮説パターン
    → 有効度スコアの高い学び
    → 「現在のルールで改善すべき点を3つ提案せよ」

(3) 提案をスプレッドシートに保存
    → 「ルール改善提案」シートに記録
    → コンパナが確認して手動で承認（安全策）
    → 承認された提案はキャラクタープロンプトシートに反映
```

### 7-3. 安全策

Phase 5は「完全自動」ではなく「提案→承認」のフローにする。
理由: プロンプトの自動変更は品質劣化のリスクがあるため。

将来的に:
- 承認率が90%を超えたら、低リスクな変更は自動適用
- 高リスクな変更（禁止事項の変更等）は常に承認制

### 7-4. 実装の難易度と工数

- 難易度: 高
- 工数: 5〜8時間
- Phase 4完了: ✅（前提条件クリア）
- 状態: ⬜ **待機中**（1ヶ月のデータ蓄積後に着手予定）
- 待機理由:
  - 仮説検証ログ: 1件（最低20件程度必要）
  - エンゲージメントログ: 17件（最低100件=約2週間分必要）
  - 学びログのスコア差: 差がつき始めた段階（使用回数の差がつくまであと1〜2週間）
  - データ不足でGeminiに分析させると的外れな提案になるリスクが高い
- 着手目安: 2026年4月上旬（運用開始から1ヶ月後）

## 10. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| ファクトチェックが過剰修正 | 正確な内容を不要に書き換える | 最終判断はコンパナの承認制。Before/After表示で確認 |
| ファクトチェックのAPI障害 | 検証なしで投稿される | エラー時は従来通りメール送信（投稿を失わない） |
| 学びが偏る | 同じパターンの投稿ばかりに | Phase 3の30%ランダム枠で多様性確保 |
| 仮説検証の判定基準が不適切 | 的中/不的中が実態と乖離 | 閾値（JPY:0.3円/USD:0.003）を運用データで調整 |
| X API課金超過 | クレジット残高不足 | 月額$0.30。残高$9.79で約30ヶ月分。アラート設定推奨 |
| Geminiの分析品質 | 的中理由/改善提案が的外れ | Phase 5は承認制にして安全策 |
| プロンプト肥大化 | 注入データ増でトークン超過 | 各Phase注入は最大500文字の上限を設定 |
| エンゲージメントデータ不足 | 品質スコアが不安定 | 同タイプ5件未満はフィードバック注入スキップ |
| autoFixPost_メタ行混入 | 「修正後の投稿テキスト：」等が投稿される | stripAIPreamble_で汎用パターン除去（v1.3で対応済み） |

  → geminiApi.gs / scheduler.gs の3箇所を修正
